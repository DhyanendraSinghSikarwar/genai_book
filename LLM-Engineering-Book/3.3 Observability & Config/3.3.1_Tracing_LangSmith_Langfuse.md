## 3.3.1.0 Where this fits (the bridge from 3.3)

The overview split observability from control; this is the observability engine. Its job is to answer, for any request your app handled: *what exactly happened, step by step, and what did each step cost?* When a RAG answer is wrong, or an agent loops forever, or the monthly bill spikes, tracing is the tool that turns "something's off" into "**this** span, with **this** input, produced **this** output, took **this** long, and cost **this** many tokens." Everything else in LLMOps — evaluation, monitoring, cost optimization — is built on the data that tracing captures.

Crucially, tracing is **low-friction to adopt**: for LangChain/LlamaIndex apps it's often a matter of setting a few environment variables and every step is captured automatically (because the frameworks are already instrumented via callbacks — the mechanism you'll meet in 3.3.2). That low cost of adoption plus enormous debugging payoff is why tracing is the *first* thing experienced teams add to a serious LLM project.

> **One-line thesis:** *Tracing records each request as a tree of spans — every prompt, retrieval, model call, tool call, and parse, with inputs, outputs, latency, tokens, cost, and errors. LangSmith and Langfuse are the platforms that capture, visualize, and evaluate those traces; pick based on integration needs and data-control requirements.*

## 3.3.1.a What a Trace Is — Runs, Spans, and the Tree

**Definition.** A **trace** is the complete, structured record of a single request through your application. It is composed of **spans** (LangSmith calls them **runs**) — one per step — arranged as a **tree** that mirrors your call structure: a top-level span for the whole request, with child spans for each sub-step (prompt render → retrieval → LLM call → parse → tool call), nested to whatever depth your pipeline goes. Each span captures its **inputs, outputs, start/end time (latency), token usage, cost, metadata/tags, and any error**.

**Why the tree structure is the key idea, explained.** A flat log ("called the model, got a response") tells you little when a request touches eight components. The **hierarchy** is what makes tracing powerful: it shows you *causality and containment* — this agent step *contained* these two tool calls, one of which *contained* a retrieval that returned the wrong chunks. You can expand exactly the branch that matters and read the **precise input each step received after all your templating, history injection, and retrieval** (recall how much processing happens before the model sees anything — 2.1.3, 3.1.2). Seeing the *actual* prompt the model got — not the template you wrote — is often the "aha" that solves a bug. The tree also lets you see **latency and token cost per branch**, so you can attribute the slow/expensive part precisely rather than guessing.

**What each span records (and why each field matters).**
- **Inputs/outputs** — the content in and out of the step; the core of debugging (see what actually flowed).
- **Latency** — start/end timing per step; find the bottleneck (2.3.3's decode-bound latency shows up here).
- **Tokens & cost** — per-step token counts and dollar cost; attribute the bill (the `usage` data from 2.3.1, now per step).
- **Metadata & tags** — your custom labels (user, feature, prompt/model version) for filtering/searching (set via config — 3.3.2).
- **Errors & status** — captured exceptions/retries per step; see *where* and *why* a request failed.

**Intuition — a stack trace that never throws, plus a stopwatch and a meter on every frame.** A programming stack trace shows the nested calls that led to an error. An LLM trace is like a stack trace you get for *every* request whether or not it errored — and each frame also carries a stopwatch (latency), a meter (tokens/cost), and the actual data that passed through. When something's wrong, you don't reproduce-and-print; you open the trace and read the exact frame.

**Interview angle.** "What is a trace vs a span/run?" (whole request vs one step; spans form a tree). "Why is the tree structure important?" (shows causality/containment; isolate the exact failing branch). "What does each span capture?" (I/O, latency, tokens/cost, metadata, errors). "What's the most common debugging 'aha'?" (seeing the *actual* prompt/context the model received after templating/retrieval).

**Finance use cases.**
1. **Pinpointing a wrong citation:** In a compliance-bot trace, expanding the retrieval span shows it returned the wrong policy nodes — so the fix is chunking/metadata (3.2.1), not the model. The tree made the root cause obvious.
2. **Latency SLA diagnosis:** Per-span timing on an advisory query reveals a re-ranking step as the bottleneck, so it can be tuned to meet a response-time SLA.

**E-commerce use cases.**
1. **Bad product answer root cause:** A trace shows the model received an unrelated SKU's spec chunk — a retrieval/metadata bug surfaced by reading the actual input span.
2. **Cost hotspot:** Token counts per span reveal an over-stuffed system prompt on every storefront-chat turn, guiding a prompt trim / caching fix (2.3.3).

## 3.3.1.b Turning Tracing On (LangSmith and Langfuse)

**Definition.** Both LangSmith and Langfuse capture traces from your app with **minimal code change**: for LangChain/LlamaIndex, tracing is largely **automatic** once configured (the frameworks emit spans through their callback/instrumentation system, 3.3.2); for arbitrary Python (raw SDK calls, custom logic), you add a lightweight **decorator or wrapper** to mark functions as traced.

**LangSmith — the first-party path.** LangSmith is LangChain's tightly-integrated observability platform. For a LangChain/LlamaIndex app you typically just set **environment variables** and traces start flowing — no code edits. For non-LangChain code, the **`@traceable`** decorator marks a function so its inputs/outputs/timing become a span. Because it's first-party, it captures LangChain internals richly and connects directly to LangSmith's **datasets and evaluations** (3.3.1.c).

**Langfuse — the open-source, self-hostable path.** Langfuse is a popular **open-source** platform you can **self-host**, which is often the deciding factor in regulated or data-sensitive environments (your prompts/outputs — which may contain PII or confidential data — never leave your infrastructure). It offers an SDK with an **`@observe`** decorator, native **LangChain/LlamaIndex integrations**, and increasingly **OpenTelemetry**-based ingestion, so it can capture traces from many frameworks (and non-LLM services) under one standard. It also bundles evals and prompt management.

**A note on OpenTelemetry (current-market direction).** The ecosystem is standardizing LLM telemetry on **OpenTelemetry (OTel)** with GenAI semantic conventions, so traces can flow to many backends in a vendor-neutral way. Both platforms are moving toward OTel compatibility. The practical upshot: knowing the trace/span *model* (3.3.1.a) transfers across tools, and you're less locked in than picking one proprietary format would suggest.

**Intuition — flip a switch, get an itemized flight recorder.** Adopting tracing isn't a rewrite; it's closer to flipping a switch (env vars) or clipping a small sensor onto each function (a decorator). Once on, every request is recorded to a flight recorder you can replay. The low effort is the point — there's rarely a good reason *not* to have tracing on in a serious project.

**Python — LangSmith (auto for LangChain) + `@traceable` for custom code:**
```python
import os
# Auto-trace ALL LangChain/LlamaIndex steps — no code changes to the chain itself:
os.environ["LANGSMITH_TRACING"] = "true"
os.environ["LANGSMITH_API_KEY"] = "ls-..."
os.environ["LANGSMITH_PROJECT"] = "finance-copilot"   # groups traces by project

# chain from 3.1 (prompt | model | parser) — invoking it now emits a full trace tree
result = chain.invoke({"topic": "duration risk"})

# For non-LangChain functions, mark them traceable to add spans:
from langsmith import traceable

@traceable(run_type="tool", name="calc_dv01")
def calc_dv01(position, duration):
    return position * duration * 0.0001      # this call now appears as a span
```

**Python — Langfuse (self-hosted friendly) with `@observe`:**
```python
from langfuse import observe, get_client
# Configured via env: LANGFUSE_HOST (your self-hosted URL), keys, etc.

@observe()                                   # wraps the function as a traced span
def answer_question(q: str) -> str:
    return chain.invoke({"topic": q})        # nested LangChain steps captured too

answer_question("convexity")
get_client().flush()                         # ensure spans are sent (batching)
```

**Interview angle.** "How do you enable tracing with minimal code?" (env vars for LangChain; `@traceable`/`@observe` for custom functions). "LangSmith vs Langfuse setup difference?" (first-party env-var auto-trace vs open-source/self-hosted SDK). "Why does OpenTelemetry matter?" (vendor-neutral standard; portability, less lock-in). "How are LangChain steps captured automatically?" (framework callbacks/instrumentation — 3.3.2).

**Finance use cases.**
1. **Self-hosted for compliance:** A bank runs **Langfuse self-hosted** so traced prompts/outputs (potentially containing customer data) stay within its network — observability without a data-egress violation.
2. **Custom step visibility:** Wrapping in-house risk-calc functions with `@traceable`/`@observe` makes those steps appear in the same trace as the LLM calls, giving end-to-end visibility.

**E-commerce use cases.**
1. **Fast prototype instrumentation:** A team ships a LangChain shopping assistant and turns on LangSmith with env vars to debug it during a launch — zero pipeline code changes.
2. **Unified multi-service tracing:** Via OpenTelemetry, LLM spans join the storefront's existing distributed traces, so an LLM step is visible alongside the checkout/inventory services.

## 3.3.1.c What You Do With Traces — Debug, Evaluate, Monitor

**Definition.** Capturing traces is the means; the value is in **using** them. The three core workflows the platforms support on top of traces are **debugging**, **evaluation**, and **monitoring** — turning raw span data into fixes, quality measurement, and operational awareness.

**Debugging (the daily workflow).** Open a problematic request, walk its span tree, and read the exact I/O of each step to find the culprit (3.3.1.a). Because traces are searchable by **tags/metadata**, you can filter to "failed requests for the finance-copilot feature last night" and inspect them. You can often **re-run** a step with a tweaked prompt/model from the trace to test a fix. This collapses debugging from guesswork to inspection.

**Evaluation (the quality flywheel — the highest-value use, foundation for Phase 4).** Both platforms let you build **datasets** (curated input/expected-output examples, often assembled *from real traces* — e.g., save interesting or failed requests into a dataset) and run **evaluations** that **score outputs** against references, rules, or an **LLM-as-judge**. You can run evals **offline** (test a change against a dataset before shipping) and **online** (score production traffic continuously). This is how you: measure RAG answer quality (faithfulness, relevance — Phase 4), safely compare models/prompts before swapping (3.1.3), and catch regressions. Evaluation is what makes quality *measurable and improvable* instead of vibes-based, and it's why observability is the backbone of the RAG evals coming in Phase 4.

**Monitoring (production awareness).** Dashboards over traces show **latency percentiles, error rates, token/cost trends, and quality-metric trends** over time, with **alerts** on regressions, cost spikes, or abuse. This is the operational layer that keeps a shipped feature healthy (and is developed fully in Phase 8's LLMOps). Because each trace carries cost/latency per step, monitoring here directly operationalizes the cost/rate discipline of 2.3.3.

**Why this trio is the whole point, explained.** Tracing without action is just storage. The debug→evaluate→monitor loop is the payoff: **debug** fixes today's bug, **evaluate** proves a change is actually better (and prevents regressions), and **monitor** catches tomorrow's problems in production. Together they form the "see → change → re-measure" loop from the 3.3 overview — and they all depend on the same span data, which is why one tracing setup unlocks all three.

**Intuition — from black-box to a lab notebook + control room.** Traces turn your app into a subject you can experiment on: debugging is examining a specimen, evaluation is running controlled experiments against a benchmark (dataset) to prove improvement, and monitoring is the control room watching the live system. You move from anecdotes ("it seemed better") to evidence ("faithfulness rose 8% on the eval set, cost fell 30%, p95 latency held").

**Interview angle.** "What are the three main things you do with traces?" (debug, evaluate, monitor). "How do you build an eval dataset?" (curate examples, often from real traces, with expected outputs). "Offline vs online evaluation?" (test a change pre-ship vs score live traffic). "What is LLM-as-judge?" (use an LLM to score outputs against criteria). "How does this enable safe model swaps and RAG tuning?" (measure quality/cost across versions on a dataset).

**Finance use cases.**
1. **Regression-gated changes:** Before updating a KYC-extraction prompt, run it against a trace-derived dataset and confirm field accuracy + disclaimer compliance didn't drop — evaluation catches regressions pre-ship (ties to 3.1.3).
2. **Production quality + cost monitoring:** Dashboards track answer-faithfulness scores and per-feature cost for an advisory bot, alerting on a quality dip or a cost spike.

**E-commerce use cases.**
1. **A/B evaluating prompts:** Two description-generation prompts are scored on the same dataset (brand-voice + accuracy) to pick the better one with evidence, not opinion.
2. **Live monitoring at scale:** During a sale, dashboards watch latency percentiles and error rates of the storefront assistant, alerting the team to degradation in real time.

# Wrap-Up: 3.3.1 Tracing with LangSmith / Langfuse

## The through-line (backward and forward)
Tracing captures each request as a **tree of spans** — every prompt, retrieval, model/tool call, and parse, with inputs, outputs, latency, tokens, cost, and errors (3.3.1.a) — and the tree's hierarchy is what lets you isolate the exact failing/expensive branch and read the *actual* prompt the model received. It's **low-friction to adopt**: env-var auto-tracing for LangChain/LlamaIndex, `@traceable`/`@observe` for custom code, with **LangSmith** (first-party, integrated) and **Langfuse** (open-source, self-hostable — the data-control choice) as the main platforms, increasingly unified by **OpenTelemetry** (3.3.1.b). The payoff is the **debug → evaluate → monitor** loop (3.3.1.c): fix bugs by inspection, prove improvements against **datasets** with **evals/LLM-as-judge**, and watch production health — all on the same span data. This directly operationalizes earlier discipline (cost/latency from 2.3.3, reliability from 2.2.3, safe swaps from 3.1.3) and is the backbone of what's ahead: **3.3.2** adds the config/caching/callbacks that *produce* and *control* these traces; **Phase 4 (RAG)** relies on this section's **evaluation** to tune retrieval; **Phase 5 (Agents)** relies on **traces** to debug multi-step loops; **Phase 8 (LLMOps)** scales the **monitoring**.

## Quick reference
| Aspect | LangSmith | Langfuse |
|---|---|---|
| Type | First-party (LangChain), hosted | Open-source, self-hostable or cloud |
| Enable (LangChain) | Env vars (`LANGSMITH_TRACING`, key, project) | Integration/callback + env config |
| Custom code | `@traceable` | `@observe` |
| Data control | Vendor-hosted (cloud) | **Self-host** → data stays in your infra |
| Extras | Datasets, evals, dashboards, prompt hub | Datasets, evals, prompt mgmt, OTel |
| Choose when | Deep LangChain integration, managed | Data sensitivity, open-source, multi-framework |

## Selected interview questions (short answers)
1. **What is a trace vs a span/run?** A trace = the full record of one request; spans/runs = individual steps forming a tree.
2. **Why is the span tree valuable?** It shows causality/containment so you can isolate the exact failing or costly step and read its real I/O.
3. **What does each span capture?** Inputs/outputs, latency, tokens & cost, metadata/tags, and errors/status.
4. **How do you enable tracing with little code?** Env vars for LangChain/LlamaIndex (auto); `@traceable`/`@observe` for custom functions.
5. **LangSmith vs Langfuse?** LangSmith = first-party, integrated, hosted; Langfuse = open-source, self-hostable (better data control).
6. **Why does self-hosting matter in finance?** Prompts/outputs may contain sensitive data; self-hosting keeps them in your environment.
7. **What is OpenTelemetry's role?** A vendor-neutral standard for LLM telemetry → portability and less lock-in.
8. **Three main uses of traces?** Debugging, evaluation, and monitoring.
9. **How do you build an eval dataset?** Curate input/expected-output examples, often harvested from real traces.
10. **Offline vs online eval; LLM-as-judge?** Test a change pre-ship vs score live traffic; use an LLM to grade outputs against criteria.

## Mini glossary
**Trace** — full structured record of one request.
**Span / run** — a single step within a trace (they form a tree).
**LangSmith** — LangChain's first-party tracing/eval platform.
**Langfuse** — open-source, self-hostable observability platform.
**@traceable / @observe** — decorators that add spans to custom code.
**OpenTelemetry (OTel)** — vendor-neutral telemetry standard.
**Dataset** — curated examples for evaluation (often from traces).
**Evaluation** — scoring outputs (reference/rule/LLM-as-judge).
**LLM-as-judge** — using an LLM to grade outputs.
**Monitoring** — dashboards/alerts over latency, cost, errors, quality.

## Hands-on
1. Enable LangSmith via env vars on a Phase-3 chain; invoke it and explore the span tree (expand each step's I/O, latency, tokens).
2. Self-host or sign up for Langfuse; instrument a raw-SDK function with `@observe` and confirm the trace appears.
3. Wrap a custom non-LLM function (e.g., a calc) with `@traceable`/`@observe` so it shows as a span in the same trace.
4. Save 10–20 real requests (including failures) into a dataset and run an evaluation (reference or LLM-as-judge) to get a quality score.
5. Compare two prompts (or two models) on that dataset and pick the winner by evidence — then set up a simple monitoring dashboard/alert.
