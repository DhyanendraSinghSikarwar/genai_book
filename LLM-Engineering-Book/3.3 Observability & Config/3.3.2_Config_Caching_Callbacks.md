## 3.3.2.0 Where this fits (the bridge from 3.3.1)

Tracing (3.3.1) told you *what happened*. This topic gives you the levers to *change what happens* — and the wiring that made tracing possible in the first place. The three subjects are deeply linked: **callbacks** are the event system that fires as a pipeline runs; **tracing is implemented as a callback handler** that records those events; and **config** is how you attach callbacks, tags, and runtime settings to a run. **Caching** rides alongside as the most direct cost/latency lever, and its effects show up *in* your traces. So this chapter both completes the observability loop (instrument → observe → tune) and hands you two of the biggest optimization tools (config-driven behavior and caching) you'll use for the rest of the course.

Everything here is about **operating** LLM systems well: keeping configuration out of code (so you can change behavior without redeploys), avoiding paying twice for the same work (caching), and hooking into the pipeline to log/measure/stream (callbacks). These are unglamorous but decisive — they're what separates a notebook demo from a maintainable, affordable production service, and they set up the LLMOps of Phase 8.

> **One-line thesis:** *Config steers a pipeline at runtime (settings, tags, callbacks) without touching its logic; caching stops you paying twice for identical work; callbacks are the event hooks that power logging, metrics, streaming — and tracing itself. Together they are the control-and-instrumentation layer under production LLM apps.*

## 3.3.2.a Configuration Management

**Definition.** **Configuration** is everything that controls a pipeline's behavior that you want to set **outside its core logic** — model choice and parameters, prompt versions, API keys/endpoints, feature flags, and per-run settings like tags, metadata, callbacks, concurrency, and recursion limits. In LangChain this operates at two levels: **application config** (secrets, environment, defaults — general software practice) and **run config** (`RunnableConfig` passed per invocation).

**Level 1 — application config (separate config from code).** A basic but critical software principle applies fully to LLM apps: **don't hard-code settings**. API keys come from the **environment / a secrets manager** (never source — the security theme from 2.1.3/2.2.3); model names, temperatures, prompt versions, vector-store URLs, and feature flags live in **config files or environment variables** so you can change them **without a code change or redeploy**. Tools like **`pydantic-settings`** load and *validate* config from env/files into typed objects (the same validation discipline as 2.2.3, applied to settings). This "12-factor" separation is what lets ops swap a model, raise a temperature, or point at a different vector DB per environment (dev/staging/prod) safely.

**Level 2 — run config (`RunnableConfig`).** LCEL chains accept a **`config`** argument per `invoke`/`batch`/`stream` that carries runtime controls without altering the chain:
- **`tags` and `metadata`** — labels attached to the run (and every child span), making **traces searchable/filterable** (3.3.1) by user, feature, prompt/model version. This is the practical glue between config and observability.
- **`run_name`** — a human-readable name for the run in traces.
- **`callbacks`** — attach handlers (3.3.2.c) for this run (logging, metrics, custom streaming).
- **`max_concurrency`** — cap parallelism for `batch` (ties to the concurrency control of 2.3.3).
- **`recursion_limit`** — bound how deep/looping a chain or agent may go (a guardrail against runaway loops — vital for agents, Phase 5).
- **`configurable`** — the runtime-swap mechanism from 3.1.3 (`configurable_fields`/`configurable_alternatives` + `.with_config`) to change model/params per call → model routing.

**Why config discipline matters, explained.** Configuration is where **flexibility and safety** come from. Externalized, validated config means you can respond to a cost spike (route to a cheaper model), a bad prompt (roll back a version), or an incident (flip a feature flag) **in minutes without shipping code** — and the tags/metadata you set flow into traces so you can *see* the effect. Conversely, config baked into code means every behavior change is a deploy, and un-tagged runs are hard to find when debugging. Good config is the quiet backbone of operability.

**Intuition — the settings panel vs rewiring the appliance.** Config is the appliance's control panel: you change temperature, mode, and timers from the outside. Baking settings into code is like soldering the dial to one position — to change it you open the device. Run config (`RunnableConfig`) is a per-use override (this load gets a delicate cycle and a label), while application config is the factory defaults and safety limits. You want to operate from the panel, not the soldering iron.

**Python — run config: tags/metadata (for traces), limits, and runtime model swap:**
```python
# Per-run config: does NOT change the chain's logic, only how this run behaves + is labeled
result = chain.invoke(
    {"topic": "duration risk"},
    config={
        "run_name": "explain-topic",
        "tags": ["prod", "finance-copilot"],
        "metadata": {"user_id": "analyst-42", "prompt_version": "v3"},  # searchable in traces
        "max_concurrency": 5,            # for .batch(...)
        "recursion_limit": 10,           # guard loops (agents)
    },
)

# Application config with validation (pydantic-settings): env/file -> typed, validated object
from pydantic_settings import BaseSettings
class Settings(BaseSettings):
    model_name: str = "gpt-5.5"
    temperature: float = 0.0
    openai_api_key: str                  # required; loaded from env, never hard-coded
settings = Settings()                    # reads env / .env, validates types
```

**Interview angle.** "Why separate config from code?" (change behavior without redeploy; safety/flexibility; secrets out of source). "What does `RunnableConfig` carry?" (tags, metadata, callbacks, run_name, max_concurrency, recursion_limit, configurable). "How does config connect to observability?" (tags/metadata make traces searchable). "What is `recursion_limit` for?" (bound runaway loops — agents). "How do you validate config?" (`pydantic-settings`).

**Finance use cases.**
1. **Environment-specific config:** Dev uses a cheap model and a test vector store; prod uses a stronger model and the real store — same code, different validated config per environment.
2. **Incident-time rollback:** A bad prompt version is rolled back via config (no deploy), and traces tagged `prompt_version` confirm the fix — fast, auditable remediation.

**E-commerce use cases.**
1. **Feature-flagged rollout:** A new description-generation prompt is enabled for 10% of traffic via config, with `metadata` tags letting the team compare its traces/costs against the old one.
2. **Per-tenant settings:** A multi-brand storefront loads brand-specific model/prompt config per tenant without branching the codebase.

## 3.3.2.b Caching

**Definition.** **Caching** stores the result of an expensive operation so an identical (or similar) future request can be served from the cache instead of re-running it — cutting **cost and latency** (the levers from 2.3.3). In LLM apps there are three main caches to know: **LLM response caching**, **embedding caching**, and **provider-side prompt caching**, plus the distinction between **exact** and **semantic** caching.

**The kinds of caching, explained.**
- **LLM response cache (exact match).** Store `(prompt, params) → response`; a repeated identical call returns instantly and free. Enabled globally in LangChain via `set_llm_cache(...)` (in-memory, SQLite, Redis, etc.). Best for **deterministic, repeated** requests. *Caveat:* it keys on exact input, so any change (even whitespace) misses; and it's only safe when you *want* the same answer (temperature 0 / stable tasks) — you wouldn't cache a "generate a fresh creative tagline" call.
- **Semantic cache.** Instead of exact match, retrieve a cached answer when a new query is **semantically similar** to a past one (using embeddings + a similarity threshold; e.g., GPTCache, Redis semantic cache). Far higher hit rates for natural-language queries ("what's your return policy?" vs "how do I return an item?"). *Caveat:* a too-loose threshold returns a subtly wrong cached answer — tune carefully.
- **Embedding cache.** Embedding the *same* text repeatedly (common when re-indexing or on overlapping documents) is wasteful; **`CacheBackedEmbeddings`** stores `text → vector` so you don't re-embed unchanged content — a real cost saver during indexing (3.2.1) and RAG.
- **Provider-side prompt caching.** Distinct from client caching: providers can cache a **stable prompt prefix** (long system prompt, few-shot block, reference docs) and bill it at a steep discount on subsequent calls (2.3.3.b). You "use" it by keeping the reusable content at the front of the prompt and enabling the feature — often the single biggest cost win for RAG/system-prompt-heavy apps.

**Why caching is high-leverage but must be applied thoughtfully, explained.** LLM calls are among the most expensive and slowest operations in your stack, so *not paying for the same work twice* is pure win — for repeated FAQs, deterministic extractions, or shared prompt prefixes, caching can cut cost and latency dramatically. But caching trades **freshness and correctness for speed/cost**: cache something that should vary (creative output, time-sensitive data) and you serve stale/wrong results; set a semantic threshold too loosely and you return a near-miss answer. The discipline is to **cache what is stable and repeated, invalidate when the source changes, and keep thresholds tight** — and to *verify hit rates and correctness in your traces* (3.3.1), which is exactly why caching lives in the same section as observability.

**Intuition — a memoized answer sheet.** Caching is memoization for LLM work: the first time someone asks a question you do the expensive computation and write the answer on a sheet; the next identical (exact cache) or similar (semantic cache) question is answered by reading the sheet. Embedding cache is not re-measuring a document you've already measured; provider prompt caching is the vendor keeping your long standing instructions "warm" so you don't re-send them at full price. The art is knowing which answers are safe to reuse and for how long.

**Python — LLM cache, embedding cache, and the semantic-cache idea:**
```python
# 1) Exact-match LLM response cache (global) — great for deterministic, repeated calls
from langchain_core.globals import set_llm_cache
from langchain_core.caches import InMemoryCache
set_llm_cache(InMemoryCache())          # or SQLiteCache / RedisCache for persistence
# now identical chain.invoke(...) calls return instantly & free (visible in traces)

# 2) Embedding cache — avoid re-embedding unchanged text during indexing (3.2.1)
from langchain.embeddings import CacheBackedEmbeddings
from langchain.storage import LocalFileStore
from langchain_openai import OpenAIEmbeddings
store = LocalFileStore("./emb_cache")
cached_embedder = CacheBackedEmbeddings.from_bytes_store(
    OpenAIEmbeddings(model="text-embedding-3-small"), store, namespace="v1")

# 3) Semantic cache (concept): serve a cached answer when a new query is similar enough
# from langchain_community.cache import RedisSemanticCache
# set_llm_cache(RedisSemanticCache(redis_url=..., embedding=..., score_threshold=0.1))
```

**Interview angle.** "Name the kinds of LLM caching." (exact response, semantic, embedding, provider prompt caching). "Exact vs semantic cache trade-off?" (precise-but-brittle vs high-hit-but-risk-of-near-miss). "When is caching unsafe?" (creative/varying output, time-sensitive data; loose semantic thresholds). "What's the biggest cost win for system-prompt-heavy apps?" (provider prompt caching of the prefix). "How do you verify caching works?" (hit rate/correctness in traces).

**Finance use cases.**
1. **Cached policy corpus prefix:** Provider prompt caching of a fixed regulatory prefix cuts input cost ~90% across millions of compliance queries (2.3.3).
2. **Deterministic extraction cache:** Re-processing the same documents hits an exact-match cache, avoiding paid re-computation during pipeline reruns.

**E-commerce use cases.**
1. **Semantic FAQ cache:** Common storefront questions phrased many ways hit a semantic cache, slashing cost and latency at high traffic (with a tight threshold to avoid wrong answers).
2. **Embedding reuse on re-index:** `CacheBackedEmbeddings` avoids re-embedding unchanged catalog text when the index is rebuilt, cutting indexing cost.

## 3.3.2.c Callbacks

**Definition.** **Callbacks** are hooks that fire on **events during a pipeline's execution** — when a chain/model/tool/retriever **starts**, **ends**, **errors**, or (for streaming) **emits a new token**. You implement a **callback handler** (in LangChain, subclass `BaseCallbackHandler`) with methods like `on_llm_start`, `on_llm_new_token`, `on_llm_end`, `on_chain_start/end`, `on_tool_start/end`, `on_retriever_end`, `on_llm_error`, and attach it via config. Callbacks are how you **observe and react to** what a pipeline is doing *as it runs*, without changing its logic.

**The crucial connection — callbacks *are* how tracing works (tying 3.3.1 together).** The tracing platforms from 3.3.1 don't magically see your pipeline; they register a **callback handler** that captures each `on_*_start`/`on_*_end` event (with inputs, outputs, timing, tokens) and ships it as a span. So "tracing is auto-enabled for LangChain" really means "a tracing callback is registered." Understanding callbacks demystifies observability and lets you build your **own** instrumentation when the built-ins aren't enough.

**What you use callbacks for, explained.**
- **Custom logging & metrics.** Emit your own logs or push metrics (latency, token counts, error rates) to your monitoring stack on each event — useful when you need telemetry beyond what a tracing platform provides, or to your existing systems.
- **Streaming.** The `on_llm_new_token` event is how token-by-token streaming (2.3.2) is delivered to a UI or forwarded downstream — streaming handlers are callbacks.
- **Cost/token accounting.** A callback can accumulate token usage across a run (e.g., LangChain's token-counter callback) to tally cost precisely (2.3.3).
- **Guardrails & side effects.** React to events — e.g., on a tool start, check authorization; on an error, alert; on completion, persist an audit record (compliance).
- **Progress/UX.** Update a progress indicator as steps complete.

**Sync vs async, and scope (practical notes).** Callbacks come in sync and **async** variants (`AsyncCallbackHandler`) — use async in async pipelines/servers so handlers don't block. Callbacks can be attached **globally**, **per-chain**, or **per-run** (via `config["callbacks"]`), letting you scope instrumentation appropriately (e.g., verbose logging only in dev). Keep handlers **fast and failure-isolated** — a slow or throwing callback can degrade the request it's observing, so heavy work (network writes) should be batched/async.

**Intuition — event listeners for your pipeline (like DOM/UI events).** If you've used UI event listeners (`onClick`, `onLoad`), callbacks are the same idea for an LLM pipeline: register a listener for "model started", "new token", "tool finished", "step errored", and run your code when it happens. Tracing is just a very thorough listener that writes everything down; your custom handlers are listeners tuned to what *you* care about (a metric, a log line, a stream to the browser). The pipeline stays unchanged; you observe and react from the outside.

**Python — a minimal custom callback handler + attaching it per run:**
```python
from langchain_core.callbacks import BaseCallbackHandler

class TokenAndTimingLogger(BaseCallbackHandler):
    def on_llm_start(self, serialized, prompts, **kwargs):
        print("LLM start:", prompts[0][:60], "...")
    def on_llm_new_token(self, token, **kwargs):
        print(token, end="", flush=True)          # streaming (2.3.2)
    def on_llm_end(self, response, **kwargs):
        usage = getattr(response, "llm_output", None)
        print("\nLLM end. usage:", usage)          # token/cost accounting (2.3.3)
    def on_tool_start(self, serialized, input_str, **kwargs):
        print("TOOL start:", serialized.get("name"), input_str)  # guardrail hook point
    def on_llm_error(self, error, **kwargs):
        print("LLM error:", error)                 # alerting hook

# Attach per-run via config (also how tracing callbacks are attached under the hood):
chain.invoke({"topic": "convexity"},
             config={"callbacks": [TokenAndTimingLogger()]})
```

**Interview angle.** "What are callbacks and when do they fire?" (event hooks on start/end/error/new-token for chain/model/tool/retriever). "How is tracing implemented?" (as a callback handler capturing those events → spans). "What do you use callbacks for?" (logging, metrics, streaming, token/cost accounting, guardrails, progress). "Sync vs async callbacks?" (use async in async apps; keep handlers fast/non-blocking). "How do you scope callbacks?" (global/per-chain/per-run via config).

**Finance use cases.**
1. **Compliance audit callback:** An `on_llm_end`/`on_tool_end` handler persists every prompt, output, and tool action to an immutable audit log — regulatory traceability built via callbacks.
2. **Real-time cost guard:** A callback accumulates token usage and trips an alert/circuit-breaker if a single request exceeds a cost threshold (2.3.3/2.2.3).

**E-commerce use cases.**
1. **Streaming to the storefront:** An `on_llm_new_token` callback forwards tokens to the browser for the typing effect (2.3.2), scoped per user session.
2. **Custom metrics pipeline:** Callbacks push per-step latency/token metrics to the team's existing dashboards, complementing the tracing platform.
