## 3.2.2.0 Where this fits (the bridge from 3.2.1)

You now have an **Index** full of retrievable **Nodes** (3.2.1). But an index is inert — it's a structure, not an application. The **engine** is what turns "I have an index" into "I can ask questions and get answers." LlamaIndex offers two engine families, and the choice between them is one of the clearest "know your requirements" decisions in RAG:

- If each request is **independent** — a fact lookup, an extraction, a report generated over your data — you want a **Query Engine**. It's stateless: same question in, same answer out, no memory of prior turns.
- If the user is having a **conversation** — following up, saying "and what about last year?", referring back to earlier answers — you want a **Chat Engine**. It's stateful: it remembers the dialogue and resolves references against it.

This is the same stateless-vs-stateful distinction you met at the raw-API level (2.3.1) and solved with memory in LangChain (3.1.2), now packaged into LlamaIndex's two front doors. Understanding what each does *internally* (retrieve → post-process → synthesize, plus history handling for chat) also demystifies the "magic" one-liners `as_query_engine()` and `as_chat_engine()`.

> **One-line thesis:** *A Query Engine answers one question over your data (stateless); a Chat Engine holds a conversation over your data (stateful, with memory). Both retrieve relevant Nodes and synthesize an answer — the Chat Engine just adds conversation history and reference resolution on top.*

## 3.2.2.a Query Engines — Stateless Question Answering

**Definition.** A **Query Engine** is the interface for asking a **single, self-contained question** over an index and getting back a **synthesized, grounded answer** (with source Nodes for citation). It is **stateless**: it has no memory of previous questions, so each query is handled independently. You create one from an index with `index.as_query_engine(...)`.

**What happens inside a query (the pipeline, made explicit).** The one-line `query_engine.query("...")` hides a three-step pipeline you already met in the 3.2 overview — and each step is a customization point:
1. **Retrieval.** A **retriever** fetches the top-k most relevant Nodes from the index for the query (by embedding similarity for a `VectorStoreIndex`). You control `similarity_top_k`, metadata filters, and the retrieval mode.
2. **Post-processing (optional).** **Node post-processors** filter or **re-rank** the retrieved Nodes — e.g., drop low-similarity nodes, apply a re-ranker for precision, or expand to neighbor nodes for context. This step is a major Phase-4 quality lever.
3. **Response synthesis.** A **response synthesizer** passes the (post-processed) Nodes + the question to the LLM and produces the final answer. *How* it combines multiple nodes is governed by the **response mode**.

**Response modes — how multiple chunks become one answer (worth knowing precisely).** When retrieval returns several Nodes, the synthesizer must fuse them:
- **`compact`** (default) — stuff as many retrieved Nodes as fit into one prompt and answer in a single call (efficient; the common choice).
- **`refine`** — answer using the first Node, then iteratively **refine** that answer with each subsequent Node (more calls, handles many nodes, higher cost/latency).
- **`tree_summarize`** — recursively summarize Nodes in a tree until one answer remains (good for "summarize a lot of material" queries).
Choosing the mode trades **cost/latency vs thoroughness** — `compact` for most Q&A, `tree_summarize` for broad summarization.

**Grounding and citations — the point of it all.** The Query Engine's response carries **`source_nodes`**: the exact chunks used to answer. This is what makes RAG *trustworthy* — you can show the user (or an auditor) where the answer came from, and it's the basis of the grounding/hallucination guardrails from 2.2.3. A Query Engine answer without traceable sources defeats the purpose of RAG.

**Intuition — a research assistant answering one memo at a time.** You hand the assistant a question; they go to the catalog, pull the few relevant sections (retrieval), discard the ones that aren't actually on point (post-processing), read them, and write a cited answer (synthesis) — then forget the whole thing and start fresh on your next, unrelated question. That deliberate amnesia is the Query Engine's defining trait: perfect for independent tasks, wrong for a running conversation.

**Python — a customized Query Engine with the pipeline exposed:**
```python
from llama_index.core import VectorStoreIndex
from llama_index.core.postprocessor import SimilarityPostprocessor

# index built in 3.2.1
query_engine = index.as_query_engine(
    similarity_top_k=5,                 # retrieval: how many Nodes to fetch
    response_mode="compact",            # synthesis strategy
    node_postprocessors=[SimilarityPostprocessor(similarity_cutoff=0.7)],  # drop weak nodes
    streaming=True,                     # stream the answer (2.3.2)
)

resp = query_engine.query("What liquidity risks were disclosed in FY2025?")
print(resp)                             # grounded answer
for sn in resp.source_nodes:            # citations / traceability
    print(sn.score, sn.node.metadata.get("file_name"))
```

**Interview angle.** "What is a Query Engine and is it stateful?" (single-question, stateless). "What three steps run inside a query?" (retrieve → post-process → synthesize). "What are response modes and their trade-offs?" (`compact` vs `refine` vs `tree_summarize`). "Why do `source_nodes` matter?" (grounding/citations).

**Finance use cases.**
1. **One-shot disclosure lookup:** An analyst asks a specific, independent question over the filings index and gets a cited answer — no conversation needed, so a stateless Query Engine is the right, cheaper choice.
2. **Automated report generation:** A batch job runs many independent queries (one per required section) over the corpus with `tree_summarize` to synthesize broad summaries, each with sources for review.

**E-commerce use cases.**
1. **Spec fact retrieval:** "What's the battery capacity of model X?" is a standalone lookup answered by a Query Engine straight from the product index.
2. **Bulk FAQ generation:** Generating cited answers for a list of common questions is a set of independent queries — no memory required.

## 3.2.2.b Chat Engines — Stateful Conversation

**Definition.** A **Chat Engine** is the interface for a **multi-turn conversation** over your data. Unlike a Query Engine, it is **stateful**: it maintains **conversation history (memory)** so it can understand follow-ups, resolve references ("it", "that year", "the second one"), and keep context across turns — all while still retrieving from your index to stay grounded. You create one with `index.as_chat_engine(chat_mode=...)`.

**The core problem a Chat Engine solves (and why it's non-trivial).** In conversation, questions become **context-dependent**. If the user first asks "What were Acme's 2025 liquidity risks?" and then "How does that compare to 2024?", the second message is meaningless in isolation — "that" refers to the prior answer, and a naive retriever would embed "How does that compare to 2024?" and retrieve nothing useful. The Chat Engine's job is to **fuse the conversation history with the new message** so retrieval and answering work correctly. *How* it does that fusion is the **chat mode**, and the different modes are essentially different strategies for combining memory with retrieval.

**Chat modes — the key design choice (know the main ones).**
- **`condense_question`** — first **rewrite** the follow-up into a standalone question using the chat history (e.g., "How does that compare to 2024?" → "How do Acme's 2025 liquidity risks compare to 2024?"), then run that through a normal retrieval+synthesis pipeline. Great for pure Q&A follow-ups; it always retrieves, but can feel less natural for chit-chat.
- **`context`** — retrieve relevant Nodes for the raw message and put them, plus the chat history, into the system prompt every turn. Simple and conversational; retrieves each turn regardless.
- **`condense_plus_context`** — the popular hybrid: **condense** the follow-up into a standalone query *and* inject retrieved **context** + history. Robust across both follow-up questions and conversational turns — a common default.
- **`react` / `best`** — agentic modes where the LLM **decides** whether/what to retrieve using tools (ReAct, 2.1.2/2.2.2) — more flexible, more overhead; a bridge toward agents (Phase 5).

**Memory in a Chat Engine (connecting to 3.1.2).** The Chat Engine holds a **chat memory buffer** (recent turns), subject to the same **context-window and cost** limits you learned in 3.1.2/1.2.5 — so long conversations need trimming or summarizing, exactly as with LangChain memory. The Chat Engine packages this for you, but the trade-offs (completeness vs window/cost) are identical; you're choosing how much history to carry.

**Intuition — the same research assistant, now remembering the meeting.** A Chat Engine is the Query Engine's assistant who *stays in the room*: they remember what you asked five minutes ago, understand "what about the other one?" without you re-explaining, and still go back to the catalog for each new sub-question to keep answers grounded. The `condense_question` mode is the assistant quietly rephrasing your shorthand into a full question before searching; the `context` mode is them keeping the relevant documents open on the desk throughout the chat. Same retrieval muscle, plus a memory.

**Python — a Chat Engine with a robust mode:**
```python
chat_engine = index.as_chat_engine(
    chat_mode="condense_plus_context",   # condense follow-ups + inject context + history
    similarity_top_k=5,
    verbose=True,
)

print(chat_engine.chat("What liquidity risks did Acme disclose in FY2025?"))
# Follow-up: "that"/"2024" resolved against history, then retrieved + answered:
print(chat_engine.chat("How does that compare to the prior year?"))

chat_engine.reset()                      # clear conversation memory to start fresh
# Streaming variant: chat_engine.stream_chat("...") -> stream tokens (2.3.2)
```

**Interview angle.** "Query Engine vs Chat Engine — the key difference?" (stateless single-shot vs stateful conversation with memory). "Why do follow-up questions break naive retrieval, and how does a Chat Engine fix it?" (context-dependent references; condense into standalone query / inject history). "Name chat modes and trade-offs." (`condense_question`, `context`, `condense_plus_context`, `react`). "How does chat memory relate to the context window?" (must trim/summarize long histories).

**Finance use cases.**
1. **Interactive diligence session:** An analyst explores a filing conversationally — "what are the risks?", "which is largest?", "how did it change YoY?" — with the Chat Engine resolving each follow-up against history while staying grounded in sources.
2. **Advisory chatbot with continuity:** A client-facing bot remembers earlier questions in the session (risk tolerance, holdings mentioned) so later answers stay consistent — with memory trimming to control cost on long chats.

**E-commerce use cases.**
1. **Conversational shopping assistant:** "Show me noise-cancelling earbuds" → "which is cheapest?" → "does *that* one have a warranty?" — the Chat Engine tracks "that one" across turns while retrieving product facts.
2. **Multi-turn support:** A customer troubleshooting step-by-step gets context-aware follow-ups grounded in help articles, without repeating their setup each message.

## 3.2.2.c Choosing and Composing (query vs chat, and beyond)

**The decision rule (simple and interview-ready).** Ask: **does the user's next request depend on the previous ones?**
- **No → Query Engine.** Independent lookups, extractions, batch report sections, API endpoints that take a full question each time. Stateless is simpler, cheaper, and easier to cache/parallelize.
- **Yes → Chat Engine.** Interactive assistants, exploratory sessions, anything with follow-ups and references. Stateful, at the cost of managing memory.

**A key relationship — a Chat Engine is (roughly) a Query Engine + memory + reference resolution.** They share the same retrieval and synthesis core; the Chat Engine wraps it with conversation handling. This is why you can often prototype with a Query Engine and "upgrade" to a Chat Engine when you add a conversational UI — the underlying index and retrieval settings carry over unchanged.

**Composing beyond a single index — a nod to what's next.** Real systems often need more than one index or more than plain Q&A. LlamaIndex supports **routing** (a router picks the right query engine/index for a question — e.g., "filings" vs "policies"), **sub-question query engines** (break a complex question into sub-queries across indexes and combine), and **query engines exposed as tools** to an **agent** (Phase 5) that decides when to query your data versus call another tool. These are the same composition ideas you saw in LangChain (routing, tools) applied to LlamaIndex — and they're the on-ramp from "a RAG engine" to "an agent that uses RAG." For now, the core takeaway is that both engines are **building blocks** that plug into larger systems, not just endpoints.

**Intuition — a form to fill vs a conversation to have vs a team to coordinate.** A Query Engine is a form: one complete question, one answer. A Chat Engine is a conversation: back-and-forth with memory. And when you have many data sources and tools, you coordinate them with routers/sub-question engines/agents — a team of specialists, each fronted by an engine. You escalate up this ladder only as your requirements demand; most features start at "form" or "conversation."

**Interview angle.** "How do you decide query vs chat engine?" (does the request depend on prior turns?). "How is a chat engine related to a query engine?" (query engine + memory + reference resolution). "How do you handle multiple indexes or complex questions?" (router / sub-question query engines / expose engines as agent tools). "When would a RAG engine become an agent tool?" (when the app needs to decide among data + other actions — Phase 5).

**Finance use cases.**
1. **Router over multiple corpora:** A router query engine sends "what's our KYC threshold?" to the policy index and "Acme's 2025 revenue?" to the filings index — right source, right answer.
2. **Sub-question decomposition:** "Compare Acme and Beta on leverage and liquidity" is split into sub-queries per company/metric and combined into one grounded comparison.

**E-commerce use cases.**
1. **Query engine as an agent tool:** A support agent (Phase 5) treats the help-center query engine as one tool among several (order lookup, returns), calling it only when a knowledge question arises.
2. **Routed catalog vs policy questions:** A router directs product-spec questions to the catalog index and returns/shipping questions to the policy index.

# Wrap-Up: 3.2.2 Query Engines vs Chat Engines

## The through-line (backward and forward)
An index (3.2.1) becomes an application through an **engine**. A **Query Engine** (3.2.2.a) answers one **independent** question via retrieve → post-process → synthesize (with response modes `compact`/`refine`/`tree_summarize` trading cost vs thoroughness, and `source_nodes` for grounding). A **Chat Engine** (3.2.2.b) adds **memory and reference resolution** for multi-turn conversation, using chat modes (`condense_question`, `context`, `condense_plus_context`, `react`) to fuse history with retrieval — subject to the same window/cost limits as any memory (3.1.2/1.2.5). The decision rule is simply *whether the next request depends on prior ones* (3.2.2.c), and both engines are **composable building blocks** — routers, sub-question engines, and "engine-as-agent-tool" — that lead directly into **agents (Phase 5)**. This completes LlamaIndex's core querying surface; **3.2.3** compares LlamaIndex with LangChain (and when to use both), and **Phase 4 (RAG)** optimizes every stage — retrieval, re-ranking, response synthesis, evaluation — that these engines orchestrate.

## Quick reference
| | Query Engine | Chat Engine |
|---|---|---|
| State | Stateless (independent) | Stateful (remembers history) |
| Use for | Lookups, extraction, batch/report Q&A | Interactive assistants, follow-ups |
| Create | `index.as_query_engine(...)` | `index.as_chat_engine(chat_mode=...)` |
| Core pipeline | retrieve → post-process → synthesize | same + history/reference resolution |
| Key settings | `similarity_top_k`, `response_mode`, post-processors | `chat_mode`, memory buffer, `similarity_top_k` |
| Modes | `compact`/`refine`/`tree_summarize` | `condense_question`/`context`/`condense_plus_context`/`react` |
| Grounding | `source_nodes` (citations) | `source_nodes` + memory |

## Selected interview questions (short answers)
1. **Query vs Chat Engine — key difference?** Stateless single-question answering vs stateful multi-turn conversation with memory.
2. **What runs inside a query?** Retrieve top-k Nodes → optional post-process/re-rank → synthesize an answer.
3. **Response modes and trade-offs?** `compact` (one call, efficient), `refine` (iterate over nodes, thorough/costly), `tree_summarize` (recursive summary for broad queries).
4. **Why do follow-ups break naive retrieval?** They're context-dependent ("that", "last year"); embedding them alone retrieves nothing useful.
5. **How does a Chat Engine fix that?** Fuses history — e.g., `condense_question` rewrites the follow-up into a standalone query; `context` injects history+retrieved nodes.
6. **Name chat modes.** `condense_question`, `context`, `condense_plus_context` (robust default), `react`/`best` (agentic).
7. **How is a chat engine related to a query engine?** Roughly a query engine + memory + reference resolution (shared retrieval/synthesis core).
8. **Why do `source_nodes` matter?** Grounding and citations — the basis of trust and hallucination guardrails (2.2.3).
9. **How do chat memory limits show up?** Long histories exceed the context window / raise cost, so trim or summarize (3.1.2/1.2.5).
10. **How do you handle multiple indexes/complex questions?** Router query engine, sub-question query engine, or expose engines as agent tools (Phase 5).

## Mini glossary
**Query Engine** — stateless single-question interface over an index.
**Chat Engine** — stateful conversational interface with memory.
**Retriever** — fetches top-k relevant Nodes.
**Node post-processor** — filters/re-ranks retrieved Nodes.
**Response synthesizer** — combines Nodes + query into an answer.
**Response mode** — `compact`/`refine`/`tree_summarize` fusion strategy.
**Chat mode** — `condense_question`/`context`/`condense_plus_context`/`react`.
**source_nodes** — the chunks used to answer (citations).
**Router query engine** — picks the right engine/index per query.
**Sub-question query engine** — splits a complex query across indexes.
