## 3.2.1.0 Where this fits (the bridge from 3.2)

The overview said the indexing stage is "load → Documents → Nodes → embed → Index." This chapter *is* that arrow, unpacked. It matters more than it looks, because of a principle that governs all of RAG: **the LLM can only answer as well as the chunks you retrieve, and you can only retrieve chunks as good as the ones you created during indexing.** If a Node is too big, it dilutes relevance and wastes tokens; too small, it loses the context needed to be meaningful; badly split, it cuts a sentence or a table in half and the answer breaks. So the humble decisions of *how you load data* and *how you cut it into Nodes* are, quietly, among the highest-leverage choices in a RAG system — far more than which fancy model you pick.

This is why LlamaIndex gives Documents, Nodes, and Indexes first-class treatment, and why we study them before query engines (3.2.2): **you cannot query well over a poorly indexed corpus.** Everything downstream inherits the quality set here.

> **One-line thesis:** *A Document is your raw data; a Node is a retrievable, metadata-rich chunk of it; an Index is the structure that finds the right Nodes fast. Retrieval quality is decided at indexing time — chunking and metadata are where RAG is won or lost.*

## 3.2.1.a Documents — Loading Your Data

**Definition.** A **Document** in LlamaIndex is a container for a **single source of data** together with its **metadata**. A source could be a PDF, a web page, a row from a database, a Notion page, a Slack export, an API response — anything. The Document holds the raw text (and optionally other modalities) plus a metadata dictionary (filename, author, date, URL, custom tags). Documents are the **input** to the indexing stage.

**How Documents are created — connectors.** You rarely build Documents by hand. LlamaIndex provides **data loaders / connectors** (many via **LlamaHub**, its connector registry) that read a source and return Documents. The simplest is **`SimpleDirectoryReader`**, which loads a folder of files (PDFs, Word, Markdown, etc.) into Documents automatically. There are hundreds of connectors for databases, SaaS apps, and web sources — this is how LlamaIndex "connects LLMs to your data" in practice: pick a loader, point it at the source, get Documents.

**Why metadata is not an afterthought (a key point).** Every Document (and later, every Node derived from it) carries metadata, and this metadata is **usable at query time for filtering**. If each filing Document is tagged with `{"fiscal_year": 2025, "doc_type": "10-K", "company": "Acme"}`, a query can restrict retrieval to *only* 2025 Acme 10-Ks — dramatically improving precision and preventing cross-document contamination. Metadata set at load time is therefore a retrieval-quality lever, not just bookkeeping. A common best practice is to enrich metadata during loading (add source, section, date, access-control tags) because it pays off in every subsequent query.

**The parsing-quality problem (current market, important for finance/enterprise).** The *quality of the text extracted* from a source hugely affects everything downstream. Naive PDF loaders mangle tables, multi-column layouts, and scanned documents — catastrophic when the numbers in a table are the whole point. This is why LlamaIndex offers **LlamaParse** (a managed, high-fidelity parser for complex documents) and why "just load the PDF" is often insufficient for real financial or enterprise data. We revisit ingestion depth in Phase 4; for now, know that **a Document is only as good as the parse that produced it.**

**Intuition — Documents are the books arriving at the library.** Before anything can be catalogued or read, the books have to arrive and be logged with their basic info (title, author, acquisition date). A Document is a book plus its intake record. If a book arrives with pages torn or text smudged (bad parsing), no amount of clever cataloguing later will fix the missing information — which is why intake (loading + parsing + metadata) matters so much.

**Python — loading Documents and enriching metadata:**
```python
from llama_index.core import SimpleDirectoryReader

# Load a folder of files into Documents (one+ Document per file)
documents = SimpleDirectoryReader("./filings").load_data()

# Inspect / enrich metadata (usable for filtering at query time)
doc = documents[0]
print(doc.text[:200])          # the raw extracted text
print(doc.metadata)            # {'file_name': 'acme_10k_2025.pdf', ...}
doc.metadata.update({"doc_type": "10-K", "fiscal_year": 2025, "company": "Acme"})

# For complex PDFs (tables/scans), a high-fidelity parser matters:
# from llama_cloud_services import LlamaParse
# documents = LlamaParse(result_type="markdown").load_data("./filings/acme_10k_2025.pdf")
```

**Interview angle.** "What is a Document in LlamaIndex?" (a data source + metadata). "How are Documents created?" (loaders/connectors, e.g., `SimpleDirectoryReader`, LlamaHub). "Why does metadata matter at load time?" (query-time filtering → precision). "Why is parsing quality a concern?" (bad extraction ruins retrieval; LlamaParse for complex docs).

**Finance use cases.**
1. **Tagged filings corpus:** Loading 10-Ks with `fiscal_year`/`company`/`doc_type` metadata lets analysts scope questions to a specific company-year, avoiding answers contaminated by the wrong period.
2. **Table-faithful ingestion:** Using LlamaParse on statement PDFs preserves the numeric tables so downstream figure extraction is accurate rather than garbled.

**E-commerce use cases.**
1. **Catalog + manuals with metadata:** Loading product docs tagged by `brand`/`category`/`sku` enables category-scoped retrieval for precise product answers.
2. **Multi-source help center:** Connectors pull articles from a CMS plus a ticketing system into one Document set, each tagged by source for filtered, trustworthy support answers.

## 3.2.1.b Nodes — The Unit of Retrieval

**Definition.** A **Node** is a **chunk of a Document** — a smaller, self-contained piece of text with its own **metadata** and **relationships** to other nodes (previous/next, parent Document, etc.). LlamaIndex converts Documents into Nodes during ingestion (via **node parsers / text splitters**), and **retrieval operates on Nodes, not Documents.** The Node is therefore the true atomic unit of a RAG system: what gets embedded, what gets fetched, and what the LLM ultimately reads as context.

**Why we chunk at all (connecting to 1.2.5).** Two reasons force chunking. First, the **context window** is finite (1.2.5) — you can't feed a whole 300-page document to the model, so you must retrieve *pieces*. Second, **retrieval precision**: embedding a huge document into one vector blurs its meaning (it's "about everything"), so similarity search can't pinpoint the relevant part. Splitting into focused Nodes means each vector represents a specific idea, and retrieval can surface exactly the passage that answers the query. Chunking is what makes semantic retrieval *work*.

**The central trade-off — chunk size (this is the crux of RAG quality).** How big should a Node be? It's a genuine tension:
- **Too large:** each Node covers multiple topics, so its embedding is unfocused (worse retrieval precision), it wastes tokens/cost when injected, and it can push out room for other relevant chunks.
- **Too small:** a Node loses surrounding context and may be meaningless alone (a sentence fragment, a table cell without its header), so even if retrieved it doesn't help the LLM answer.

The right size is corpus- and task-dependent, and finding it is one of the main tuning activities of Phase 4. LlamaIndex exposes this via `Settings.chunk_size` and node-parser parameters. A widely used refinement is **chunk overlap** — letting adjacent Nodes share a bit of text at their boundaries — so an idea split across a boundary isn't lost by either chunk.

**Node parsers / splitters — how chunks are made.** LlamaIndex offers several strategies, and the choice affects quality:
- **`SentenceSplitter`** (common default) — splits on sentence boundaries up to a target size with overlap, avoiding mid-sentence cuts.
- **Token/text splitters** — split by token or character counts.
- **Structure-aware splitters** — split by Markdown headers, code, or document structure, so Nodes respect natural sections (great for keeping a table or a clause intact).
- **Semantic / hierarchical splitters** — split by semantic similarity or build parent/child node hierarchies (small child chunks for precise retrieval, larger parent chunks for context — the "small-to-big" pattern revisited in Phase 4).

**Node relationships and metadata.** Because Nodes remember their **relationships** (which Document they came from, their neighbors) and carry inherited + custom **metadata**, retrieval can do smart things: pull a node *and* its neighbors for more context, filter by metadata, or trace an answer back to its exact source location (citations). This relational structure is why LlamaIndex Nodes are richer than a naive "list of text chunks."

**Intuition — Nodes are the catalogued, shelved sections of a book.** A whole book (Document) is too much to hand someone who asked a specific question. So the librarian breaks it into labeled sections and index cards (Nodes) — each focused enough to be *findable* and self-contained enough to be *useful*. Cut the sections too coarsely and a card points at 50 pages about many topics; too finely and a card is a single sentence with no context. The art of good cataloguing (chunking) is exactly the art of good RAG.

**Python — turning Documents into Nodes with a splitter:**
```python
from llama_index.core import Settings
from llama_index.core.node_parser import SentenceSplitter

# Global chunking config (used by high-level index building):
Settings.chunk_size = 512
Settings.chunk_overlap = 50

# Or explicitly parse Documents into Nodes with a chosen splitter:
splitter = SentenceSplitter(chunk_size=512, chunk_overlap=50)
nodes = splitter.get_nodes_from_documents(documents)

n = nodes[0]
print(n.get_content()[:200])   # the chunk text
print(n.metadata)              # inherited + custom metadata (used for filtering)
print(n.relationships)         # links to source Document / neighbor nodes
```

**Interview angle.** "What is a Node and why is it the unit of retrieval?" "Why chunk documents at all?" (context window + retrieval precision). "The chunk-size trade-off?" (too big = unfocused/costly; too small = loses context). "What is chunk overlap for?" "Name node-parser strategies." (sentence, token, structure-aware, semantic/hierarchical). "How do node relationships/metadata help?" (neighbor expansion, filtering, citations).

**Finance use cases.**
1. **Clause-preserving chunking:** A structure-aware splitter keeps each contractual/risk clause intact in one Node, so retrieval returns complete, correct provisions rather than half-clauses.
2. **Small-to-big for precision + context:** Small child Nodes give precise matches on a specific figure, while their larger parent Node supplies surrounding context to the LLM for a correct explanation.

**E-commerce use cases.**
1. **Spec-table integrity:** Chunking that respects table/section structure keeps a product's spec block together, so "what's the battery capacity?" retrieves the right, complete spec.
2. **Right-sized FAQ chunks:** Sentence-splitting help articles into focused Nodes with overlap yields precise, self-contained answers to specific support questions.

## 3.2.1.c Indexes — Structuring Nodes for Retrieval

**Definition.** An **Index** is a data structure built over your Nodes that enables **efficient retrieval** at query time. Different index types organize and retrieve Nodes differently; you choose based on your query pattern. The default and most common is the **`VectorStoreIndex`**, which embeds each Node into a vector and retrieves by **semantic similarity**.

**The main index types and when each fits (breadth worth knowing).**
- **`VectorStoreIndex` (semantic search) — the default and the RAG workhorse.** Each Node is embedded; a query is embedded; retrieval returns the top-k most similar Nodes. Best for "find the passages most relevant to this question" — i.e., almost all RAG. Backed by an in-memory store by default or an external **vector database** (Chroma, Qdrant, Pinecone, pgvector — Phase 4).
- **Summary / List Index.** Stores Nodes in a simple sequence and can iterate over *all* of them — useful when a query needs the *whole* corpus (e.g., "summarize everything") rather than a targeted lookup.
- **Tree Index.** Builds a hierarchy of summaries (leaf Nodes → parent summaries), traversed top-down — good for hierarchical summarization over large corpora.
- **Keyword Table Index.** Maps keywords to Nodes for keyword-based retrieval — useful when exact term matching matters (and a component of hybrid search in Phase 4).
- **Property-Graph Index.** Represents entities and relationships as a graph, enabling relationship-aware / multi-hop retrieval (the LlamaIndex path toward GraphRAG, Phase 4).

**Why the vector index dominates, and how it connects forward.** Because most real questions are "what does my data say about X?", semantic similarity (embeddings) is the natural fit, so `VectorStoreIndex` is the default you'll use ~90% of the time. Its quality depends on the pieces below it — the **embedding model** (how well meaning is captured) and the **vector store** (how the vectors are stored and searched at scale) — both of which are Phase 4 topics. Building the index is where the offline "prepare the knowledge base" work culminates: after this, you have a queryable structure and can attach a query/chat engine (3.2.2).

**Persistence (a practical necessity).** Building an index (especially embedding thousands of Nodes) costs time and money, so you don't want to rebuild it every run. LlamaIndex lets you **persist** an index to disk or to an external vector store and **reload** it, so indexing happens once and querying is cheap thereafter — the offline/online split from the overview made concrete.

**Intuition — the index is the library's catalog system.** Once books are broken into shelved, labeled sections (Nodes), you need a system to *find* the right ones fast. A vector index is a "meaning-based catalog": ask a question and it points to the sections whose *ideas* match, even if the words differ. A keyword index is the old card catalog by exact term; a tree index is a hierarchy of section summaries; a graph index is a map of how entities relate. You pick the catalog that matches how people will search — and for most questions, the meaning-based one wins.

**Python — building, persisting, and reloading a `VectorStoreIndex`:**
```python
from llama_index.core import (VectorStoreIndex, StorageContext,
                              load_index_from_storage, Settings)
from llama_index.embeddings.openai import OpenAIEmbedding

Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")

# --- Build the index from Documents (chunks -> Nodes -> embeddings -> index) ---
index = VectorStoreIndex.from_documents(documents)      # or .from_nodes(nodes)

# --- Persist so you don't rebuild/re-embed every run ---
index.storage_context.persist(persist_dir="./storage")

# --- Reload later (cheap) ---
storage = StorageContext.from_defaults(persist_dir="./storage")
index = load_index_from_storage(storage)

# The index is now ready to answer queries (3.2.2):
# query_engine = index.as_query_engine()
```

**Interview angle.** "What is an Index and why do you need one?" (efficient retrieval over Nodes). "Name index types and when to use each." (vector=semantic RAG; summary=whole-corpus; tree=hierarchical summary; keyword=exact terms; graph=relationships). "Why is `VectorStoreIndex` the default?" "What determines vector-index quality?" (embedding model + vector store — Phase 4). "Why persist an index?" (avoid costly re-embedding).

**Finance use cases.**
1. **Semantic filings search:** A `VectorStoreIndex` over a company's filings lets analysts find relevant disclosures by meaning ("liquidity concerns") even when the document uses different wording.
2. **Relationship queries via graph index:** A property-graph index over entities (issuers, subsidiaries, guarantors) supports multi-hop questions like "which subsidiaries guarantee this note?".

**E-commerce use cases.**
1. **Meaning-based product search:** A vector index retrieves relevant products/docs for "budget noise-cancelling earbuds" despite differing catalog wording.
2. **Whole-catalog summaries:** A summary index answers "what product lines do we cover?" by iterating over the corpus rather than a targeted lookup.

# Wrap-Up: 3.2.1 Documents, Nodes & Indexes

## The through-line (backward and forward)
The indexing stage from the 3.2 overview is: **Documents** (raw data + metadata, created by connectors and only as good as the parse — 3.2.1.a) → **Nodes** (metadata-rich, related chunks that are the true unit of retrieval, whose **chunk size/overlap and splitter choice decide RAG quality** — 3.2.1.b) → **Index** (the structure that retrieves the right Nodes, with `VectorStoreIndex`/semantic search as the default among summary/tree/keyword/graph options — 3.2.1.c). The governing principle: **retrieval quality is set at indexing time**, which is why chunking and metadata are the highest-leverage decisions and why we build the index (and persist it) before ever querying. This is the offline half of LlamaIndex; **3.2.2** covers the online half — the **query and chat engines** that retrieve from this index and synthesize answers — and all of it is optimized stage-by-stage in **Phase 4 (RAG)**, where embeddings, vector databases, chunking strategies, and re-ranking get the full treatment.

## Quick reference
| Concept | What it is | Key API / knob |
|---|---|---|
| Document | Raw data source + metadata | `SimpleDirectoryReader`, LlamaHub, LlamaParse |
| Metadata | Tags on Docs/Nodes for filtering | `doc.metadata`, metadata filters |
| Node | Retrievable chunk of a Document | node parsers; `chunk_size`, `chunk_overlap` |
| Node parser | Splits Docs → Nodes | `SentenceSplitter`, structure/semantic splitters |
| VectorStoreIndex | Semantic (embedding) index — default | `from_documents(...)` |
| Summary/Tree/Keyword/Graph index | Whole-corpus / hierarchical / exact-term / relationships | choose by query pattern |
| Persistence | Save/reload index | `storage_context.persist`, `load_index_from_storage` |

## Selected interview questions (short answers)
1. **Document vs Node?** A Document is a raw source + metadata; a Node is a retrievable chunk of it — retrieval happens on Nodes.
2. **Why chunk documents?** Fit the context window (1.2.5) and improve retrieval precision (focused embeddings).
3. **The chunk-size trade-off?** Too big = unfocused/expensive; too small = loses context; overlap prevents boundary loss.
4. **Name node-parser strategies.** Sentence, token/text, structure-aware (Markdown/code), semantic/hierarchical (small-to-big).
5. **Why does metadata matter?** Enables query-time filtering (e.g., by year/company) → precision and citations.
6. **Why is parsing quality important?** Bad extraction (mangled tables/scans) ruins retrieval; use LlamaParse for complex docs.
7. **What is an Index and the main types?** A retrieval structure; vector (semantic, default), summary/list, tree, keyword, property-graph.
8. **Why is `VectorStoreIndex` the default?** Most queries are semantic "what does my data say about X?".
9. **What determines vector-index quality?** Chunking + embedding model + vector store (Phase 4).
10. **Why persist an index?** Embedding is costly; build once, reload cheaply (offline/online split).
