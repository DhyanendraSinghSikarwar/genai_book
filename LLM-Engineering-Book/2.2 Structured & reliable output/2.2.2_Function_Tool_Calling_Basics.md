## 2.2.2.0 Where this fits (the bridge from 2.2.1 and 2.1.2)

Two threads from earlier converge here:
- From **2.1.2 (ReAct)**: we saw that reasoning is far more powerful when the model can **act** — call a search engine, a calculator, a database — and read the **observation**. But we hand-crafted that loop with fragile text parsing.
- From **2.2.1 (Structured Outputs)**: we learned to make the model emit **schema-valid JSON** reliably.

**Function calling is the productized fusion of the two.** The provider exposes a first-class way to (a) tell the model which functions exist (as JSON schemas), (b) let the model **decide** when to call one and emit **schema-valid arguments**, and (c) feed the function's result back so the model can continue. It fixes Phase-1 weaknesses directly: **stale knowledge** (call live data), **bad arithmetic** (call code), and **no side effects** (call APIs to actually *do* things).

> **One-line thesis:** *Function calling turns the LLM from a text generator into a controller: it reads your tool schemas, chooses a tool, emits valid arguments, and you execute — the model orchestrates, your code acts. Master this loop and agents are just this loop, repeated, with more tools.*

## 2.2.2.a What Tool Calling Is — and the Core Loop

**Definition.** **Function (tool) calling** lets you register a set of **tools** — each described by a name, a natural-language description, and a **JSON-Schema for its parameters** — with the model. Given a user request, the model may respond not with prose but with a **structured tool call**: `{"name": "...", "arguments": {...}}`. **Your application executes the function**, returns the result to the model, and the model uses it to produce the final answer (or call more tools).

**Crucial mental model (interview essential):** *The model does not run your code.* It only **emits a request** to call a function with arguments. **You** execute it in your environment and pass the result back. This separation is what makes tool calling safe(ish) and controllable — and why *you* own validation, auth, and side effects.

**The canonical loop (recite this):**
```
1. You send: user message + tool definitions (schemas)
2. Model responds with either:
     (a) a normal text answer, OR
     (b) one or more tool calls (name + JSON args)
3. If tool call(s): YOUR code executes the function(s)
4. You append the tool result(s) to the conversation
5. Model reads the result(s) → answers or calls more tools
6. Repeat 2–5 until the model returns a final text answer
```
This is exactly the **ReAct loop** (2.1.2.c), now with structured, provider-native calls instead of hand-parsed text.

**Intuition.** Think of the model as a smart dispatcher that knows *what* it needs but not *how* to get it. You hand it a toolbox (schemas); it points at the right tool and fills in the form (arguments); you operate the tool; it interprets the result. The **descriptions and schemas are the model's entire understanding** of each tool — so they must be clear.

**Interview angle.** "Walk through the tool-calling loop." "Does the model execute the function?" (no — it requests; you execute). "How does tool calling relate to ReAct and to agents?" "What makes a good tool description/schema?"

**Finance use cases.**
1. **Live market data:** A `get_quote(ticker)` tool lets the model answer "what's AAPL's price?" with real data instead of stale weights.
2. **Account actions:** A `get_balance(account_id)` / `transfer(...)` tool turns a chatbot into a banking assistant that performs real (authorized, confirmed) operations.

**E-commerce use cases.**
1. **Order status:** An `get_order_status(order_id)` tool grounds "where's my package?" in the live orders API.
2. **Inventory & pricing:** A `check_inventory(sku)` tool gives accurate stock/price answers rather than guesses.

## 2.2.2.b Defining and Executing Tools (the code)

**Definition — a tool spec** has three parts the model sees: **name** (what to call), **description** (when/why to use it — the model's usage guide), and **parameters** (a JSON Schema, ideally generated from Pydantic per 2.2.1).

**Python — end-to-end tool calling with OpenAI (the standard pattern):**
```python
import json
from openai import OpenAI
client = OpenAI()

# 1) The actual function your code runs
def get_order_status(order_id: str) -> dict:
    # real impl would hit your DB/API; stubbed here
    return {"order_id": order_id, "status": "shipped", "eta": "2026-07-05"}

# 2) The tool schema the MODEL sees (JSON Schema for params)
tools = [{
    "type": "function",
    "function": {
        "name": "get_order_status",
        "description": "Look up the current status and ETA of a customer order by its ID.",
        "parameters": {
            "type": "object",
            "properties": {"order_id": {"type": "string", "description": "e.g. 'A12345'"}},
            "required": ["order_id"],
            "additionalProperties": False,
        },
    },
}]
AVAILABLE = {"get_order_status": get_order_status}

def run(user_msg: str) -> str:
    messages = [{"role": "user", "content": user_msg}]
    while True:
        resp = client.chat.completions.create(
            model="gpt-5.5", messages=messages, tools=tools, temperature=0,
        )
        msg = resp.choices[0].message
        if not msg.tool_calls:                      # (a) plain answer -> done
            return msg.content
        messages.append(msg)                        # record the assistant's tool call(s)
        for call in msg.tool_calls:                 # (b) execute each requested tool
            args = json.loads(call.function.arguments)     # schema-valid JSON (see 2.2.1)
            result = AVAILABLE[call.function.name](**args) # YOUR code runs it
            messages.append({                       # feed the observation back
                "role": "tool",
                "tool_call_id": call.id,
                "content": json.dumps(result),
            })
        # loop: model now reads the tool result and continues

print(run("Where is my order A12345 and when will it arrive?"))
```

**Python — the same idea with Anthropic (tool_use / tool_result blocks):**
```python
from anthropic import Anthropic
a = Anthropic()
tools = [{
    "name": "get_order_status",
    "description": "Look up order status/ETA by order ID.",
    "input_schema": {"type": "object",
                     "properties": {"order_id": {"type": "string"}},
                     "required": ["order_id"]},
}]
messages = [{"role": "user", "content": "Status of order A12345?"}]
resp = a.messages.create(model="claude-opus-4-8", max_tokens=512,
                         tools=tools, messages=messages)
# If resp.stop_reason == "tool_use": run the tool, then send a
# {"role":"user","content":[{"type":"tool_result","tool_use_id":...,"content":...}]} turn.
```

**Python — LangChain (`@tool`) auto-generates the schema from type hints/docstring:**
```python
from langchain_core.tools import tool

@tool
def get_order_status(order_id: str) -> dict:
    """Look up the status and ETA of an order by its ID."""   # becomes the description
    return {"order_id": order_id, "status": "shipped"}
# model = llm.bind_tools([get_order_status])  # schema derived automatically (Phase 3)
```

**Best practices (interview gold).**
- **Descriptions are prompts:** write the tool `description` and parameter descriptions as carefully as any prompt — they decide whether/when the model calls correctly.
- **Keep tools few and orthogonal:** too many overlapping tools confuse selection; group or route if you have many (Phase 5).
- **Generate schemas from Pydantic** for a single source of truth + validation.
- **Validate arguments before executing** (never trust args blindly — types, ranges, allow-lists).
- **Return concise, structured results** (the model pays tokens to read them — 1.2.6); include an `error` field on failure so the model can recover.
- **Set a step/loop limit** so a misbehaving model can't call tools forever.

**Interview angle.** "What are the three parts of a tool definition?" "Why generate the schema from Pydantic?" "How do you handle a tool that fails?" (return an error result; let the model retry or apologize; cap retries). "Why cap the loop?"

**Finance use cases.**
1. **Grounded advisory bot:** `get_quote`, `get_portfolio`, `calc_returns` tools let the assistant compute with real numbers and code, not tokenized math (1.2.7).
2. **Validated transactions:** Argument validators + confirmation gate ensure a `transfer` tool call is checked (amount, payee allow-list) before execution.

**E-commerce use cases.**
1. **Multi-tool shopping assistant:** `search_catalog`, `check_inventory`, `apply_coupon`, `create_cart` compose a real buying flow.
2. **Automated support:** `lookup_order`, `initiate_return`, `issue_refund` (with confirmation) resolve tickets end-to-end.

## 2.2.2.c Control, Parallelism, Streaming & Security

**2.2.2.c.i Controlling *whether* a tool is called — `tool_choice`.**
- **`auto`** (default): model decides (call a tool or answer).
- **`required`/`any`**: force it to call *some* tool.
- **force a specific tool** (e.g., `{"type":"function","function":{"name":"emit_json"}}`): guarantees a call — this is also the trick for **structured extraction** (2.2.1.b uses a forced tool).
- **`none`**: forbid tool calls (plain text only).

**2.2.2.c.ii Parallel tool calls.** Modern models can request **multiple tool calls in one turn** (e.g., fetch two orders at once). Execute them (optionally concurrently), return all results, then continue. Know that this exists and that your loop must handle a **list** of calls, not just one.

**2.2.2.c.iii Sequential / multi-hop.** The model may call tool A, read the result, then decide to call tool B (e.g., search → then calculate). This chaining *is* the agent behavior of Phase 5.

**2.2.2.c.iv Streaming.** Tool-call arguments can be streamed token-by-token (useful for UX/latency), but you must **buffer until the arguments JSON is complete** before executing.

**2.2.2.c.v Security & reliability (critical — 2026 focus).**
- **The model's tool call is untrusted input.** **Validate & authorize** every argument server-side; never `eval` args or build SQL/shell directly from them (injection).
- **Least privilege:** give tools the minimum scope; separate read tools from state-changing tools.
- **Human-in-the-loop for high-impact actions** (payments, deletes, transfers): require explicit confirmation.
- **Prompt-injection via tool results / retrieved content** (indirect injection, recall 2.1.3.c): a malicious document or API response can try to hijack the model into calling dangerous tools — treat tool results as **data, not instructions**, and gate side effects.
- **Guardrails:** loop limits, timeouts, rate limits, audit logging of every tool call (compliance).
- **OWASP LLM Top 10** explicitly flags "excessive agency" — don't grant tools more power than the use case needs.

**Python — argument validation + confirmation gate (defensive pattern):**
```python
from pydantic import BaseModel, field_validator

class TransferArgs(BaseModel):
    to_account: str
    amount: float
    @field_validator("amount")
    @classmethod
    def cap(cls, v):
        if not (0 < v <= 10_000): raise ValueError("amount out of allowed range")
        return v

def execute_transfer(raw_args: dict, user_confirmed: bool):
    args = TransferArgs(**raw_args)                 # validate & type-check
    if args.to_account not in ALLOWED_PAYEES:       # allow-list
        return {"error": "payee not allowed"}
    if not user_confirmed:                          # human-in-the-loop gate
        return {"status": "awaiting_confirmation", "details": args.model_dump()}
    return do_transfer(args)                         # only now perform the side effect
```

"How do you secure tool calling?" (validate/authorize args, least privilege, human-in-the-loop, treat results as data, loop limits, audit). "What is 'excessive agency'?"

1. **Guarded money movement:** Validation + allow-list + confirmation gate prevent an injected instruction from triggering an unauthorized `transfer`.
2. **Parallel data gathering:** Fetch quote + balance + FX rate in parallel to answer a portfolio question with one round-trip.

**E-commerce use cases.**
1. **Safe refunds:** `issue_refund` requires confirmation + amount caps; audited for dispute handling.

# Wrap-Up: 2.2.2 Function / Tool Calling

## The through-line (backward and forward)
Tool calling fuses **ReAct** (act + observe, 2.1.2) with **Structured Outputs** (schema-valid arguments, 2.2.1): you register tools as schemas, the model **chooses** one and emits **valid arguments**, **your code executes**, and the result flows back through the **canonical loop** (2.2.2.a–b). You steer it with `tool_choice`, handle **parallel/sequential** calls and **streaming**, and — most importantly — **secure** it (validate args, least privilege, human-in-the-loop, treat tool results as untrusted data) (2.2.2.c). This is the atom of agents: **an agent is this loop run repeatedly with memory, planning, and many tools (Phase 5)**; RAG becomes just a `search` tool (Phase 4); and **MCP (Phase 5)** standardizes how tools/servers are exposed to models.

## The tool-calling loop (memorize)
`user + tool schemas → model (text | tool_call[s]) → your code executes → append result(s) → model reads → answer | more calls → repeat (with a step cap)`

## Expanded interview Q&A bank (short answers)
1. **What is function/tool calling?** A protocol where the model, given tool schemas, emits structured calls (name + valid args) that your app executes and feeds back.
2. **Does the model run the function?** No — it only *requests* a call; your code executes it. This separation is key to control/security.
3. **What are the three parts of a tool definition?** Name, description (when to use), and JSON-Schema parameters.
4. **Walk the tool-calling loop.** Send message + tools → model returns text or tool calls → execute → append results → model continues → repeat until a text answer.
5. **How does tool calling relate to ReAct?** It's the productized ReAct loop with native structured calls instead of parsed text.
6. **How does it relate to agents?** Agents are this loop repeated with planning, memory, and many tools.
7. **Why are tool descriptions important?** They're the model's only guide to *when* and *how* to call — effectively prompts.
8. **Why generate schemas from Pydantic?** Single source of truth + argument validation (2.2.1).
9. **What is `tool_choice` and its options?** Controls calling: `auto`, `required`/`any`, force a specific tool, or `none`.
10. **How do you force structured extraction with tools?** Define one emit-tool and force `tool_choice` to it.
11. **What are parallel tool calls?** The model requests several tools in one turn; your loop must execute a list and return all results.
12. **What is multi-hop tool use?** Model calls a tool, reads it, then calls another (e.g., search → calculate).
13. **How do you handle a failing tool?** Return a structured `error`; let the model retry/apologize; cap retries.
14. **Why cap the loop?** To prevent infinite/expensive tool-calling loops from a misbehaving model.
15. **How do you handle streaming tool calls?** Buffer streamed argument tokens until the JSON is complete, then execute.
16. **Why is a tool call untrusted input?** The model can be wrong or injected; validate/authorize args server-side.
17. **Name key tool-calling security practices.** Validate + authorize args, least privilege, separate read vs write tools, human-in-the-loop for high-impact actions, audit logging, loop/timeout limits.
18. **What is indirect prompt injection via tools?** Malicious instructions inside a tool result/retrieved content that try to hijack the model; treat results as data, gate side effects.
19. **What is "excessive agency" (OWASP)?** Giving the model tools/permissions beyond what the task needs — a top LLM risk.
20. **How do you keep tool results cheap?** Return concise, structured results (tokens cost money — 1.2.6).
21. **How many tools is too many?** Enough overlap/volume to confuse selection; group, route, or use retrieval over tools (Phase 5).
22. **How does RAG fit tool calling?** Retrieval is exposed as a `search`/`retrieve` tool the model can call.
23. **What is MCP's relationship to tool calling?** MCP standardizes how tools/resources are exposed to any model/app (Phase 5).
24. **How do you test tool calling?** Eval whether the model picks the right tool with correct args on labeled scenarios; test failure/edge cases.
25. **When should the model NOT use a tool?** When it can answer directly; `auto` lets it decide, or set `none`.
26. **How do you confirm high-impact actions?** Return an `awaiting_confirmation` state and require explicit user approval before the side effect.
27. **Can tool calling reduce hallucination?** Yes — grounding answers in live tool/data results instead of memory.
28. **Difference between structured output and tool calling?** Structured output returns typed *data*; tool calling returns a request to *act*, then continues with the result.

## Mini glossary
**Tool / function** — a callable your app exposes to the model.
**Tool schema** — name + description + JSON-Schema parameters.
**Tool call** — model's structured request `{name, arguments}`.
**Tool result / observation** — the executed function's return fed back.
**`tool_choice`** — control over whether/which tool is called.
**Parallel tool calls** — multiple calls in one turn.
**Human-in-the-loop** — required approval before high-impact actions.
**Excessive agency** — over-privileged tools/permissions (OWASP risk).
**Indirect injection** — malicious instructions arriving via tool/retrieved content.
**Least privilege** — grant tools minimal scope.
