## 2.2.3.0 Where this fits (the bridge from 2.2.1–2.2.2)

Think of an LLM call as an **unreliable external service** that happens to speak natural language. In classical backend engineering you would never trust such a service blindly: you'd **validate its responses**, **retry transient failures**, and **contain the blast radius** when it misbehaves. LLMs need exactly this discipline, plus two extra concerns that ordinary services don't have: their output is **non-deterministic** (1.3.5) and can be **adversarially manipulated** (prompt injection, 2.1.3/2.2.2).

So the reliability stack for an LLM call has three layers, and they map onto the three parts of this chapter:

1. **Validation** — *Is the output well-formed and does it satisfy my rules?* (Pydantic)
2. **Retries** — *If not, can I recover automatically?* (re-prompt with the error; backoff)
3. **Guardrails** — *Is the whole exchange safe, on-policy, and grounded?* (input + output checks, moderation, PII, hallucination/grounding)

These layers are **complementary, not alternatives**. A mature pipeline uses all three. The reason this matters is stark: at scale, "usually correct" is a failure. If your extraction is right 98% of the time but you process a million documents, that's 20,000 broken records silently entering your systems. Validation + retries + guardrails are how you push that error rate down and, crucially, **fail loudly and safely** instead of quietly.

> **One-line thesis:** *Structured outputs guarantee the JSON parses; validation, retries, and guardrails guarantee it's correct, recoverable, and safe. This is the layer that separates a demo from a system you'd put in front of customers or regulators.*

## 2.2.3.a Output Validation with Pydantic

**Definition.** **Validation** is the process of checking that a model's output satisfies both its **structure** (correct fields and types) and its **semantics** (values obey your business rules), and rejecting or correcting it if not. **Pydantic** is the de-facto Python library for this: you declare a data model as a class, and Pydantic parses raw JSON into a typed object, coercing and checking as it goes, raising a precise error when something is off.

**Why Pydantic specifically, and why it's central to the whole LLM stack.** Three reasons make Pydantic the backbone rather than just "a validation library":

First, it is a **single source of truth**. The same Pydantic class (a) generates the JSON Schema you send to the model for structured outputs or tool calls (2.2.1/2.2.2), (b) parses and validates the response, and (c) gives you a typed Python object your IDE and the rest of your code understand. You define the contract once; every layer uses it.

Second, it distinguishes **two kinds of correctness** that people often conflate. **Structural** validity ("there is a field `amount` and it's a number") is what schema-constrained decoding already gives you. **Semantic** validity ("`amount` is positive, the currency is one we support, the maturity date is after the issue date") is a *different* problem that no decoder can guarantee, because it depends on your domain, not on JSON grammar. Pydantic is where you encode semantic rules, through **field validators** (checks on one field) and **model validators** (checks across multiple fields).

Third, when validation fails, Pydantic produces a **structured, human-readable error** that names the offending field and the reason. That error is not just for your logs — it's the exact text you feed back to the model on a retry (2.2.3.b), which is what makes automatic self-correction work.

**Intuition — validation as a contract enforcer.** Imagine the model is a contractor filling in a form. Structured outputs guarantee they use the right form with the right boxes. Validation is the clerk who then checks the *content* of the boxes: the phone number has the right number of digits, the total isn't negative, the date isn't in the year 3000. The clerk doesn't rewrite the form — they either accept it or hand it back with a precise note about what's wrong. That "precise note" is the linchpin of the whole reliability loop.

**A worked walk-through.** Consider extracting a bond record. Schema-constrained decoding (2.2.1) can guarantee you get `{"issuer": str, "coupon_rate": number, "issue_date": str, "maturity_date": str, "currency": str}`. But it cannot guarantee that `coupon_rate` is a sensible fraction rather than a percentage the model wrote as `5.0` instead of `0.05`, that `currency` is one your ledger supports, or that `maturity_date` comes *after* `issue_date`. Those are semantic invariants, and each is a place where a wrong-but-well-formed value would corrupt downstream systems. Pydantic lets you state each invariant declaratively and turns a violation into a catchable, describable error.

**Python — structural + semantic validation in one model:**
```python
from datetime import date
from typing import Literal
from pydantic import BaseModel, Field, field_validator, model_validator

class BondRecord(BaseModel):
    issuer: str = Field(min_length=1)
    coupon_rate: float = Field(ge=0, le=1, description="annual coupon as a DECIMAL, e.g. 0.05")
    issue_date: date
    maturity_date: date
    currency: Literal["USD", "EUR", "GBP", "INR"]   # enum = structural + semantic

    # --- field validator: rule about ONE field ---
    @field_validator("coupon_rate")
    @classmethod
    def rate_is_fraction_not_percent(cls, v: float) -> float:
        # If the model returned 5.0 meaning "5%", flag it rather than booking 500%.
        if v > 1:
            raise ValueError("coupon_rate looks like a percent; express as a decimal (0.05)")
        return v

    # --- model validator: rule ACROSS fields ---
    @model_validator(mode="after")
    def maturity_after_issue(self):
        if self.maturity_date <= self.issue_date:
            raise ValueError("maturity_date must be after issue_date")
        return self

# Parsing raw model JSON:
raw = '{"issuer":"Acme","coupon_rate":0.05,"issue_date":"2025-06-01",' \
      '"maturity_date":"2030-06-01","currency":"USD"}'
bond = BondRecord.model_validate_json(raw)   # -> typed, fully validated object
```

**What each construct buys you, explained.** `Field(ge=0, le=1)` is a **constraint** — a lightweight structural+range check Pydantic enforces automatically. `Literal[...]` turns `currency` into an **enum**, so an unsupported currency is rejected at parse time (and, if you put the same model in the request schema, the model is *steered* to only emit those values in the first place). The `field_validator` catches a subtle, common model mistake (percent vs decimal) that would otherwise be a silent, expensive bug. The `model_validator(mode="after")` runs *after* individual fields are validated and can reason about **relationships** between fields — the kind of rule that is impossible to express in a JSON Schema alone. Together they convert "the JSON parsed" into "the data is actually usable."

**A note on coercion (and when to turn it off).** Pydantic will, by default, coerce compatible types — the string `"0.05"` becomes the float `0.05`. This is usually helpful with LLM output, which is stringy by nature. But in strict domains you may want `strict=True` on a field to *reject* a string where a number is required, forcing the model (via retry) to emit the correct type. The judgment call — lenient coercion for robustness vs strict typing for rigor — is itself a design decision worth making consciously.

**Finance use cases.**
1. **Booking-grade extraction:** Field/model validators enforce that amounts are positive, currencies are supported, and dates are internally consistent before a record is written to a ledger — turning a plausible-looking hallucination into a caught error rather than a bad trade.
2. **Decision records with invariants:** A credit decision object can enforce that `confidence ∈ [0,1]`, that `reasons` is non-empty when `decision == "deny"`, and that a required disclaimer field is present — invariants a regulator would expect to hold on every record.

**E-commerce use cases.**
1. **Catalog integrity:** Validators reject a product record whose `price <= 0`, whose `size` isn't in the allowed set, or whose `discount_price > price`, so malformed enrichment never reaches the storefront.
2. **Order parsing safety:** A parsed order enforces `quantity >= 1`, a known `sku` pattern, and a shipping country in your served-regions list before it touches the fulfilment API.

## 2.2.3.b Retries & Self-Correction

**Definition.** A **retry** is an automatic re-attempt of a failed LLM call. There are two fundamentally different failure types, and conflating them is a common mistake: **transport failures** (the API timed out, rate-limited, or 5xx'd — the model never really "answered") and **content failures** (the model answered, but the output failed validation). They need different remedies.

**Transport failures → backoff.** These are the same failures any network service has, and the same classical remedy applies: **retry with exponential backoff and jitter**. You wait a little, then a bit more, adding randomness so many clients don't retry in lockstep and hammer the API simultaneously. You cap the number of attempts and, ideally, only retry **idempotent** operations. The `tenacity` library is the standard tool. The key insight is that transport retries should *not* change the prompt — the request was fine; the network wasn't.

**Content failures → re-prompt with the error (self-correction).** This is the LLM-specific and more interesting case, and it's why 2.2.3.a's "precise error message" mattered so much. When validation fails, you don't just blindly re-ask the same question — you **append the validation error to the conversation and ask the model to fix its previous answer**. Because the model now sees *exactly* what was wrong ("coupon_rate looks like a percent; express as a decimal"), it can usually correct itself on the next attempt. This closed loop — generate → validate → on failure, feed the error back → regenerate — is the single most effective reliability pattern in applied LLM work, and it's what libraries like `instructor` automate.

**Intuition — the model as a junior analyst who responds well to specific feedback.** If you tell a junior analyst "that's wrong, redo it," you get another guess. If you tell them "the maturity date must be after the issue date, and yours isn't," they fix the actual problem. Specific, structured feedback dramatically outperforms a bare retry. The whole trick of self-correction is that a validation error *is* specific, structured feedback — you already generated it in 2.2.3.a, so you might as well hand it back.

**Why you still need limits and fallbacks.** Retrying is not free: each attempt costs tokens, latency, and money (1.2.6), and some outputs will *never* validate — the request may be genuinely ambiguous, the schema too strict, or the model too weak for the task. So every retry loop needs a **cap** (e.g., 2–3 attempts) and a **fallback**: escalate to a stronger/more expensive model, relax or simplify the schema, return a safe default, or route to a **human-in-the-loop**. A pipeline that retries forever is as broken as one that never retries — it just fails more expensively. This is the LLM version of the classic **circuit breaker**: after repeated failures, stop trying and degrade gracefully rather than melting down.

**Python — content retry with `instructor` (automatic self-correction):**
```python
import instructor
from openai import OpenAI

client = instructor.from_openai(OpenAI())

bond = client.chat.completions.create(
    model="gpt-5.5",
    response_model=BondRecord,   # the validated model from 2.2.3.a
    max_retries=3,               # on validation error, instructor re-prompts WITH the error
    messages=[{"role": "user", "content": "Extract the bond fields: <document...>"}],
)
# If all retries fail, instructor raises — you catch it and trigger a fallback.
```

**Python — the same loop by hand (so you see the mechanism), plus transport backoff:**
```python
from tenacity import retry, stop_after_attempt, wait_exponential_jitter
from pydantic import ValidationError

# Transport-level: retry the raw API call on network/5xx/rate-limit errors
@retry(stop=stop_after_attempt(4),
       wait=wait_exponential_jitter(initial=1, max=20))
def call_model(messages):
    return client_raw.chat.completions.create(model="gpt-5.5", messages=messages,
                                              temperature=0).choices[0].message.content

# Content-level: validate, and on failure re-prompt WITH the exact error
def extract(document, Model, max_tries=3):
    messages = [{"role": "user", "content": f"Extract fields as JSON.\n{document}"}]
    last_error = None
    for attempt in range(max_tries):
        raw = call_model(messages)
        try:
            return Model.model_validate_json(raw)          # success
        except ValidationError as e:
            last_error = str(e)
            messages.append({"role": "assistant", "content": raw})
            messages.append({"role": "user",
                             "content": f"That output was invalid:\n{last_error}\n"
                                        f"Return corrected JSON only."})
    # Fallback: escalate / default / human review
    return escalate_to_human_or_stronger_model(document, last_error)
```

**The subtle interaction with temperature.** On a content retry, it can help to keep temperature at 0 for the *first* attempt (you want the model's best single guess) but nudge it slightly higher on retries if the model keeps making the *same* deterministic mistake — a tiny bit of variation can shake it out of a bad local answer. This is a small, practical lever most tutorials skip.

**Finance use cases.**
1. **Resilient statement processing:** Transport backoff absorbs rate limits during a nightly batch of thousands of statements, while content self-correction fixes the occasional bad extraction — so the batch completes without manual babysitting, and genuinely unparseable documents are routed to a review queue instead of silently dropped.
2. **Escalation for hard cases:** A cheap model handles the 95% of standard filings; on repeated validation failure the pipeline escalates that specific document to a stronger model, balancing cost against reliability.

**E-commerce use cases.**
1. **Self-healing catalog enrichment:** When attribute extraction returns an out-of-range value, the error-feedback retry usually fixes it automatically, keeping a million-SKU pipeline flowing without human intervention.
2. **Graceful support degradation:** If parsing a customer's order request keeps failing, the assistant falls back to asking the customer a clarifying question rather than acting on garbage — a safe, user-friendly degradation.

## 2.2.3.c Guardrails — Making the Whole Exchange Safe and On-Policy

**Definition.** **Guardrails** are the checks and constraints placed *around* the LLM — on the way **in** (validating/sanitizing the user's input and context) and on the way **out** (checking the model's response before it reaches a user or triggers an action). Where Pydantic validation asks "is this data well-formed and rule-abiding?", guardrails ask the broader questions: "**Is this safe? Is it on-policy? Is it grounded in real sources? Does it leak anything it shouldn't?**" Guardrails are a superset concern; schema validation is one specific guardrail among several.

**Why guardrails are a distinct layer, explained.** Validation (2.2.3.a) is about the *data contract*. Guardrails are about *behavioral and safety policy*, which is broader and often not expressible as a schema. A response can be perfect JSON, pass every field validator, and still be a disaster: it might contain toxic language, reveal another customer's personal data, give regulated financial advice you're not licensed to give, follow an injected instruction hidden in a retrieved document, or confidently state a "fact" that appears nowhere in your sources. None of those are schema problems. They are policy and safety problems, and they need their own layer.

**Input guardrails (before the model runs).** These protect the model and your policies from bad or malicious input. The main ones, and *why* each exists:
- **Content moderation on input** — screen for disallowed or abusive content so you don't process (or appear to endorse) it. Providers expose moderation endpoints for this.
- **Prompt-injection screening** — detect attempts like "ignore your instructions" in user input or, more insidiously, in retrieved/tool content (indirect injection, recall 2.1.3/2.2.2). You can't catch everything, so this pairs with output guardrails and the "treat data as data" prompt discipline.
- **PII detection/redaction on input** — strip or mask personal data before it's sent to a third-party API, which is frequently a hard compliance requirement (especially in finance and healthcare).
- **Topic/scope enforcement** — reject requests outside the assistant's remit ("only answer questions about our products") before spending a model call.

**Output guardrails (after the model responds, before anyone sees or acts on it).** These are usually the more important set because they gate what actually reaches users and systems:
- **Structural/semantic validation** — the Pydantic layer from 2.2.3.a, now understood as *one* output guardrail.
- **Content moderation on output** — ensure the model didn't produce toxic, unsafe, or policy-violating text.
- **PII/secret leakage checks** — make sure the response doesn't expose personal data, credentials, or another user's information.
- **Grounding / hallucination checks** — for RAG and factual tasks (Phase 4), verify the answer is actually supported by the retrieved sources, e.g., by requiring citations and checking that claims trace back to context. This is how you catch the "well-formed but fabricated" failure.
- **Policy/compliance checks** — enforce domain rules: a financial assistant must include a disclaimer and must *not* give personalized advice; a medical assistant must not diagnose. These are often the difference between a legal product and a liability.
- **Action gating** — for tool-calling/agents (2.2.2, Phase 5), require confirmation and authorization before high-impact side effects. An output guardrail can block a `transfer` call that fails policy even if the model requested it.

**Intuition — guardrails as the airport, not just the metal detector.** Validation is one checkpoint. Guardrails are the whole security architecture: screening on the way in, screening on the way out, restricted zones for dangerous actions, and staff empowered to stop the process. The model is a powerful but fallible traveler; guardrails ensure that no matter what it tries to carry through, nothing unsafe reaches the other side. Critically, guardrails assume the model *will* sometimes be wrong or manipulated — they are defense-in-depth, not a single wall.

**How this is implemented in practice (current-market tooling).** You can hand-roll guardrails as ordinary functions (a moderation call here, a regex PII scan there, a Pydantic validation there), and for many apps that's the right, simple choice. When policies get numerous or need to be managed centrally, dedicated frameworks help: **Guardrails AI** (declarative validators/"guards" over LLM output, with automatic re-asking on failure), **NVIDIA NeMo Guardrails** (programmable rails for dialogue, topical boundaries, and safety, defined in a config language), and provider-native **moderation** and **safety-setting** APIs. `instructor` covers the validation-and-retry guardrail specifically. The important conceptual point is that all of these implement the same idea: **interpose checks between the model and the world, and re-ask or block when a check fails.**

**Python — a compact guardrail wrapper around a call:**
```python
def guarded_answer(user_input: str, retrieve_context) -> dict:
    # ---- INPUT GUARDRAILS ----
    if is_flagged(moderate(user_input)):                 # provider moderation
        return {"status": "refused", "reason": "input violated content policy"}
    user_input = redact_pii(user_input)                  # strip PII before external call
    if looks_like_injection(user_input):                 # heuristic/classifier screen
        return {"status": "refused", "reason": "possible prompt injection"}

    context = retrieve_context(user_input)               # RAG (Phase 4)

    # ---- MODEL CALL + VALIDATION/RETRY (2.2.3.a-b) ----
    try:
        answer = extract_answer(user_input, context)     # returns a validated Pydantic obj
    except Exception as e:
        return {"status": "error", "reason": str(e)}

    # ---- OUTPUT GUARDRAILS ----
    if is_flagged(moderate(answer.text)):                # moderate the OUTPUT
        return {"status": "blocked", "reason": "unsafe output"}
    if leaks_pii(answer.text):                            # leakage check
        return {"status": "blocked", "reason": "PII leakage"}
    if not is_grounded(answer.text, context):            # hallucination/grounding check
        return {"status": "needs_review", "reason": "answer not supported by sources"}
    if not includes_required_disclaimer(answer.text):    # policy check
        answer.text += "\n\nThis is not financial advice."

    return {"status": "ok", "answer": answer.text}
```

The structure of this function *is* the lesson: input checks, then the validated model call, then output checks, with a clear, safe outcome at every branch. Nothing reaches the user or an action until it has passed the relevant rails, and every failure path returns a defined, safe result rather than an exception bubbling up to the customer.

**Finance use cases.**
1. **Regulated advisory assistant:** Input PII redaction protects customer data before it leaves the perimeter; output policy guardrails guarantee the mandated disclaimer and block any personalized-advice phrasing; grounding checks ensure every stated figure traces to a retrieved document — together making the assistant defensible to auditors.
2. **Fraud-ops copilot with action gating:** Even if the model suggests an account action, an output guardrail requires analyst confirmation and verifies the action against policy before anything executes, containing "excessive agency" risk.

**E-commerce use cases.**
1. **Brand-safe public chatbot:** Output moderation and topic-scope rails keep a storefront assistant from producing off-brand, unsafe, or off-topic content, and injection screening stops a malicious review or product page from hijacking it.
2. **Grounded product answers:** A grounding guardrail blocks answers about specs or availability that aren't backed by the catalog, preventing hallucinated features that drive returns and complaints.

# Wrap-Up: 2.2.3 Validation, Retries & Guardrails

## The through-line (backward and forward)
Structured outputs (2.2.1) and tool calls (2.2.2) guarantee **shape**; this chapter adds the three layers that guarantee **trustworthiness**. **Pydantic validation** (2.2.3.a) checks structure *and* semantics, distinguishing "the JSON parsed" from "the data is usable," and — critically — emits a precise error. **Retries** (2.2.3.b) consume that error: transport failures get backoff, content failures get *re-prompted with the exact validation message* (self-correction), always under a cap with a fallback. **Guardrails** (2.2.3.c) wrap the whole exchange with input and output safety, moderation, PII, grounding, and policy/action checks — defense-in-depth that assumes the model will sometimes be wrong or manipulated. Understood together, they turn an unreliable, non-deterministic, occasionally-adversarial text service into a **dependable software component**. This completes Phase 2's reliability foundation and is precisely what makes the rest of the course safe to build: **RAG** (Phase 4) leans on grounding guardrails, **agents** (Phase 5) live or die by action gating and loop limits, and **deployment/LLMOps** (Phase 8) operationalizes all of it with monitoring.

## The production reliability pipeline (the mental model to keep)
`input guardrails (moderate, redact PII, screen injection, scope) → model call (structured output/tool call) → validate (Pydantic: structure + semantics) → on failure: retry with the error, capped, else fallback/human → output guardrails (moderate, PII leakage, grounding, policy, action gating) → safe result`

## Selected interview questions (short answers)
1. **Structure vs semantics — what's the difference and who enforces each?** Structure (fields/types) is guaranteed by schema-constrained decoding; semantics (business rules, cross-field invariants) must be enforced by validators (Pydantic).
2. **Why is the validation *error message* so important?** It's the specific feedback you feed back to the model on a content retry, enabling self-correction.
3. **Two kinds of retry and their remedies?** Transport failures → exponential backoff + jitter (don't change the prompt); content failures → re-prompt with the validation error.
4. **Why cap retries and add a fallback?** Retries cost tokens/latency/money and some outputs never validate; cap + fallback (stronger model, relaxed schema, default, human) prevents infinite, expensive failure (circuit-breaker idea).
5. **How do guardrails differ from validation?** Validation checks the data contract; guardrails enforce broader safety/policy (moderation, PII, grounding, injection, action gating) on both input and output.
6. **What's a grounding/hallucination guardrail?** An output check that the answer is actually supported by retrieved sources (e.g., citation/claim verification) — catches well-formed but fabricated content.
7. **Why validate a tool call's arguments before executing (tie to 2.2.2)?** The model's output is untrusted and possibly injected; validate/authorize before any side effect, and gate high-impact actions.
8. **Name current guardrail tooling.** Guardrails AI, NeMo Guardrails, provider moderation/safety APIs, `instructor` for validation+retries, plus hand-rolled function checks.

## Mini glossary
**Validation** — checking output structure + semantics.
**Field validator / model validator** — Pydantic checks on one field / across fields.
**Coercion** — Pydantic converting compatible types (e.g., "0.05" → 0.05).
**Transport failure** — network/rate-limit/5xx error (no real answer).
**Content failure** — a returned answer that fails validation.
**Self-correction** — re-prompting with the validation error to fix output.
**Backoff + jitter** — increasing, randomized retry delays.
**Fallback / circuit breaker** — safe degradation after repeated failure.
**Guardrail** — input/output safety & policy check around the model.
**Grounding check** — verifying answers are supported by sources.
**Action gating** — requiring authorization/confirmation before side effects.
