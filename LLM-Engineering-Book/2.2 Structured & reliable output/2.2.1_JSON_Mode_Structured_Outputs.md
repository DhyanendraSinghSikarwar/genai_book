## 2.2.0 Where this fits (the bridge from 2.1)

Everything in Phase 1 said the model emits **free-form text**, one token at a time (1.3.5), sampled from a distribution. Free text is great for humans and **useless for programs**: a downstream service needs `{"issuer": "...", "coupon": 0.05}`, not a friendly paragraph that *contains* those facts. In 2.1.3 we asked for JSON politely via the prompt — but a purely prompt-based request **can still drift**: the model adds prose ("Sure! Here's your JSON:"), forgets a field, emits trailing commas, or wraps it in Markdown fences. At scale, even a 1% malformed rate breaks an automated pipeline.

**Structured outputs** close this gap by moving from *asking* to *guaranteeing*. The spectrum runs from soft ("please return JSON") to hard ("the decoder is mathematically constrained to emit only tokens that keep the output valid against a schema"). This is the foundation for **function/tool calling (2.2.2)**, **agents (Phase 5)** — which are just tool calls with structured arguments — and any RAG/extraction pipeline.

> **One-line thesis:** *Prompting for JSON is a request; structured outputs are a contract. The strongest form constrains the decoder itself so invalid JSON is literally impossible — turning the LLM into a typed function you can trust in code.*

---

## 2.2.1.a Why Structured Output Is Hard (and the spectrum of solutions)

**Definition — the problem.** An LLM's native output is a **probability distribution over the whole vocabulary** at every step (1.3.5). Nothing in that process *inherently* respects JSON grammar or your schema. Reliability techniques add constraints on top.

**The spectrum (know all four levels — a favorite interview framing):**

1. **2.2.1.a.i Prompt-only ("please return JSON").** Just instruct the format. **Easiest, least reliable.** Fails via preamble text, Markdown fences, missing/extra fields, wrong types, invalid JSON. Needs parsing + retries.
2. **2.2.1.a.ii JSON mode.** The provider guarantees the output is **syntactically valid JSON** (parseable), but **not** that it matches *your* schema (fields/types can still be wrong). A partial guarantee.
3. **2.2.1.a.iii Schema-constrained / "Structured Outputs".** You supply a **JSON Schema**; the provider guarantees the output **conforms to it exactly** (right keys, types, enums, required fields). This is the modern gold standard (OpenAI *Structured Outputs* with `strict: true`; Google Gemini `responseSchema`; Anthropic via tool-use schemas).
4. **2.2.1.a.iv Constrained decoding / grammars (open models).** At the token level, mask out any token that would break the grammar/schema **before sampling** — so invalid output is *impossible by construction*. Implemented via **GBNF grammars** (llama.cpp), **Outlines**, **vLLM guided decoding**, JSON-Schema→FSM. This is how you get hard guarantees on **self-hosted** models (Phase 8).

**Intuition for level 4 (the deep one).** Because generation is autoregressive (1.3.5), at each step you can **zero out the probability** of any token that would make the partial string un-parseable against the schema's finite-state machine, then renormalize and sample only among *legal* tokens. The model still chooses *content*, but can only ever produce *valid structure*. Near-zero overhead, 100% structural validity.

**Interview angle.** "Why can't you just ask for JSON?" "JSON mode vs Structured Outputs — what's guaranteed by each?" "How does constrained/grammar decoding guarantee validity?" (token masking against a schema FSM). "How would you guarantee valid JSON from a self-hosted open model?" (Outlines / GBNF / vLLM guided decoding).

**Finance use cases.**
1. **Filing/invoice extraction into a database:** Downstream ETL needs guaranteed schema (`issuer`, `amount`, `currency`, `date`) — a malformed field breaks ingestion, so schema-constrained output is mandatory.
2. **Risk decisions as typed records:** A credit/KYC decision must be `{"decision": "approve|review|deny", "reasons": [...]}` with a strict enum — no free text.

**E-commerce use cases.**
1. **Product attribute extraction:** Populate the catalog with `{material, color, size, weight}` typed fields at scale — structural validity is non-negotiable.
2. **Order intent parsing:** Turn a chat message into `{"intent": "...", "order_id": "...", "qty": 2}` for the fulfillment API.

## 2.2.1.b JSON Mode & Schema-Constrained Structured Outputs (closed APIs)

**Definition.** Modern APIs expose a `response_format` (or equivalent) parameter:
- **JSON mode** (`{"type": "json_object"}`): output is **valid JSON** (syntactic guarantee only).
- **Structured Outputs** (`{"type": "json_schema", "json_schema": {...}, "strict": true}`): output is **valid JSON that conforms to your schema** (structural + type guarantee).

**Best practice: define the schema with Pydantic, don't hand-write JSON Schema.** Pydantic models give you a Python type, auto-generate the JSON Schema, and **validate** the response — one source of truth.

**Python — OpenAI Structured Outputs via a Pydantic model (the standard 2026 pattern):**
```python
from pydantic import BaseModel, Field
from typing import Literal
from openai import OpenAI
client = OpenAI()

class BondExtract(BaseModel):
    issuer: str
    coupon_rate: float = Field(description="annual coupon as a decimal, e.g. 0.05")
    maturity: str      # ISO date
    currency: Literal["USD", "EUR", "GBP", "INR"]
    callable: bool

doc = "Acme Corp 5% senior notes, maturity 2030-06-01, USD, callable."

# .parse() enforces the schema (strict) AND returns a typed object
resp = client.chat.completions.parse(
    model="gpt-5.5",
    messages=[
        {"role": "system", "content": "Extract bond fields from the document."},
        {"role": "user", "content": doc},
    ],
    response_format=BondExtract,   # SDK converts the Pydantic model to a strict JSON schema
)
bond = resp.choices[0].message.parsed   # -> BondExtract instance, already validated
print(bond.issuer, bond.coupon_rate, bond.currency)
```

**Python — Anthropic (structured JSON via a tool schema — a common idiom):**
```python
from anthropic import Anthropic
a = Anthropic()
schema = {
    "type": "object",
    "properties": {
        "issuer": {"type": "string"},
        "coupon_rate": {"type": "number"},
        "currency": {"type": "string", "enum": ["USD","EUR","GBP","INR"]},
    },
    "required": ["issuer", "coupon_rate", "currency"],
}
msg = a.messages.create(
    model="claude-opus-4-8", max_tokens=512,
    tools=[{"name": "emit_bond", "description": "Return bond fields",
            "input_schema": schema}],
    tool_choice={"type": "tool", "name": "emit_bond"},   # force the structured output
    messages=[{"role": "user", "content": doc}],
)
data = next(b.input for b in msg.content if b.type == "tool_use")  # dict matching schema
```

**Notes & gotchas (interview-relevant).**
- **Schema features supported vary by provider** (enums, nested objects, arrays, `required`); some require *all* fields be `required` and disallow open-ended `additionalProperties` in strict mode. Read the provider's constraints.
- **Descriptions matter:** field `description`s in the schema steer the model's *content* choices (e.g., "decimal, not percent").
- **Use low temperature** (1.3.5) for extraction; structure is guaranteed but *values* still benefit from determinism.
- **Structured output ≠ correct output:** the schema guarantees *shape*, not *truth*. A model can still hallucinate a plausible but wrong `coupon_rate`. Combine with RAG grounding (Phase 4) and validation.

**Interview angle.** "How do you get schema-conformant JSON from OpenAI/Anthropic/Gemini?" "Why define the schema in Pydantic?" "Does Structured Outputs guarantee the values are correct?" (no — only the shape). "What are strict-mode schema limitations?"

**Finance use cases.**
1. **Straight-through processing:** Guaranteed-schema extraction lets filings/invoices flow directly into accounting systems with no human JSON-fixing.
2. **Auditable decision objects:** Typed `{decision, confidence, reasons[]}` records are logged and reviewable for compliance.

**E-commerce use cases.**
1. **Catalog enrichment API:** Structured Outputs feed a typed attribute service that rejects malformed records automatically.
2. **Chat-to-cart:** Parse shopper messages into a strict order schema consumed by checkout.

## 2.2.1.c Constrained Decoding for Open Models + Validation & Retries

**Definition — constrained (guided) decoding.** For **self-hosted open-weight models** (Phase 8) you don't have a provider's `response_format`; instead you constrain the **decoder** yourself so it can only emit tokens consistent with a JSON Schema or grammar. Key tools:
- **Outlines** — regex/JSON-Schema/Pydantic-guided generation (compiles the schema to a finite-state machine that masks illegal tokens).
- **llama.cpp GBNF grammars** — a grammar file restricts output (great for local models).
- **vLLM guided decoding** (`guided_json`, `guided_regex`, `guided_grammar`) — schema-constrained serving at throughput.
- **`instructor`** — a popular wrapper that adds **Pydantic validation + automatic retries** on top of many providers/models (works with or without native structured outputs).

**Python — Outlines (guaranteed JSON from an open model):**
```python
import outlines
from pydantic import BaseModel
from typing import Literal

class Ticket(BaseModel):
    category: Literal["billing", "shipping", "technical", "other"]
    urgent: bool
    summary: str

model = outlines.models.transformers("Qwen/Qwen2.5-3B-Instruct")
generator = outlines.generate.json(model, Ticket)      # decoder now constrained to Ticket
ticket = generator("Customer: my package never arrived and I was double-charged!")
print(ticket.category, ticket.urgent)                  # always a valid Ticket
```

**Python — `instructor` (validation + retries; provider-agnostic):**
```python
import instructor
from openai import OpenAI
from pydantic import BaseModel, field_validator

class Invoice(BaseModel):
    total: float
    currency: str
    @field_validator("total")
    @classmethod
    def positive(cls, v):
        if v <= 0: raise ValueError("total must be positive")   # triggers a retry
        return v

client = instructor.from_openai(OpenAI())
inv = client.chat.completions.create(
    model="gpt-5.5",
    response_model=Invoice,
    max_retries=3,                       # re-prompts with the validation error until valid
    messages=[{"role": "user", "content": "Invoice total: $250.00 USD"}],
)
print(inv.total, inv.currency)
```

**The validation-and-retry pattern (essential even with guarantees).** Structure can be guaranteed but **semantics** can't. A robust pipeline: **generate → parse (Pydantic) → validate business rules → on failure, re-prompt with the specific error → cap retries → fall back/human review.** This closed loop is what production reliability actually looks like.

```python
# Generic manual pattern (works with any model)
def extract_with_retry(call_fn, Model, max_tries=3):
    last_err = None
    for _ in range(max_tries):
        raw = call_fn(hint=last_err)          # pass previous error back into the prompt
        try:
            return Model.model_validate_json(raw)   # Pydantic parse + validate
        except Exception as e:
            last_err = str(e)                 # feed the exact error to the next attempt
    raise ValueError(f"failed after retries: {last_err}")
```

**Interview angle.** "How do you guarantee valid JSON from a local/open model?" (Outlines / GBNF / vLLM guided decoding). "What does `instructor` add?" (Pydantic validation + auto-retries). "Structure is guaranteed — why still validate?" (semantics/business rules, hallucinated values). "Design a reliable extraction pipeline." (generate→parse→validate→retry-with-error→fallback).

**Finance use cases.**
1. **On-prem extraction under compliance:** A self-hosted model + Outlines/GBNF gives guaranteed-schema extraction without sending data to an API.
2. **Business-rule validation:** `field_validator`s enforce invariants (amounts positive, currency in allow-list, dates plausible) before a record is booked.

**E-commerce use cases.**
1. **Guaranteed catalog records at scale:** vLLM guided decoding serves millions of attribute-extraction calls with 100% structural validity.
2. **Self-healing parsing:** Retry-with-error recovers transient bad outputs automatically, keeping the pipeline flowing.

# Wrap-Up: 2.2.1 JSON Mode / Structured Outputs

## The through-line (backward and forward)
LLMs emit free text (Phase 1), but software needs typed data — so we escalate from **prompt-only JSON** (unreliable) → **JSON mode** (syntactic guarantee) → **Structured Outputs / schema-constrained** (structural + type guarantee, via Pydantic) → **constrained decoding/grammars** for open models (validity by construction), always wrapped in **validate-and-retry** for semantics (2.2.1.a–c). This directly enables **function/tool calling (2.2.2)** — a tool call *is* a structured output — and therefore **agents (Phase 5)**, plus every **RAG/extraction** pipeline. Remember the boundary: *structure ≠ truth* — pair guaranteed schemas with grounding (RAG, Phase 4) and evals.

## Decision guide
| Situation | Use |
|---|---|
| Closed API, need typed data | **Structured Outputs** (`json_schema`, `strict`) via Pydantic |
| Closed API, just need parseable JSON | **JSON mode** |
| Anthropic structured data | **Tool schema** + forced `tool_choice` |
| Self-hosted open model | **Outlines / vLLM guided / GBNF** constrained decoding |
| Any model + validation + retries | **`instructor`** or manual generate→validate→retry |
| Values must be correct, not just shaped | Add **RAG grounding + business-rule validators + evals** |

## Expanded interview Q&A bank (short answers)
1. **Why can't you just ask an LLM for JSON?** It may add prose/fences, miss fields, use wrong types, or emit invalid JSON; native output is unconstrained text.
2. **What are the four reliability levels?** Prompt-only → JSON mode → schema-constrained Structured Outputs → grammar/constrained decoding.
3. **JSON mode vs Structured Outputs?** JSON mode guarantees *valid JSON*; Structured Outputs guarantees *valid JSON matching your schema*.
4. **How does constrained decoding guarantee validity?** At each step it masks tokens that would violate the schema/grammar FSM, then samples only legal tokens.
5. **Why define schemas in Pydantic?** Single source of truth: Python type + auto JSON Schema + response validation.
6. **Does Structured Outputs guarantee correct values?** No — only shape/types; values can still be hallucinated.
7. **How do you get structured output from Anthropic?** Define a tool `input_schema` and force `tool_choice` to that tool.
8. **How with Gemini?** Set `responseSchema` (+ `responseMimeType: application/json`).
9. **How from a self-hosted open model?** Constrained decoding: Outlines, llama.cpp GBNF grammars, or vLLM guided decoding.
10. **What does the `instructor` library add?** Pydantic `response_model` validation plus automatic retries with the error fed back.
11. **What is the generate→validate→retry pattern?** Generate, parse/validate with Pydantic, re-prompt with the exact error on failure, cap retries, then fall back.
12. **Why still validate if structure is guaranteed?** To enforce business rules/semantics and catch hallucinated values.
13. **What temperature for extraction?** Low (often 0) for determinism, even though structure is enforced separately.
14. **Common strict-mode schema limitations?** May require all fields `required`, disallow `additionalProperties`, and support only a subset of JSON-Schema features.
15. **How do field descriptions help?** They steer the model's *content* (units, formats), improving value quality.
16. **What's the overhead of grammar-constrained decoding?** Typically small; the FSM/token-mask is cheap and can even reduce wasted tokens.
17. **How does structured output relate to function calling?** A function/tool call is itself a schema-constrained structured output (arguments).
18. **How does it relate to agents?** Agents rely on structured tool arguments; guaranteed schemas make loops reliable.
19. **What breaks pipelines without structured output?** Even a low malformed-rate causes parse failures downstream at scale.
20. **How do you handle nested/complex objects?** Nested Pydantic models / nested JSON Schema; verify provider support for nesting/arrays.
21. **Streaming + structured output?** Possible but partial JSON is invalid until complete; use incremental parsers or wait for completion.
22. **How do you test structured extraction?** Labeled eval set measuring both schema validity (100% target) and field-level accuracy.
23. **What about enums?** Use `Literal`/`enum` to constrain categorical fields (decisions, categories, currencies).
24. **Cost implication of structured output?** Minimal for schema constraints; retries add cost — cap them.
25. **When prefer tool-calling over raw JSON schema?** When the model must *choose* actions or you want native argument extraction (2.2.2).
26. **How to defend structured extraction from prompt injection?** Delimit/label untrusted source text as data (2.1.3) and validate outputs before use.
27. **What's a good fallback when validation keeps failing?** Route to a stronger model, simplify the schema, or send to human review.
28. **Structure vs truth — one-liner?** Structured outputs guarantee the *shape*, not the *facts*; ground with RAG and validate.

## Mini glossary
**Structured output** — model output constrained to a defined schema.
**JSON mode** — guarantees syntactically valid JSON only.
**JSON Schema** — spec describing allowed structure/types.
**Structured Outputs (strict)** — schema-conformant output guarantee.
**Constrained/guided decoding** — token masking to enforce a grammar/schema.
**GBNF** — llama.cpp grammar format.
**Outlines / vLLM guided** — open-model constrained-generation tools.
**Pydantic** — Python data-validation models (schema + validation).
**instructor** — library adding Pydantic validation + retries.
**Validator** — code enforcing business rules on parsed output.
