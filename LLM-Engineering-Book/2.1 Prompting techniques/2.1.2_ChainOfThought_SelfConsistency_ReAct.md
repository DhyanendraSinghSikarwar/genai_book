## 2.1.2.0 Where this fits (the bridge from 2.1.1)

Recall the core fact from Phase 1: an LLM is a **next-token predictor** (1.3.1) that produces an answer in a single left-to-right pass. For a hard, multi-step problem ("compute this, then compare, then decide"), forcing the answer into the *very next token* gives the model no room to work — it must "blurt." The insight behind this topic: **if we let the model generate intermediate reasoning tokens first, those tokens become a scratchpad that conditions a better final answer.** More generated reasoning = more **test-time compute** spent on the problem before committing.

This idea scales into three layers you must know in 2026:
1. **Chain-of-Thought (CoT)** — make the *single* pass show its steps.
2. **Self-Consistency & Tree-of-Thoughts** — run/branch *multiple* reasoning paths and aggregate.
3. **ReAct** — interleave reasoning with **actions** (tool calls), which is the doorway to agents (Phase 5).

> **One-line thesis (2026 update):** *Reasoning is "thinking out loud in tokens." You can elicit it with prompts (CoT), make it more robust by sampling many paths (self-consistency), let it act on the world (ReAct) — or buy it built-in from a dedicated reasoning model. Knowing when to prompt vs when to pick a reasoning model is the modern skill.*

## 2.1.2.a Chain-of-Thought (CoT) Prompting

**Definition.** **Chain-of-Thought prompting** instructs or demonstrates the model to produce **explicit intermediate reasoning steps** before the final answer, rather than jumping straight to it. Introduced by Wei et al. (2022) as a **few-shot** technique (show worked, step-by-step examples); Kojima et al. (2022) then showed a **zero-shot** trigger — appending *"Let's think step by step"* — elicits reasoning with no examples.

**Intuition (why it works).** Each generated reasoning token is fed back in as context (autoregression, 1.3.5), so the model literally *builds up* a solution and conditions the final token on its own worked steps. It also decomposes one hard prediction into several easy ones. Empirically, CoT's biggest gains are on **arithmetic, logic, and multi-step** tasks; it does little for simple lookups.

**Two flavors:**
- **Zero-shot CoT:** add a trigger phrase ("Let's think step by step", "Work through this carefully before answering").
- **Few-shot CoT:** provide examples whose answers *include the reasoning* (builds on 2.1.1.b).

**Example (zero-shot CoT):**
```
Q: A portfolio holds $10,000 at 6% annual simple interest.
   After 3 years, what is the total value?
A: Let's think step by step.
   Interest per year = 10,000 × 0.06 = 600.
   Over 3 years = 600 × 3 = 1,800.
   Total = 10,000 + 1,800 = 11,800.
   Final answer: $11,800.
```

**Critical caveats (2026, interview-relevant):**
- **Faithfulness problem:** the printed chain is **not guaranteed** to be the model's true computation — it can reach the right answer for reasons other than the steps shown, or rationalize a wrong one. *Don't treat CoT as a verified audit trail.* For real correctness on math/logic, offload to **tools/code** (Phase 5) — tokenized arithmetic is unreliable (recall 1.2.7).
- **CoT vs reasoning models (the big 2026 shift):** dedicated **reasoning models** (o-series, Claude *extended thinking*, Gemini *Deep Think*, DeepSeek **R1**, Qwen **QwQ**) already do a **built-in deliberation pass** (often on hidden or separately-budgeted "thinking" tokens) before answering. With these, hand-holding "think step by step" is **usually unnecessary and can even hurt** — the right move is a **clear brief + a difficulty/effort setting**, not manual CoT. On classic (non-reasoning) chat models, explicit CoT still helps a lot.
- **Cost/latency:** reasoning tokens are real output tokens → **more cost and latency** (1.2.6). Use reasoning depth proportional to task difficulty.
- **Hide the chain in production:** you typically want the *answer* shown to users, not the raw reasoning (both for UX and to avoid exposing shaky intermediate steps). Ask for a final answer in a delimited field, or use a model whose thinking is separate.

**Python — zero-shot CoT, then parse only the final answer:**
```python
from openai import OpenAI
client = OpenAI()

def cot_answer(question: str) -> str:
    prompt = (
        f"{question}\n\n"
        "Think step by step inside <reasoning>...</reasoning>, "
        "then give ONLY the final result inside <answer>...</answer>."
    )
    r = client.chat.completions.create(
        model="gpt-5.5",
        messages=[{"role": "user", "content": prompt}],
        temperature=0,   # deterministic for a single reasoning pass
    )
    text = r.choices[0].message.content
    # extract just the answer for downstream use / display
    import re
    m = re.search(r"<answer>(.*?)</answer>", text, re.S)
    return m.group(1).strip() if m else text

print(cot_answer("If revenue grows 8% on a base of $250,000, what is next-year revenue?"))
```

**Python — using a reasoning model instead (2026 pattern; no manual CoT):**
```python
# Reasoning models expose an effort/thinking budget instead of CoT prompting.
r = client.chat.completions.create(
    model="o4-mini",                     # a reasoning model
    reasoning_effort="high",             # let it think harder (provider-specific)
    messages=[{"role": "user",
               "content": "Value a 3-year bond, 5% annual coupon, face $1,000, "
                          "yield 6%. Show the final price only."}],
)
print(r.choices[0].message.content)
# For heavy numeric accuracy, still prefer a tool/calculator (Phase 5).
```

**Interview angle (see the big Q&A bank at the end too).** "Why does CoT improve reasoning?" "Zero-shot vs few-shot CoT?" "Is the printed chain faithful?" "When is manual CoT unnecessary in 2026?" (with reasoning models).

**Finance use cases.**
1. **Multi-step calculations with an audit narrative:** Loan amortization or bond pricing where an explainable step-by-step rationale is required for review (with the *actual* math done by a tool).
2. **Risk decision reasoning:** Walk through eligibility rules ("income ≥ X and DTI ≤ Y and …") step by step to justify a credit or KYC decision.

**E-commerce use cases.**
1. **Complex eligibility/promo logic:** Reason through stacked discount rules ("member + first order + coupon, but not on sale items") to compute the correct price path.
2. **Diagnostic support:** Step-by-step troubleshooting ("if device won't charge → check cable → check port → …") for guided customer help.

## 2.1.2.b Self-Consistency (and Tree-of-Thoughts / Least-to-Most)

**Definition.** **Self-consistency** (Wang et al., 2022) improves CoT by **sampling multiple independent reasoning paths** (temperature > 0) for the *same* question and then **taking the majority-vote answer** across them, instead of trusting a single greedy chain. The reasoning may differ, but correct answers tend to *converge*.

**Intuition.** A single chain can go wrong via one bad step. If you generate, say, 5–20 diverse chains and most land on the same final answer, that consensus is far more reliable than any one path. It trades **more compute for more accuracy** — a form of test-time scaling.

**Example (concept).** Ask the same math/logic question 7 times at `temperature≈0.7`; collect final answers `[$11,800, $11,800, $12,000, $11,800, $11,800, $11,800, $12,000]` → majority = **$11,800**.

**Python — self-consistency by majority vote:**
```python
from collections import Counter
import re

def self_consistent(question: str, n: int = 7) -> str:
    finals = []
    for _ in range(n):
        r = client.chat.completions.create(
            model="gpt-5.5",
            messages=[{"role": "user",
                       "content": question + "\nReason step by step, then end with 'Final: <value>'."}],
            temperature=0.7,           # diversity across paths
        )
        m = re.search(r"Final:\s*(.+)", r.choices[0].message.content)
        if m: finals.append(m.group(1).strip())
    return Counter(finals).most_common(1)[0][0] if finals else "no answer"

print(self_consistent("A cart has 3 items at $19.99 and a 10% coupon. Total?"))
```

**Related methods to name-drop (breadth interviewers like):**
- **Tree-of-Thoughts (ToT):** instead of linear chains, **branch and search** over intermediate "thoughts" (BFS/DFS with evaluation), backtracking from dead ends. Best for puzzles/planning; expensive.
- **Least-to-Most prompting:** first **decompose** a hard problem into ordered sub-problems, then solve them in sequence, feeding earlier answers forward. Great for compositional tasks.
- **Reflection / Self-Critique (Reflexion):** the model **critiques and revises** its own answer ("find errors, then fix them"), often in a second pass — a cheap accuracy boost and a building block of agents (Phase 5).

**Cost note.** All of these multiply token usage (N× for self-consistency, more for ToT). In 2026, **weigh them against simply using a reasoning model**, which internalizes much of this deliberation — sometimes cheaper/simpler than orchestrating many chat-model samples.

**Interview angle.** "How does self-consistency work and why does it beat a single chain?" "Self-consistency vs Tree-of-Thoughts?" "When is least-to-most better than plain CoT?" "What's the cost trade-off, and how does it compare to a reasoning model?"

**Finance use cases.**
1. **Higher-stakes numeric reasoning:** Majority-vote across several reasoning paths for a valuation/eligibility figure where a single wrong chain is costly (still validate with tools).
2. **Robust classification of ambiguous filings:** Vote across paths to stabilize a risk-category label on borderline text.

**E-commerce use cases.**
1. **Reliable complex pricing:** Self-consistency on stacked-discount computations reduces one-off arithmetic slips before showing a price.
2. **Consistent policy decisions:** Majority-vote on "is this return eligible?" across reasoning paths for fairness and stability.

## 2.1.2.c ReAct-Style Reasoning (Reason + Act)

**Definition.** **ReAct** (Yao et al., 2022) interleaves **Reasoning** ("Thought") with **Acting** ("Action" = calling an external tool) and **Observation** (the tool's result), in a loop:
```
Thought → Action → Observation → Thought → … → Answer
```
The model reasons about what it needs, **calls a tool** (search, calculator, database, API), reads the result, and continues reasoning with that new grounding. This is the conceptual foundation of **tool-using agents** (Phase 5) and directly fixes two Phase-1 weaknesses: **stale knowledge** (call live data) and **bad arithmetic** (call a calculator/code).

**Intuition.** Pure CoT reasons in a closed world — it can only use what's in its weights + prompt, and it can hallucinate. ReAct opens the loop: **reasoning decides *what to look up or compute*, actions bring in ground truth, observations correct the course.** It's the difference between "think hard from memory" and "think, then check."

**Example (ReAct trace):**
```
Question: What is the current 10-year Treasury yield times our $2M position's duration of 7?

Thought: I need the current 10-year yield; I don't know it reliably. I'll look it up.
Action: web_search["current US 10-year Treasury yield"]
Observation: 4.2%
Thought: Now compute price sensitivity ≈ duration × yield change... but I need exact math.
Action: calculator["0.042 * 7"]     # or a proper DV01 formula
Observation: 0.294
Thought: I now have grounded inputs to answer.
Answer: ~0.294 (29.4% notional sensitivity per 100% yield move; ~$5,880 per 1bp — verify with DV01).
```

**Python — a minimal ReAct loop with real tools (this is "an agent in 20 lines"):**
```python
import json
from openai import OpenAI
client = OpenAI()

def calculator(expr): return eval(expr, {"__builtins__": {}})     # demo only; sandbox in prod
def web_search(q):     return "4.2%"                              # stub; real API in prod
TOOLS = {"calculator": calculator, "web_search": web_search}

SYSTEM = (
  "You solve tasks by reasoning and using tools. Respond ONLY as JSON:\n"
  '  {"thought": "...", "action": "calculator|web_search|final", "input": "..."}\n'
  "Use action 'final' when you can answer; 'input' is then the final answer."
)

def react(question, max_steps=6):
    msgs = [{"role":"system","content":SYSTEM},{"role":"user","content":question}]
    for _ in range(max_steps):
        r = client.chat.completions.create(model="gpt-5.5", messages=msgs, temperature=0)
        step = json.loads(r.choices[0].message.content)
        if step["action"] == "final":
            return step["input"]
        obs = TOOLS[step["action"]](step["input"])               # ACT
        msgs.append({"role":"assistant","content":json.dumps(step)})
        msgs.append({"role":"user","content":f"Observation: {obs}"})  # OBSERVE
    return "stopped: step limit"

# print(react("What is 4.2% of a $2,000,000 position, times a duration of 7?"))
```
*(In production you'd use native **function/tool calling** — the topic of 2.2 — plus loop limits, timeouts, and error handling. Frameworks like LangGraph, CrewAI, and the OpenAI Agents SDK implement robust ReAct loops for you — Phase 5.)*

**2026 note.** Reasoning models + native tool-calling have largely **absorbed ReAct into the platform**: modern models can decide to call tools mid-reasoning without you hand-crafting the Thought/Action/Observation text. Understand the pattern (interviews still ask), but in practice you'll lean on structured tool-calling and agent frameworks.

**Interview angle.** "What is ReAct and what problem does it solve over CoT?" "Draw the loop." "How does ReAct relate to agents and tool-calling?" "What guardrails does a ReAct loop need?" (step limits, timeouts, tool-error handling, output validation).

**Finance use cases.**
1. **Grounded market Q&A:** Reason about a query, **call a market-data API** for live prices/yields, then compute — no reliance on stale weights.
2. **Compliance lookups + action:** Reason, **query a policy database**, and draft a compliant response citing the retrieved rule.

**E-commerce use cases.**
1. **Order-status assistant:** Reason, **call the orders API**, read tracking, and reply — a real agentic support flow.
2. **Live inventory & pricing:** Reason, **query inventory/pricing services**, compute the correct total with a tool, then answer.

# Wrap-Up: 2.1.2 CoT, Self-Consistency & ReAct

## The through-line (backward and forward)
Because a model answers in one forward pass (1.3), hard problems benefit from **generating reasoning tokens first**. **CoT** (2.1.2.a) elicits that reasoning in a single pass; **self-consistency / ToT / least-to-most** (2.1.2.b) spend more test-time compute across multiple paths for robustness; **ReAct** (2.1.2.c) interleaves reasoning with **tool actions**, grounding the model and fixing stale knowledge + bad arithmetic — the direct precursor to **function calling (2.2)** and **agents (Phase 5)**. The 2026 twist: dedicated **reasoning models** internalize much of this, so the modern skill is *choosing* between "prompt the reasoning" and "pick a model that reasons," and always **offloading exact math/facts to tools**.

## The reasoning toolkit — quick decision guide
| Technique | Use when | Cost | 2026 status |
|---|---|---|---|
| Zero/few-shot CoT | Multi-step task on a *classic* chat model | Low–med | Often replaced by reasoning models |
| Self-consistency | Need robustness on numeric/logic answers | High (N×) | Useful; compare vs reasoning model |
| Tree-of-Thoughts | Search/planning/puzzles | Very high | Niche |
| Least-to-most | Compositional decomposition | Med | Useful |
| Reflection/self-critique | Cheap accuracy boost; agent step | Med | Common in agents |
| ReAct + tools | Need live data / real actions / exact math | Med | Absorbed into tool-calling & agents |
| **Reasoning model** (o-series, Claude thinking, R1, QwQ) | Hard reasoning with minimal prompt effort | Pay per thinking tokens | **Default for hard reasoning** |

## Expanded interview Q&A bank (short answers)
1. **What is chain-of-thought prompting?** Prompting the model to produce explicit intermediate steps before the final answer, improving multi-step accuracy.
2. **Why does CoT work mechanically?** Generated reasoning tokens feed back as context, so the final answer is conditioned on worked steps; it also splits one hard prediction into several easy ones.
3. **Zero-shot vs few-shot CoT?** Zero-shot uses a trigger like "Let's think step by step"; few-shot shows examples whose answers include reasoning.
4. **On which tasks does CoT help most / least?** Most on arithmetic, logic, multi-hop; little on simple lookups/classification.
5. **Is the printed chain faithful to the real computation?** Not guaranteed — models can rationalize; don't treat CoT as a verified audit trail.
6. **How do you get correct math despite CoT's limits?** Offload to tools/code (calculator, Python) — tokenized arithmetic is unreliable (1.2.7).
7. **What changed with 2026 reasoning models?** They deliberate on hidden/separately-budgeted "thinking" tokens by default; manual CoT is usually unnecessary and can hurt.
8. **How do you prompt a reasoning model well?** Give a clear brief and set an effort/thinking budget; avoid step-by-step hand-holding.
9. **What is test-time compute?** Compute spent reasoning at inference before answering; more of it can raise accuracy on hard tasks.
10. **What is self-consistency?** Sample multiple CoT paths (temp > 0) and majority-vote the final answer.
11. **Why does self-consistency beat a single greedy chain?** Correct answers converge across diverse paths; it averages out single-step errors.
12. **Self-consistency vs Tree-of-Thoughts?** Self-consistency samples independent linear chains and votes; ToT branches/searches/backtracks over a tree of thoughts.
13. **What is least-to-most prompting?** Decompose into ordered sub-problems, solve sequentially, feeding earlier results forward.
14. **What is Reflexion / self-critique?** The model reviews and revises its own output, often in a second pass, to catch errors.
15. **What is ReAct?** Interleaving Thought → Action (tool call) → Observation in a loop until an answer.
16. **What two Phase-1 weaknesses does ReAct fix?** Stale/knowledge-cutoff facts (via live tools) and unreliable arithmetic (via calculator/code).
17. **How does ReAct relate to agents?** It's the canonical agent loop; frameworks (LangGraph, CrewAI, OpenAI Agents SDK) implement it robustly.
18. **What guardrails does a ReAct/agent loop need?** Max-step/loop limits, timeouts, tool-error handling, output validation, cost/latency caps.
19. **CoT vs ReAct — key difference?** CoT reasons in a closed world; ReAct can act on the world (tools) and incorporate observations.
20. **When would you NOT use CoT/reasoning?** Simple, latency-sensitive, or cost-sensitive tasks where a direct zero-shot answer suffices.
21. **How does reasoning affect cost and latency?** Reasoning/thinking tokens are billed output tokens → more cost and slower responses (1.2.6).
22. **How do you keep the chain hidden from users?** Ask for a delimited final answer, parse it out, or use a model with separate thinking tokens.
23. **Does higher temperature help reasoning?** It powers self-consistency diversity, but for a single deterministic chain use temperature 0; reasoning models often ignore/limit temperature.
24. **What is "underthinking/overthinking" in reasoning models?** Spending too little (early wrong commit) or too much (wandering) test-time compute; tune the effort budget.
25. **How do you evaluate reasoning quality?** Task accuracy on a labeled eval set, plus faithfulness/consistency checks — not the eloquence of the chain (ties to Phase 4 evals).
26. **Self-consistency vs just using a reasoning model in 2026?** Reasoning models often match/beat orchestrated sampling with less engineering; benchmark both on cost + accuracy.
27. **Can CoT leak sensitive intermediate content?** Yes — don't expose raw chains in regulated/consumer UIs; show only vetted final answers.
28. **What's a cheap first step before self-consistency/ToT?** A single self-critique/reflection pass often boosts accuracy at ~2× cost, far less than N× voting.

## Mini glossary
**CoT** — chain-of-thought; explicit intermediate reasoning.
**Zero-shot CoT** — trigger phrase ("think step by step").
**Self-consistency** — sample many chains, majority-vote.
**Tree-of-Thoughts** — branching search over intermediate thoughts.
**Least-to-most** — decompose then solve sub-problems in order.
**Reflection/Reflexion** — self-critique and revise.
**ReAct** — reason + act (tool) + observe loop.
**Test-time compute** — inference-time reasoning effort.
**Reasoning model** — model with a built-in deliberation pass (o-series, Claude thinking, R1, QwQ).
**Faithfulness** — whether the shown reasoning reflects the true computation.
