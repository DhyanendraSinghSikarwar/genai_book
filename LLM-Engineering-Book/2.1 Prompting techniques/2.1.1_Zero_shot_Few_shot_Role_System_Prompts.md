## 2.1.0 Where this fits (the bridge from Phase 1)

A deployed LLM's weights are **frozen**. You cannot retrain it per request; the *only* thing you control at call time is the **sequence of tokens you send** — the prompt. Recall from 1.3.5 that the model turns your prompt into a probability distribution over the next token and samples from it. **Prompting is the art and engineering of shaping that distribution toward the output you want**, without touching the weights.

Why this is the highest-leverage skill in the whole course:
- It is **free** (no training) and **instant** (no infra).
- It is the **first thing to try** before RAG (Phase 4) or fine-tuning (Phase 6) — the "prompt vs RAG vs fine-tune" decision (1.3.1) almost always starts here.
- Everything later — RAG, agents, tool-calling — is *prompting with extra context and structure*.

> **One-line thesis:** *Zero-shot tells the model what to do; few-shot shows it; role/system prompts tell it who to be. Master these three and you control most of an LLM's behavior before writing a single line of retrieval or fine-tuning code.*

## 2.1.1.a Zero-Shot Prompting

**Definition.** **Zero-shot prompting** gives the model an **instruction with no worked examples** and relies on the knowledge and instruction-following ability it acquired during pretraining + instruction tuning (1.3.2–1.3.3). "Zero-shot" = zero demonstrations.

**Intuition.** Because the model was **instruction-tuned** (1.3.3) on millions of `(instruction → response)` pairs, it already "knows how to be told what to do." For common, well-specified tasks (summarize, classify, translate, extract), a clear instruction alone often suffices. This is why modern instruct models feel so capable out of the box — you're cashing in on their SFT stage.

**Example.**
```
Classify the sentiment of this review as Positive, Negative, or Neutral.
Review: "The delivery was late but the product quality is excellent."
Sentiment:
```
No examples given — the model infers the task from the instruction.

**Best practices (interview-worthy).**
- Be **explicit and specific**: state the task, the allowed outputs, and the format.
- **Constrain the output** ("Answer with one word", "Return JSON with keys x, y").
- Put the **instruction first**, data second, and use clear **delimiters** (```` ``` ````, XML tags) to separate them.
- Zero-shot pairs naturally with **low temperature** (1.3.5) for deterministic tasks.

**When it fails → escalate.** If the task is niche, the output format is subtle, or the model is inconsistent, add **examples (few-shot)** or richer context (RAG). This escalation ladder is itself an interview answer.

**Python — zero-shot with structured, deterministic output:**
```python
from openai import OpenAI
client = OpenAI()

def classify_sentiment(review: str) -> str:
    prompt = (
        "Classify the sentiment as exactly one of: Positive, Negative, Neutral.\n"
        "Return only the label.\n\n"
        f"Review: \"\"\"{review}\"\"\"\n"
        "Sentiment:"
    )
    r = client.chat.completions.create(
        model="gpt-5.5",
        messages=[{"role": "user", "content": prompt}],
        temperature=0,          # deterministic — see 1.3.5
        max_tokens=3,
    )
    return r.choices[0].message.content.strip()

print(classify_sentiment("Late delivery, but the quality is excellent."))
```

**Interview angle.** "What is zero-shot prompting and why does it work at all?" (instruction tuning). "How do you make a zero-shot classifier reliable?" (constrain outputs, delimiters, temperature 0). "When would you *not* use zero-shot?"

**Finance use cases.**
1. **Transaction categorization:** Zero-shot label a transaction memo into {Salary, Transfer, Bill, Investment, Suspicious} with a fixed label set.
2. **Filing summary on demand:** "Summarize this risk-factors paragraph in 2 sentences" — no examples needed for a well-understood task.

**E-commerce use cases.**
1. **Review sentiment/aspect tagging:** Zero-shot classify reviews as Positive/Negative/Neutral at scale.
2. **Support ticket routing:** Zero-shot assign an incoming message to a department from a fixed list.

## 2.1.1.b Few-Shot Prompting (In-Context Learning)

**Definition.** **Few-shot prompting** includes a **small number of worked examples** (input → desired output "demonstrations") *inside the prompt*, before the actual query. The model infers the pattern and applies it — a behavior called **in-context learning (ICL)**, famously highlighted in the GPT-3 paper (Brown et al., 2020). No weights change; the "learning" happens purely within the forward pass.

**Intuition.** You're not teaching the model new facts — you're **disambiguating the task and pinning the exact output format** by demonstration. Humans do the same: "here are three solved examples, now do the fourth." (Terminology: **one-shot** = 1 example, **few-shot** = a handful.)

**Example (few-shot classification with a custom scheme):**
```
Label each message's intent.

Message: "Where is my order?"           Intent: OrderStatus
Message: "I want a refund."             Intent: Refund
Message: "Do you ship to Canada?"       Intent: ShippingInfo
Message: "This item arrived broken."    Intent:
```
The model outputs `Damage` / `Refund`-style label following the demonstrated format.

**Best practices (very interview-relevant).**
- **Format consistency is everything:** every example must use the *identical* structure, delimiters, and label style — the model imitates format aggressively.
- **Coverage & diversity:** pick examples spanning the classes/edge cases you care about; include a hard/ambiguous one if that's where it fails.
- **Order matters:** models can be sensitive to example order (and exhibit recency bias — the last example carries weight). If needed, test a couple of orderings.
- **Label balance:** don't over-represent one class, or the model skews toward it.
- **Cost trade-off (connect to 1.2.6):** every example is **input tokens on every call** — few-shot raises cost and latency. If you need *many* examples permanently, that's a signal to consider **fine-tuning** (Phase 6) instead.

**Python — few-shot with a reusable message list:**
```python
few_shot = [
    {"role": "system", "content": "You label customer messages with a single intent tag."},
    {"role": "user", "content": "Where is my order?"},
    {"role": "assistant", "content": "OrderStatus"},
    {"role": "user", "content": "I want a refund."},
    {"role": "assistant", "content": "Refund"},
    {"role": "user", "content": "Do you ship to Canada?"},
    {"role": "assistant", "content": "ShippingInfo"},
]

def classify_intent(message: str) -> str:
    msgs = few_shot + [{"role": "user", "content": message}]
    r = client.chat.completions.create(
        model="gpt-5.5", messages=msgs, temperature=0, max_tokens=4
    )
    return r.choices[0].message.content.strip()

print(classify_intent("This item arrived broken."))
```
*(Note we encode the demonstrations as alternating user/assistant turns — the chat-template-native way to do few-shot, tying back to 1.2.4.)*

**Interview angle.** "Zero-shot vs few-shot — when do you add examples?" "What is in-context learning and does it change the weights?" (no). "Why does example order/format matter?" "When do many-shot needs indicate fine-tuning instead?" "How does few-shot affect cost?" (more input tokens every call).

**Finance use cases.**
1. **Custom risk taxonomy extraction:** A few labeled examples teach the model *your firm's* categories ("Credit / Market / Operational / Liquidity") more precisely than a bare instruction.
2. **Consistent statement parsing:** Two or three `(bank statement line → structured JSON)` demonstrations lock the exact output schema for downstream systems.

**E-commerce use cases.**
1. **Brand-specific attribute extraction:** Show 3 examples of `(product title → {material, color, size})` to match your catalog's exact field conventions.
2. **Tone-matched reply drafts:** A few example `(customer message → on-brand reply)` pairs make support drafts sound like your team.

## 2.1.1.c Role & System Prompts

**Definition.** A **system prompt** is a special, high-priority instruction (delivered in the `system` role of the chat format — recall the role tokens from 1.2.4) that sets the model's **persona, rules, scope, tone, and output conventions** for the *entire* conversation, separate from the user's per-turn messages. **Role prompting** is the technique of assigning the model an identity/expertise ("You are a senior compliance analyst…") to bias its style and knowledge framing.

**Intuition.** The chat models you use were **instruction-tuned with distinct roles** (system / user / assistant, 1.2.4) and often **aligned** (1.3.4) to *prioritize* system instructions. So the system prompt is your most durable, high-leverage control: it applies to every turn, survives the conversation, and is where you place **guardrails, format contracts, and persona**. User turns then carry the actual task.

**What belongs in a system prompt (a practical checklist).**
- **Identity/role:** "You are a meticulous financial-document analyst."
- **Goal & scope:** what to do, and explicitly what *not* to do.
- **Rules/guardrails:** refusals, required disclaimers, safety limits.
- **Output contract:** format, length, language, JSON schema.
- **Tone/voice:** brand voice, formality.

**Example (system + user):**
```
SYSTEM:
You are a compliance-aware financial assistant.
Rules:
- Never give personalized investment advice; give general information only.
- Always end with: "This is not financial advice."
- If asked something outside personal finance, say you can't help.
Output: 3 short bullet points, plain English.

USER:
What are the risks of investing in long-duration bonds?
```
The system prompt governs *every* subsequent user turn — persona, guardrail, and format all at once.

**Best practices & pitfalls (interview gold).**
- Keep it **clear and ordered**; put non-negotiable rules explicitly.
- **Role prompting helps style/framing more than raw factual accuracy** — don't expect "You are a Nobel physicist" to fix wrong facts (that's a RAG/tool job).
- **System prompts are strong but not bulletproof:** a determined user can attempt **prompt injection / jailbreaks** to override them. Treat the system prompt as policy, but add real guardrails/validation for anything security-critical (revisited in Phase 8). **Never put secrets** (API keys, hidden data) in a system prompt assuming users can't extract them.
- Precedence (typical): **system > developer/tool instructions > user**, but alignment isn't perfect — design defensively.

**Python — reusable system prompt + task turns:**
```python
SYSTEM = (
    "You are a compliance-aware financial assistant.\n"
    "Rules: give general info only, never personalized advice; "
    "always end with 'This is not financial advice.'\n"
    "Format: at most 3 short bullet points in plain English."
)

def ask(question: str) -> str:
    r = client.chat.completions.create(
        model="gpt-5.5",
        messages=[
            {"role": "system", "content": SYSTEM},
            {"role": "user", "content": question},
        ],
        temperature=0.3,
    )
    return r.choices[0].message.content

print(ask("What are the risks of long-duration bonds?"))
```

**Combining all three (the real-world pattern).** Production prompts usually **stack** the techniques: a **system prompt** sets role + rules + format, **few-shot** examples pin the pattern, and the **user turn** is the zero-shot task. Layering them is the norm, not the exception.

**Interview angle.** "What's the difference between a system and a user prompt, and why does the model respect the system one?" (instruction tuning + alignment on roles). "Does role prompting improve factual accuracy?" (mainly style/framing, not facts). "How reliable are system-prompt guardrails?" (strong but injectable — add real validation). "Where would you put output-format rules?"

**Finance use cases.**
1. **Compliance guardrails:** A system prompt enforces mandatory disclaimers and forbids personalized advice across an entire advisory-chat session.
2. **Consistent analyst persona:** "You are a fixed-income analyst; explain to a retail client at an 8th-grade level" standardizes tone across all outputs.

**E-commerce use cases.**
1. **Brand voice enforcement:** A system prompt holds the store's tone, banned claims, and formatting for every generated description or reply.
2. **Scoped shopping assistant:** "Only answer questions about our catalog and policies; politely decline off-topic requests" keeps the bot on-task across turns.

# Wrap-Up: 2.1.1 Zero-Shot, Few-Shot, and Role/System Prompts

## The through-line (backward and forward)
Because a deployed model's weights are frozen (Phase 1), the **prompt** is our only lever, and it works *because* of instruction tuning + role-aware alignment (1.3.3–1.3.4). **Zero-shot** exploits that instruction-following directly; **few-shot** adds in-context demonstrations to pin task + format (at a token cost — 1.2.6); **role/system** prompts set durable persona, rules, and output contracts across the whole conversation. In practice you **stack all three**. This sets up the rest of Phase 2: **chain-of-thought & reasoning** (2.1.2) build on few-shot/zero-shot; **structured output & function-calling** (2.2) formalize the "output contract" you learned to write in system prompts; and everything feeds **RAG** (Phase 4, richer context) and **agents** (Phase 5, system prompts define the agent's role and tools).

## Rapid-fire interview checklist
- Define zero-shot, one-shot, few-shot; what is in-context learning (and does it change weights? — no).
- Why zero-shot works at all (instruction tuning).
- Few-shot: why format consistency, example order, label balance, and coverage matter.
- When many-shot needs signal fine-tuning instead of prompting.
- System vs user prompt; why the model prioritizes system; precedence order.
- Does role prompting fix facts? (no — style/framing).
- Guardrail reliability + prompt injection; never store secrets in prompts.
- Token-cost impact of few-shot (input tokens every call).
- The escalation ladder: zero-shot → few-shot → RAG → fine-tune.

## Mini glossary
**Zero-shot** — instruction, no examples.
**Few-shot / one-shot** — a handful / one worked example in the prompt.
**In-context learning (ICL)** — learning the task from prompt examples, no weight change.
**System prompt** — high-priority, conversation-wide instruction (persona, rules, format).
**Role prompting** — assigning an identity/expertise to steer style.
**Delimiter** — markers (```` ``` ````, XML tags) separating instruction from data.
**Output contract** — the format/length/schema you require.
**Prompt injection** — user input attempting to override system instructions.

## Hands-on
1. Build a zero-shot classifier with a fixed label set and `temperature=0`; measure consistency over 20 runs.
2. Convert it to few-shot with 3 demonstrations; compare accuracy and note the added input-token cost.
3. Shuffle the few-shot example order and observe any output changes (order sensitivity).
4. Write a system prompt with a persona + a hard guardrail + a JSON output contract; try to make a user turn violate the guardrail and see how it holds.
5. Stack all three (system + few-shot + user) for a finance or e-commerce extraction task and lock a JSON schema.
