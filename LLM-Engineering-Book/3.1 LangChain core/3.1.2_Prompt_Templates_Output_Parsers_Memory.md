## 3.1.2.0 Where this fits (the bridge from 3.1.1)

A chain is only as good as what enters it and what leaves it. The **prompt template** is the front door: it turns your variables (and conversation history, and few-shot examples) into the exact messages the model sees — and doing this *well and reusably* is what 2.1.3 argued for, now as a first-class Runnable. The **output parser** is the back door: it converts the model's raw message into the clean, typed value your code needs — the composable version of the parse-and-validate work from 2.2. And **memory** addresses the fact you learned in 2.3.1: **the API is stateless**, so if you want a multi-turn conversation, *something* has to carry prior turns forward and inject them back into the prompt. That "something" is memory.

So this topic is really about **the boundaries of the chain and how state crosses them**: shape the input (templates), shape the output (parsers), and persist the conversation across calls (memory). Get these right and your chains graduate from single Q&A to real, stateful assistants — the direct on-ramp to RAG (Phase 4) and agents (Phase 5).

> **One-line thesis:** *Prompt templates build the messages going in, output parsers extract typed values coming out, and memory carries the conversation between calls. Together they turn a stateless `prompt | model | parser` chain into a coherent, multi-turn application.*

## 3.1.2.a Prompt Templates in LangChain

**Definition.** A **prompt template** in LangChain is a Runnable that holds a **parameterized prompt** and produces a finished prompt (or list of chat messages) when given input variables. The two you'll use constantly are **`PromptTemplate`** (for a single text string, older completion-style) and **`ChatPromptTemplate`** (for a list of role-tagged messages — system/user/assistant — the modern default for chat models). This is the LangChain realization of the templating discipline from 2.1.3, now composable via `|`.

**Why `ChatPromptTemplate` is the workhorse, explained.** Chat models expect a *list of messages with roles* (recall chat templates and role tokens from 1.2.4 and system/user framing from 2.1.1). `ChatPromptTemplate.from_messages([...])` lets you declare exactly that structure with placeholders in each message, so your system-prompt rules (2.1.1.c), your few-shot examples (2.1.1.b), and the user's question all live in one declarative object. When `invoke`d with a dict of variables, it renders the placeholders and outputs the message list the model needs — and because it's a Runnable, it's simply the first link in `prompt | model | parser`.

**The pieces that make templates powerful:**
- **`MessagesPlaceholder`** — reserves a slot in the message list for a **list of messages injected at runtime**. This is the mechanism by which **conversation history** (memory, 3.1.2.c) or a running scratchpad gets spliced into the prompt. It's the single most important construct for turning a one-shot template into a conversational one.
- **`FewShotPromptTemplate` / few-shot chat templates** — a structured way to render a set of example input/output pairs into the prompt (the LangChain form of 2.1.1.b). You can even pair it with an **example selector** that chooses the most relevant examples per query (e.g., by embedding similarity — a nice bridge to Phase 4).
- **Partials (`.partial(...)`)** — pre-fill *some* variables now and leave the rest for later (e.g., bind a fixed `role` or today's date once, supply `question` per call). This keeps call sites clean.

**Intuition — a form with reserved sections.** Think of `ChatPromptTemplate` as a printed form: fixed instructions are pre-printed (system rules), some boxes you fill per request (`{question}`), and — crucially — there's a **blank insert page** (`MessagesPlaceholder`) where you staple in the running conversation each time. The form's layout never changes; what changes is what you slot into the marked places. That stable structure with well-defined insertion points is exactly what makes prompts reliable and reusable.

**Python — a conversational `ChatPromptTemplate` with a history slot and a partial:**
```python
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a {role}. Answer concisely and always add a disclaimer."),
    MessagesPlaceholder("history"),          # <- conversation history injected here (memory)
    ("user", "{question}"),
])

prompt = prompt.partial(role="fixed-income analyst")   # pre-fill the fixed variable

messages = prompt.invoke({
    "history": [],                            # list of prior messages (empty on turn 1)
    "question": "What is a bond ladder?",
})
# `messages` is the rendered message list, ready to pipe into a model.
```

**Interview angle.** "`PromptTemplate` vs `ChatPromptTemplate`?" "What is `MessagesPlaceholder` for?" (injecting a runtime list of messages — history/scratchpad). "How do partials help?" "How do few-shot templates + example selectors relate to 2.1.1/Phase 4?"

**Finance use cases.**
1. **Standardized advisory prompt:** One `ChatPromptTemplate` encodes the compliance system rules + a `MessagesPlaceholder` for the client's chat history, reused across every advisory session for consistency and auditability.
2. **Dynamic few-shot for extraction:** An example selector picks the most similar labeled filing examples per document, improving extraction accuracy without a bloated static prompt.

**E-commerce use cases.**
1. **Brand-voice template:** A partialed `role="brand copywriter"` template plus per-product variables generates on-brand descriptions at scale.
2. **Conversational shopping assistant:** `MessagesPlaceholder` splices the shopper's session history so the assistant remembers earlier preferences within the chat.

## 3.1.2.b Output Parsers

**Definition.** An **output parser** is a Runnable that sits at the end of a chain and **transforms the model's raw message into a usable Python value** — plain text, a dict, or a validated Pydantic object. It's the composable equivalent of the manual extraction + validation from 2.2. Parsers do two jobs: they can **inject "format instructions"** into the prompt (telling the model how to shape its output) and they **parse/validate** the response afterward.

**The common parsers, and when to use each.**
- **`StrOutputParser`** — the simplest: pulls the plain text string out of the model's message. Used in almost every "just give me the text" chain (as in 3.1.1's canonical example).
- **`JsonOutputParser`** — parses the output into a Python dict; can supply format instructions and even **stream partial JSON** as it arrives.
- **`PydanticOutputParser`** — parses *and validates* into a Pydantic model (2.2.3), giving you a typed, rule-checked object. It generates **format instructions** from your schema that you inject into the prompt so the model knows the exact shape to produce.
- **`StructuredOutputParser`** — a lighter schema parser for simple field sets without full Pydantic.

**The modern, preferred path — `with_structured_output()` (important 2026 guidance).** Rather than relying on prompt-based format instructions and hoping the model complies, LangChain now recommends **`model.with_structured_output(MySchema)`** when the underlying model supports **native structured outputs or tool/function calling** (2.2.1/2.2.2). This returns a Runnable that yields a **validated object directly**, using the provider's *guaranteed* schema mechanism under the hood rather than fragile text parsing. In other words, it's the LangChain front-end to the "structure by construction" you learned in 2.2.1 — and it's what you should reach for first. Fall back to `PydanticOutputParser` (prompt-instruction style) only for models without native structured-output support.

**Handling parse failures — fixing and retrying.** Even with format instructions, a model can emit output the parser can't read. LangChain provides **`OutputFixingParser`** (wraps a parser and, on failure, sends the broken output back to an LLM to *fix* it) and **`RetryOutputParser`** (re-runs with the error/context to try again). These are the composable form of the validate-and-re-prompt loop from 2.2.3 — but note that with `with_structured_output` on a capable model, you often don't need them, because validity is guaranteed at generation time.

**Intuition — a translator at the exit door.** The model speaks "message object"; your code speaks "typed Python value." The parser is the translator standing at the chain's exit, converting one to the other — and, in its stronger forms, *refusing to let malformed output through* (validation) or *sending it back for correction* (fixing parser). `with_structured_output` is the premium translator who arranges for the speaker to only ever produce well-formed sentences in the first place.

**Python — the preferred structured path vs the classic parser:**
```python
from pydantic import BaseModel, Field
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate

class BondFields(BaseModel):
    issuer: str
    coupon_rate: float = Field(ge=0, le=1)
    currency: str

model = ChatOpenAI(model="gpt-5.5", temperature=0)

# PREFERRED (2026): native structured output -> returns a validated BondFields
structured_model = model.with_structured_output(BondFields)
chain = ChatPromptTemplate.from_messages(
    [("user", "Extract bond fields:\n{doc}")]
) | structured_model
result = chain.invoke({"doc": "Acme 5% notes, USD"})   # -> BondFields(...), validated
print(result.issuer, result.coupon_rate)

# CLASSIC fallback: PydanticOutputParser injects format instructions + parses text
from langchain_core.output_parsers import PydanticOutputParser
parser = PydanticOutputParser(pydantic_object=BondFields)
prompt = ChatPromptTemplate.from_messages([
    ("user", "Extract bond fields.\n{format_instructions}\n\n{doc}")
]).partial(format_instructions=parser.get_format_instructions())
classic_chain = prompt | model | parser
```

**Interview angle.** "What does an output parser do (both jobs)?" (format instructions + parse/validate). "`StrOutputParser` vs `PydanticOutputParser`?" "Why prefer `with_structured_output` over prompt-based parsing?" (uses native guaranteed structured output/tool calling; returns validated objects). "What do `OutputFixingParser`/`RetryOutputParser` handle?"

**Finance use cases.**
1. **Booking-grade extraction:** `with_structured_output(BondRecord)` returns a validated record with amounts/currencies checked, feeding a ledger with no manual JSON handling.
2. **Consistent decision objects:** A structured parser guarantees `{decision, confidence, reasons[]}` shape for auditable credit decisions.

**E-commerce use cases.**
1. **Catalog attribute records:** `with_structured_output` yields validated `{material, color, size}` objects across a million SKUs with guaranteed shape.
2. **Self-healing legacy models:** For an older on-prem model without native structured output, `OutputFixingParser` recovers occasional malformed extractions automatically.

## 3.1.2.c Memory — Carrying the Conversation

**Definition.** **Memory** is the mechanism that **persists prior conversation turns (and other state) across the otherwise-stateless API calls** (2.3.1), then **injects the relevant history back into the prompt** on each new turn (via `MessagesPlaceholder`, 3.1.2.a). Without memory, every call is amnesiac; with it, the assistant "remembers" what was said earlier in the session.

**Why memory is a real engineering problem, not just "save the messages."** The naive approach — keep the full transcript and prepend it every turn — collides with two limits you already know. First, the **context window** (1.2.5): a long conversation will eventually exceed it. Second, **cost and latency** (1.2.6, 2.3.1): every prior turn you resend is billed as input tokens *every* turn, so a growing transcript means linearly growing cost per message. So memory strategies are really **strategies for deciding what to keep, summarize, or drop** — a trade-off between completeness and cost/fit.

**The main memory strategies (conceptually — these persist across framework versions):**
- **Full buffer** — keep the entire history. Simple, but unbounded (breaks on long chats).
- **Windowed / trimmed** — keep only the last *N* turns or the last *N* tokens. LangChain's **`trim_messages`** does exactly this (by count or token budget), the standard way to stay within the window. Simple and cheap; loses old context.
- **Summary memory** — periodically **summarize** older turns into a compact synopsis and keep that plus recent turns verbatim. Preserves the gist of long conversations at far fewer tokens; costs an extra summarization call and can lose detail.
- **Retrieval-backed (long-term) memory** — store past messages/facts in a vector store and **retrieve** only the relevant ones per turn (this *is* RAG applied to conversation history — Phase 4). Best for long-lived or cross-session memory.

**The important 2026 framing — where memory lives now.** In older LangChain, memory was a set of dedicated classes (`ConversationBufferMemory`, etc.). Those are **legacy/deprecated**. The current picture is:
- For **LCEL chains**, wrap the chain in **`RunnableWithMessageHistory`**, which automatically loads a session's history (keyed by a session id, from a store you provide) into a `MessagesPlaceholder` and saves new turns after each call.
- For **stateful/agentic apps**, memory is handled by **LangGraph's persistence layer (checkpointers)** — the graph's state (including message history) is saved between steps and across sessions, enabling long-term memory, resumption, and human-in-the-loop. This is the direction LangChain has pushed memory in 1.x, and it's why memory increasingly belongs to the agent layer (Phase 5) rather than to standalone chains.

**Intuition — memory is note-taking for an amnesiac expert.** The model is a brilliant consultant with no memory between meetings. Memory is the assistant who, before each meeting, hands the consultant a briefing: the full notes (buffer), just the last few pages (trim), an executive summary of older meetings plus recent notes (summary), or only the pages relevant to today's question (retrieval). The consultant is the same each time; the quality and cost of the conversation depend entirely on how well the briefing is curated.

**Python — a conversational chain with automatic message history (LCEL):**
```python
from langchain_core.runnables.history import RunnableWithMessageHistory
from langchain_community.chat_message_histories import ChatMessageHistory
from langchain_core.messages import trim_messages

# The chain from 3.1.2.a (prompt with a MessagesPlaceholder("history")) | model | parser
chain = prompt | model  # (StrOutputParser optional)

_store = {}
def get_history(session_id: str):
    return _store.setdefault(session_id, ChatMessageHistory())

conversational = RunnableWithMessageHistory(
    chain,
    get_history,                       # loads/saves history per session id
    input_messages_key="question",
    history_messages_key="history",    # matches MessagesPlaceholder("history")
)

cfg = {"configurable": {"session_id": "user-42"}}
conversational.invoke({"question": "What is a bond ladder?"}, config=cfg)
conversational.invoke({"question": "How is it different from a barbell?"}, config=cfg)
# Turn 2 automatically sees turn 1 -- the assistant "remembers".

# Keep the transcript within budget with trim_messages (windowing):
# trimmed = trim_messages(history.messages, max_tokens=2000, strategy="last", token_counter=model)
```

**Interview angle.** "Why is memory needed if you have the model's context?" (the API is stateless — you must resend history). "Name memory strategies and their trade-offs." (buffer/trim/summary/retrieval — completeness vs cost/window). "What's the modern way to add memory to an LCEL chain vs an agent?" (`RunnableWithMessageHistory` vs LangGraph checkpointers). "How does long-term memory relate to RAG?" (retrieve relevant past facts).

**Finance use cases.**
1. **Session-aware advisory chat:** `RunnableWithMessageHistory` keeps a client's questions in context within a session, with `trim_messages` bounding cost on long consultations.
2. **Summary memory for long reviews:** A lengthy portfolio-review conversation summarizes earlier turns to stay within the window while retaining the key decisions made.

**E-commerce use cases.**
1. **Preference continuity:** The shopping assistant remembers "I want vegan, under $50" from earlier in the chat so later recommendations respect it (via windowed/summary memory).
2. **Cross-session memory (retrieval):** A returning customer's past preferences are stored and retrieved per session, personalizing the experience across visits (RAG-style long-term memory, Phase 4).

# Wrap-Up: 3.1.2 Prompt Templates, Output Parsers & Memory

## The through-line (backward and forward)
A chain's quality lives at its **boundaries and its state**. **Prompt templates** (3.1.2.a — `ChatPromptTemplate`, `MessagesPlaceholder`, few-shot, partials) build the exact messages going in, realizing 2.1.3's templating discipline as a Runnable. **Output parsers** (3.1.2.b) turn the raw message into typed values — with **`with_structured_output`** now the preferred route (native guaranteed structured output/tool calling from 2.2.1/2.2.2, returning validated objects), and `OutputFixingParser`/`RetryOutputParser` as the composable form of 2.2.3's fix-and-retry. **Memory** (3.1.2.c) solves the stateless-API problem (2.3.1) by persisting history and re-injecting it via `MessagesPlaceholder`, with strategies (buffer/trim/summary/retrieval) that trade completeness against context-window fit and cost (1.2.5–1.2.6); modern memory lives in **`RunnableWithMessageHistory`** for chains and **LangGraph checkpointers** for agents. Together these make a chain conversational and production-shaped — and every one of them reappears in **RAG** (Phase 4: retrieval as dynamic context + retrieval-backed memory) and **agents** (Phase 5: scratchpads, persistent state, structured tool arguments).

## Quick reference
| Component | Purpose | Key API (current) |
|---|---|---|
| ChatPromptTemplate | Build role-tagged messages | `from_messages([...])`, `.partial()` |
| MessagesPlaceholder | Slot for runtime message list | history / agent scratchpad |
| Few-shot template | Inject example pairs | `FewShotChatMessagePromptTemplate` + example selector |
| StrOutputParser | Get plain text | end of most chains |
| PydanticOutputParser | Parse+validate to a model | `get_format_instructions()` |
| with_structured_output | **Preferred** typed output | `model.with_structured_output(Schema)` |
| OutputFixing/Retry parser | Recover malformed output | wrap a parser |
| Memory (chain) | Persist+inject history | `RunnableWithMessageHistory`, `trim_messages` |
| Memory (agent) | Persistent state | LangGraph checkpointers (Phase 5) |

## Selected interview questions (short answers)
1. **`PromptTemplate` vs `ChatPromptTemplate`?** Single text string vs a list of role-tagged messages (the chat-model default).
2. **What is `MessagesPlaceholder`?** A slot to inject a runtime list of messages — used for conversation history or an agent scratchpad.
3. **What two jobs does an output parser do?** Provide format instructions to the prompt and parse/validate the model's output.
4. **Why prefer `with_structured_output` over `PydanticOutputParser`?** It uses the model's native guaranteed structured output/tool calling and returns validated objects, instead of fragile prompt-based text parsing.
5. **What do `OutputFixingParser`/`RetryOutputParser` do?** Recover from parse failures by re-asking/fixing — the composable validate-and-retry (2.2.3).
6. **Why is memory needed at all?** The chat API is stateless; you must persist and resend prior turns.
7. **Memory strategies and trade-offs?** Buffer (complete, unbounded), trim/window (cheap, loses old context), summary (compact, costs a call/loses detail), retrieval (scalable long-term = RAG).
8. **Modern way to add memory?** `RunnableWithMessageHistory` for LCEL chains; LangGraph checkpointers for agents.
9. **What limits force memory strategies?** Context window (1.2.5) and per-turn token cost (1.2.6).
10. **How does long-term memory relate to RAG?** It's retrieval applied to past conversation/facts (Phase 4).

## Mini glossary
**ChatPromptTemplate** — template producing role-tagged messages.
**MessagesPlaceholder** — runtime slot for a list of messages.
**FewShotPromptTemplate** — renders example pairs into the prompt.
