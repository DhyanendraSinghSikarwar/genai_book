## 3.1.0 Where this fits (the bridge from Phase 2)

In Phase 2 a typical feature looked like a sequence of manual steps: build a prompt string from a template, call the API, pull `.choices[0].message.content` out of the response, parse it with Pydantic, retry on failure. Each step was fine on its own, but you wrote the plumbing between them every time, and swapping a piece (say, a different model, or adding a step) meant editing that plumbing. There was no shared "shape" to the steps, so nothing composed cleanly.

**LangChain's core insight is to give every step the *same interface*.** If a prompt template, a chat model, an output parser, a retriever, and even an arbitrary Python function all expose the *identical* set of methods — a way to "run" them on an input — then you can **snap them together like Lego**, in any order, and the composition itself is just another one of these units. That common interface is the **Runnable**, and the language for snapping them together is **LCEL**. This is what turns Phase 2's ad-hoc glue into clean, reusable pipelines — and it's the substrate that RAG (Phase 4) and agents (Phase 5) are built on.

> **One-line thesis:** *A Runnable is any step with a standard run interface; LCEL is the `|` pipe that composes Runnables into a chain; a chain is itself a Runnable. Because everything shares one interface, you get sync, async, batch, and streaming — and easy swapping — for free.*

---

## 3.1.1.a What LCEL Is — Declarative Composition

**Definition.** **LCEL (LangChain Expression Language)** is a **declarative** way to compose LangChain components into a pipeline using the **pipe operator `|`**. "Declarative" means you describe *what* the pipeline is (this step, then that step) rather than writing the imperative control flow (call this, capture the result, pass it to that). The composed pipeline is called a **chain**, and — crucially — the chain is itself a Runnable, so chains can be nested inside other chains.

**Why declarative composition is the right idea, explained.** When you write `chain = prompt | model | parser`, you're not just saving keystrokes over manual wiring. You're handing LangChain a *description* of your pipeline, and because it understands the whole structure, it can give you a set of powerful behaviors **automatically**, without you implementing any of them:

- **Uniform invocation modes.** Every LCEL chain supports `invoke` (one input), `batch` (many inputs, parallelized), `stream` (token-by-token — recall 2.3.2), and **async** versions of all three (`ainvoke`, `abatch`, `astream`). You write the chain once; you get all four execution styles for free. Implementing batch parallelism or streaming by hand (as in Phase 2) was real work — here it's inherent.
- **Streaming that "just works" end to end.** Because a chain knows its steps, it can stream output through the whole pipeline where possible (a chain preserves the streaming properties of its components), so the last step's tokens can surface to the user as they're produced.
- **Swappability.** Any component can be replaced with another of the same type (swap OpenAI for Anthropic, one parser for another) without touching the rest of the chain — the composition doesn't care about the concrete implementation, only the interface.
- **Built-in resilience and observability hooks.** LCEL chains integrate with retries, fallbacks, and tracing (LangSmith, covered in 3.3) as first-class features rather than bolt-ons.

**Intuition — LCEL is Unix pipes for LLM steps.** If you've ever written `cat file | grep error | sort | uniq`, you already understand LCEL. Each command does one thing, reads from the previous, and writes to the next; the `|` composes them; and you can reorder or swap any stage. LCEL applies exactly this philosophy to LLM building blocks: `prompt | model | parser` is the same mental model, and the elegance is the same — small, single-purpose units combined into something larger without any of them needing to know about the others.

**Interview angle.** "What is LCEL and why is it declarative?" "Name behaviors you get for free from an LCEL chain." (sync/async/batch/stream, swappability, tracing/retries/fallbacks). "Why is a chain itself a Runnable?"

**Finance use cases.**
1. **Reusable analysis pipeline:** A `prompt | model | parser` chain for extracting figures from filings is defined once and reused across features; swapping to a cheaper model for a bulk run is a one-line change, not a rewrite.
2. **Batch + streaming from one definition:** The same chain runs in **batch** for nightly processing of thousands of statements and in **stream** mode for an interactive analyst copilot — no duplicated code.

**E-commerce use cases.**
1. **Description-generation chain:** One LCEL chain generates product copy; running it over the whole catalog is `chain.batch([...])`, while the admin preview uses `chain.stream(...)`.
2. **Provider portability:** A support-reply chain can switch its model component between vendors for cost/availability without changing the surrounding prompt/parse logic.

## 3.1.1.b The Runnable — the Universal Interface

**Definition.** A **Runnable** is LangChain's base abstraction: **any object that implements a standard interface for taking an input and producing an output**. That interface centers on a small set of methods — most importantly `invoke(input)` (run on a single input and return the output), plus `batch`, `stream`, and their async counterparts (`ainvoke`, `abatch`, `astream`). Essentially every LangChain building block **is** a Runnable: prompt templates, chat models, output parsers, retrievers, tools, and — via wrappers — plain Python functions.

**Why one interface changes everything, explained.** The power here is *uniformity*. Because a chat model and a prompt template and a parser all answer to `invoke`, LCEL's `|` can connect any of them to any other: the pipe simply feeds one Runnable's output into the next Runnable's `invoke`. There's no special-case glue for "model to parser" versus "prompt to model" — it's the same mechanism every time. This is the abstraction that makes composition *possible*, and it's why the docs describe the Runnable as "the concept that makes chains, agents, and LCEL work." When you later meet retrievers (Phase 4) and agents (Phase 5), they slot into chains precisely because they, too, are Runnables.

**The key Runnable utilities you'll use constantly (each is itself a Runnable):**
- **`RunnableLambda`** — wraps an **arbitrary Python function** so it becomes a Runnable you can drop into a chain. This is your escape hatch for any custom logic (formatting, post-processing, calling your own code) mid-pipeline.
- **`RunnablePassthrough`** — passes its input through unchanged, optionally **adding** computed fields (`.assign(...)`). Indispensable for shaping data as it flows — e.g., carrying the original question forward while also retrieving context for it (the canonical RAG pattern, Phase 4).
- **`RunnableParallel`** (also written as a plain dict in a chain) — runs **several Runnables concurrently on the same input** and returns a dict of their outputs. Perfect for "fetch context *and* keep the question" or computing multiple things at once.
- **`RunnableBranch`** — conditional routing: pick which sub-Runnable to run based on the input (e.g., route a query to a "math" chain vs a "lookup" chain).
- **`.bind(...)`** — pre-set arguments on a Runnable (e.g., bind `stop` sequences or tools onto a model) so the composed chain carries them.
- **`.with_retry(...)` / `.with_fallbacks(...)`** — attach the resilience patterns from 2.2.3/2.3.3 declaratively to any Runnable.

**Intuition — Runnables are electrical components with a standard plug.** Imagine every device in your house used the exact same plug and voltage. You could chain any device after any other, add adapters (`RunnableLambda`) for custom gadgets, split power to several at once (`RunnableParallel`), or route it conditionally (`RunnableBranch`) — all without rewiring. The Runnable interface is that standard plug; LCEL is the wiring. The whole system is composable *because* the plug is universal.

**Python — Runnables and the utilities in action:**
```python
from langchain_core.runnables import (RunnableLambda, RunnablePassthrough,
                                       RunnableParallel)

# A plain function becomes a Runnable:
to_upper = RunnableLambda(lambda x: x.upper())
print(to_upper.invoke("hello"))          # "HELLO"  -- same invoke interface

# Run two things on the SAME input, concurrently, returning a dict:
parallel = RunnableParallel(
    original=RunnablePassthrough(),        # pass the input through unchanged
    shout=to_upper,                        # transform it
)
print(parallel.invoke("hi"))             # {"original": "hi", "shout": "HI"}
```

**Interview angle.** "What is a Runnable and what methods define it?" "Why does one interface enable composition?" "What do `RunnableLambda`, `RunnablePassthrough`, and `RunnableParallel` do?" "How do you add custom Python logic to a chain?" (`RunnableLambda`).

**Finance use cases.**
1. **Custom post-processing step:** A `RunnableLambda` normalizes extracted currency/date formats mid-chain, keeping domain logic inside the composable pipeline rather than scattered around it.
2. **Parallel enrichment:** `RunnableParallel` fetches a filing's summary and its risk-classification simultaneously from the same input, halving latency for a dashboard.

**E-commerce use cases.**
1. **Carry-forward pattern:** `RunnablePassthrough.assign` keeps the raw product title while adding generated attributes, so downstream steps have both.
2. **Conditional routing:** `RunnableBranch` sends a customer message to a returns-chain or a product-question-chain based on detected intent.

## 3.1.1.c Building Chains with the Pipe Operator

**Definition.** A **chain** is a composition of Runnables connected so that each step's **output becomes the next step's input**. In LCEL you build one with the `|` operator, which under the hood creates a **`RunnableSequence`** — "the most important composition operator in LangChain," used in virtually every chain. `prompt | model | parser` and `RunnableSequence(prompt, model, parser)` are equivalent; the pipe form is preferred for readability.

**The canonical chain, explained step by step.** The archetypal LangChain pipeline is **prompt template → chat model → output parser**, and walking its data flow makes the whole abstraction concrete:
1. You `invoke` the chain with a dict of variables (e.g., `{"topic": "bond ladders"}`).
2. The **prompt template** (a Runnable) fills its placeholders and outputs a formatted prompt / list of messages.
3. That prompt flows into the **chat model** (a Runnable), which returns a message object.
4. The message flows into an **output parser** (a Runnable) — e.g., `StrOutputParser` to get plain text, or a Pydantic parser to get a validated object (2.2.3) — producing the final, typed result.

Each arrow is just "output of one `invoke` becomes input of the next `invoke`." Because the chain is itself a Runnable, you immediately get `chain.invoke(...)`, `chain.batch([...])`, `chain.stream(...)`, and the async variants — the "for free" behaviors from 3.1.1.a.

**Why the output parser matters in the chain.** A raw model step returns a message object, not the clean value your code wants. The parser is the composable equivalent of the manual `.choices[0].message.content` + Pydantic validation you did in Phase 2 — now it's just the last link, swappable for a different output shape without touching the rest.

**Python — the canonical chain, plus a RAG-style shape (preview of Phase 4):**
```python
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnableParallel, RunnablePassthrough

model = ChatOpenAI(model="gpt-5.5", temperature=0)
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a concise financial analyst."),
    ("user", "Explain {topic} in two sentences."),
])

# Canonical chain: prompt -> model -> parser
chain = prompt | model | StrOutputParser()
print(chain.invoke({"topic": "bond ladders"}))         # single
print(chain.batch([{"topic": "duration"}, {"topic": "yield curves"}]))  # many, parallel
for token in chain.stream({"topic": "convexity"}):     # streamed
    print(token, end="", flush=True)

# RAG-shaped chain (Phase 4 preview): retrieve context AND keep the question,
# then feed both into a prompt -> model -> parser pipeline.
def retrieve(q): return "…relevant retrieved text…"     # stub retriever
rag_prompt = ChatPromptTemplate.from_messages([
    ("system", "Answer using ONLY the context."),
    ("user", "Context:\n{context}\n\nQuestion: {question}"),
])
rag_chain = (
    RunnableParallel(context=RunnableLambda(retrieve), question=RunnablePassthrough())
    | rag_prompt | model | StrOutputParser()
)
# rag_chain.invoke("What is a bond ladder?")
```
Notice how the RAG chain uses `RunnableParallel` + `RunnablePassthrough` (3.1.1.b) to build the `{context, question}` dict the prompt needs — the exact pattern you'll reuse throughout Phase 4.

**Interview angle.** "What does `|` create under the hood?" (`RunnableSequence`). "Walk through the data flow of `prompt | model | parser`." "Why include an output parser?" "How do you feed two inputs (context + question) into one prompt?" (`RunnableParallel`/dict + `RunnablePassthrough`).

**Finance use cases.**
1. **Summarize-and-structure chain:** `prompt | model | PydanticOutputParser` turns a filing section into a validated record in one composable pipeline, replacing Phase 2's manual parse/validate glue.
2. **Grounded Q&A skeleton:** The RAG-shaped chain above is the backbone of a compliance assistant that answers strictly from retrieved policy text.

**E-commerce use cases.**
1. **Attribute-extraction chain:** `prompt | model | parser` extracts `{material, color, size}` per product; `chain.batch(...)` scales it to the catalog.
2. **Support-answer chain:** A retrieve-then-answer chain drafts replies grounded in the help center, with the parser enforcing a consistent reply format.

## 3.1.1.d LCEL vs LangGraph — Choosing the Right Tool (2026)

**Definition & current guidance.** LCEL chains are **directed, mostly linear pipelines** — data flows forward through the steps. This is ideal for the vast majority of "prompt → model → parse", RAG, and simple tool-augmented flows. But some applications need **cycles, branching state, and control flow** — an agent that loops "reason → act → observe" until done (2.1.2/2.2.2), multi-agent coordination, human-in-the-loop pauses. For those, LangChain's guidance in 2025–2026 is to use **LangGraph** (Phase 5), which models the app as a **stateful graph** with nodes and edges, supporting loops, persistence, and interruption. The rule of thumb: **LCEL for pipelines, LangGraph for stateful/looping agents.**

**Why this distinction matters, explained.** LCEL's strength — a clean, declarative, forward pipeline — is also its limit: expressing "keep calling tools until the task is complete" as a pure pipe chain is awkward and fragile, because that's a *loop with state*, not a straight line. LangGraph was built precisely for that shape. Knowing the boundary keeps you from forcing LCEL to do something it's not meant to (a common beginner mistake) and tells you when to reach for the heavier tool. Importantly, **they compose**: LangGraph nodes are often themselves LCEL chains (again, because a chain is a Runnable). So LCEL isn't replaced by LangGraph — it's the building block inside it.

**Intuition — a pipeline vs a flowchart with loops.** LCEL is an assembly line: parts move forward, each station transforms them, out comes a product. LangGraph is a flowchart that can loop back, wait for a human, and hold state between steps — the right picture for an agent that revisits earlier stages. Use the assembly line when the work is linear; use the flowchart when the work has cycles and memory.

**Interview angle.** "When do you use LCEL vs LangGraph?" (linear pipelines vs stateful/looping agents). "Can you use both together?" (yes — LangGraph nodes are often LCEL chains). "Why is a loop awkward in pure LCEL?" (it's forward/linear; loops need managed state).

**Finance use cases.**
1. **LCEL for the pipeline, LangGraph for the workflow:** A document-QA *chain* (LCEL) becomes one node inside a *LangGraph* agent that loops to gather multiple documents before answering a complex diligence question.
2. **Human-in-the-loop approvals:** A high-value action flow uses LangGraph to pause for analyst confirmation (2.2.2 action gating), with each step implemented as an LCEL chain.

**E-commerce use cases.**
1. **Simple vs agentic support:** Straightforward FAQ answering is a single LCEL chain; a multi-step order-resolution assistant that checks status, initiates a return, and confirms uses LangGraph around LCEL sub-chains.
2. **Routing then pipeline:** A `RunnableBranch` (LCEL) handles simple intent routing, while genuinely multi-turn, stateful shopping journeys graduate to LangGraph.

# Wrap-Up: 3.1.1 LCEL, Runnables & Chains

## The through-line (backward and forward)
Phase 2's hand-wired steps become **composable** once every step shares one interface — the **Runnable** (`invoke`/`batch`/`stream` + async, 3.1.1.b). **LCEL** composes Runnables declaratively with the `|` pipe (3.1.1.a), producing a **chain** — a `RunnableSequence` whose canonical form is `prompt | model | parser` (3.1.1.c) — and because the chain is itself a Runnable, you get sync/async/batch/streaming, swappability, and tracing/retries/fallbacks for free. Utility Runnables (`RunnableLambda`, `RunnablePassthrough`, `RunnableParallel`, `RunnableBranch`, `.bind/.with_retry/.with_fallbacks`) let you shape data, run steps in parallel, add custom logic, route, and harden — declaratively. For linear pipelines LCEL is the tool; for **stateful, looping agents** you graduate to **LangGraph** (3.1.1.d, Phase 5), where LCEL chains become the nodes. This foundation carries directly into **3.1.2–3.1.3** (prompt templates/parsers/memory, model abstraction), **LlamaIndex** (3.2), **RAG** (Phase 4 — the RAG-shaped chain shown here), and **agents** (Phase 5).

## Quick reference
| Concept | What it is | Key API |
|---|---|---|
| Runnable | Universal step interface | `invoke`, `batch`, `stream`, `ainvoke`/`abatch`/`astream` |
| LCEL | Declarative composition | the `\|` pipe |
| Chain | Composed Runnables | `RunnableSequence` (`prompt \| model \| parser`) |
| RunnableLambda | Wrap a Python function | `RunnableLambda(fn)` |
| RunnablePassthrough | Pass input through / add fields | `.assign(...)` |
| RunnableParallel | Run steps concurrently → dict | dict literal in a chain |
| RunnableBranch | Conditional routing | `RunnableBranch((cond, r), default)` |
| Resilience | Retries/fallbacks/binding | `.with_retry()`, `.with_fallbacks()`, `.bind()` |
| LangGraph | Stateful/looping graphs | use for agents (Phase 5) |

## Selected interview questions (short answers)
1. **What is a Runnable?** LangChain's universal interface for a step: `invoke`/`batch`/`stream` (+ async). Almost every component is one.
2. **What is LCEL?** A declarative way to compose Runnables with the `|` pipe into a chain.
3. **What does `|` build under the hood?** A `RunnableSequence` — output of each step feeds the next.
4. **Behaviors you get for free from an LCEL chain?** Sync/async, batch, streaming, swappable components, and tracing/retry/fallback integration.
5. **Why is a chain itself a Runnable?** So chains nest and compose uniformly (and slot into LangGraph nodes).
6. **What are `RunnableLambda`/`RunnablePassthrough`/`RunnableParallel` for?** Wrap custom functions / pass-and-augment input / run steps concurrently returning a dict.
7. **How do you feed context + question into one prompt?** `RunnableParallel` (dict) with `RunnablePassthrough` — the RAG pattern.
8. **The canonical chain and its parts?** `prompt | model | parser` — template fills vars, model generates, parser extracts/validates the output.
9. **LCEL vs LangGraph — when each?** LCEL for linear pipelines; LangGraph for stateful, looping, human-in-the-loop agents (and LangGraph nodes are often LCEL chains).
10. **Why is a loop awkward in pure LCEL?** LCEL is forward/linear; loops need managed state, which LangGraph provides.

## Mini glossary
**Runnable** — universal step interface (`invoke`/`batch`/`stream`).
**LCEL** — LangChain Expression Language; `|`-based composition.
**Chain** — composed Runnables; a `RunnableSequence`.
**Pipe `|`** — composes two Runnables into a sequence.
**RunnableLambda** — wraps a Python function as a Runnable.
**RunnablePassthrough** — passes input through, can add fields.
**RunnableParallel** — runs Runnables concurrently → dict.
**RunnableBranch** — conditional routing between Runnables.
**Output parser** — final Runnable turning a message into text/typed data.
**LangGraph** — stateful graph framework for agents (Phase 5).
