## 1.4.0 Where this fits (the bridge from 1.3)

In 1.3 we saw the pipeline **pretrain → SFT → align** produce an *instruct model*. Different labs run that pipeline with different data, scale, and choices — yielding a **zoo of models** with different strengths, costs, context lengths, and licenses. In industry, "which model?" is one of the most frequent and consequential decisions you'll make: it drives quality, latency, cost, privacy/compliance, and lock-in. This chapter turns your *internal* understanding into *practical selection judgment*.

> **One-line thesis:** *There is no single "best" model — only the best model for a given task, budget, latency target, privacy requirement, and license. Your job is to reason about the trade-offs, not memorize a leaderboard.*

## 1.4.1 Closed / Proprietary Models (API-only)

**Definition.** **Closed (proprietary) models** are accessed **only via a hosted API**; the weights are not released. You send tokens to the provider's servers and get tokens back. You trade **control** for **convenience, top-tier quality, and managed scaling**.

**The major closed families (mid-2026 — illustrative):**
- **1.4.1.a OpenAI — GPT family** (GPT-5 / 5.5 generation): the versatile all-rounder with the deepest tooling/ecosystem and strong function-calling.
- **1.4.1.b Anthropic — Claude family** (Opus / Sonnet / Haiku tiers, plus frontier "preview" models): strong reasoning, coding, and writing; safety focus; long context.
- **1.4.1.c Google — Gemini family** (Gemini 3.x Pro / Flash): very long context, deep cloud integration, strong multimodal reasoning.
- **1.4.1.d xAI — Grok** (Grok 4): lean, agentic, cost-conscious option.

**Tiering (a pattern to recognize).** Providers ship a **capability/price ladder** — a small/fast/cheap tier (e.g., Flash/Haiku/mini) up to a large/expensive frontier tier (Pro/Opus). *You mix tiers*: cheap models for routing/simple subtasks, frontier models for hard reasoning (model routing, Phase 9).

**Trade-offs (interview gold).**
- ✅ Best quality, least ops burden, instant scaling, multimodal, maintained safety.
- ❌ **Per-token cost** at scale, **data leaves your perimeter** (privacy/compliance concern), **rate limits**, **vendor lock-in**, and **silent model updates** that can shift behavior (pin versions!).

**Python — calling closed APIs (two SDKs, near-identical shape):**
```python
# OpenAI
from openai import OpenAI
client = OpenAI()  # reads OPENAI_API_KEY
resp = client.chat.completions.create(
    model="gpt-5.5",  # pin the exact version for reproducibility
    messages=[{"role": "user", "content": "Summarize this 10-K risk section: ..."}],
    temperature=0,    # deterministic for extraction (see 1.3.5)
)
print(resp.choices[0].message.content)

# Anthropic
from anthropic import Anthropic
a = Anthropic()  # reads ANTHROPIC_API_KEY
msg = a.messages.create(
    model="claude-opus-4-8",
    max_tokens=512,
    messages=[{"role": "user", "content": "Explain a bond ladder in one line."}],
)
print(msg.content[0].text)
```

**Interview angle.** "When would you choose a closed API over an open model?" (speed to market, top quality, no infra, multimodal). "What are the risks?" (cost, privacy, lock-in, version drift). "How do you protect against silent model updates?" (pin versions, regression-test with your evals).

**Finance use cases.**
1. **Rapid prototyping:** Stand up a client-advisory prototype in days on a frontier API before committing to in-house hosting.
2. **Hardest reasoning:** Use a top closed model for nuanced regulatory interpretation where quality outweighs per-token cost — *if* data-handling terms permit.

**E-commerce use cases.**
1. **Fast launch:** Ship a capable shopping assistant quickly with no GPU ops.
2. **Multimodal product Q&A:** Use a strong multimodal API for image-based search ("find items like this photo").

## 1.4.2 Open-Weight Models (self-hostable)

**Definition.** **Open-weight models** have **downloadable weights**, so you can run them **privately** (your servers/VPC/on-prem), **fine-tune** them freely (Phase 6), and **control cost, latency, and data residency**. "Open-weight" ≠ fully "open-source": weights are released but training data/code often aren't, and licenses vary (see 1.4.3).

**The major open families (mid-2026 — illustrative and shifting):**
- **1.4.2.a Meta — Llama** (Llama 4 Scout/Maverick): broad ecosystem/tooling backbone; Scout notable for very long context.
- **1.4.2.b Mistral / Mixtral** (Mistral Large 3, Small 4): efficient dense + **MoE** models; notably moved to permissive **Apache-2.0**.
- **1.4.2.c Alibaba — Qwen** (Qwen 3.x): strong multilingual (200+ languages) and coding.
- **1.4.2.d DeepSeek** (V4 family): top open performance, **MoE**, long context, often **MIT-licensed** and aggressively priced.
- **1.4.2.e Zhipu — GLM, Moonshot — Kimi, Google — Gemma:** other strong open options across sizes.

> **Trend to know (2026):** several **Chinese labs** (DeepSeek, Alibaba/Qwen, Zhipu/GLM, Moonshot/Kimi) hold many top **open-weight** spots, and licenses have trended **more permissive** (MIT/Apache). This is a common "do you follow the field?" interview probe.

**Trade-offs (interview gold).**
- ✅ **Data privacy & residency** (nothing leaves your environment), **lower marginal cost at scale**, **full fine-tuning freedom**, no vendor lock-in, deployable offline/edge.
- ❌ You own the **GPUs, serving, and ops** (Phase 8: vLLM/TGI, KV-cache memory from 1.3.6), and the very best closed models may still lead on the hardest tasks. Also **you** are responsible for safety/guardrails.

**Python — running an open model locally (two common paths):**
```python
# Path A: HuggingFace transformers (full control, research/fine-tune)
from transformers import AutoModelForCausalLM, AutoTokenizer
tok = AutoTokenizer.from_pretrained("meta-llama/Llama-3.2-3B-Instruct")
model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3.2-3B-Instruct",
                                             device_map="auto", torch_dtype="auto")

# Path B: Ollama (dead-simple local serving; great for prototyping)
#   $ ollama run llama3.2
import ollama
r = ollama.chat(model="llama3.2",
                messages=[{"role": "user", "content": "Tag this product: ..."}])
print(r["message"]["content"])
```
*(For production throughput you'd serve open weights behind **vLLM** or **TGI** — Phase 8.)*

**Interview angle.** "Open vs closed — decision factors?" (privacy/compliance, cost at scale, customization, ops capacity, quality ceiling). "What does open-weight *not* guarantee?" (open data/training; permissive license). "What is MoE and why do open labs favor it?" (capacity at ~constant per-token compute — ties to 1.1.5/1.1.9).

**Finance use cases.**
1. **Data residency / privacy:** Self-host so sensitive customer and transaction data never leaves the bank's VPC — often a hard regulatory requirement.
2. **Cost at high volume:** Serve millions of internal document queries on owned GPUs, avoiding per-token API fees.

**E-commerce use cases.**
1. **Custom fine-tuned catalog model:** Fine-tune an open model on proprietary product/review data for tailored search & descriptions without sharing data externally.
2. **Low-latency / edge personalization:** Run a smaller open model close to the app for fast recommendations at scale.

## 1.4.3 Reading Model Cards, Licenses, and Benchmarks

Choosing well means reading three things critically: the **card**, the **license**, and the **benchmarks** — plus running **your own eval**.

### 1.4.3.a Model cards
**Definition.** A **model card** is the model's documentation: intended use, (high-level) training data, size/parameters, **context length**, modalities, capabilities, **limitations/biases**, and **license**. Reading it critically is a core practitioner skill.
- *What to extract fast:* parameters & whether MoE, context window, supported modalities, tool/function-calling support, languages, and the license.

### 1.4.3.b Licenses (check BEFORE you build — a real legal/interview trap)
**Definition.** The license governs **commercial use, redistribution, and fine-tuning rights**. Categories:
- **Permissive open-source:** **Apache-2.0, MIT** (e.g., recent Mistral, many DeepSeek releases) — broadest commercial freedom.
- **Community / custom licenses:** (e.g., some **Llama** releases) — generally usable but with conditions (e.g., very-large-scale usage clauses, naming/attribution, acceptable-use policies).
- **Non-commercial / research-only:** cannot ship commercially.
- **Closed API terms of service:** govern data usage, retention, and whether your prompts can be used for training (**check for zero-retention / no-train options** in regulated settings).

### 1.4.3.c Benchmarks (and their traps)
**Definition.** Standardized tests estimating capability. Common ones:
- **MMLU / MMLU-Pro** — broad knowledge; **GPQA Diamond** — hard graduate-level reasoning; **GSM8K / MATH** — math; **HumanEval / MBPP / SWE-bench (Verified)** — coding; **HELM** — holistic; **Chatbot Arena / LMArena (Elo)** — human-preference head-to-head; **MT-Bench** — multi-turn.
- **Traps (say these in interviews):**
  - **Contamination:** benchmark data leaked into training inflates scores.
  - **Overfitting to leaderboards:** models tuned to the test, not the task.
  - **Construct mismatch:** a benchmark rarely equals *your* task; frontier models now cluster within a few points, so **benchmarks rarely decide** real choices.
  - **Self-reported numbers** can be cherry-picked — prefer independent leaderboards.

### 1.4.3.d The practical selection framework (the actual answer interviewers want)
Weigh, roughly in this order:
1. **Task quality** on *your* eval set (not a public benchmark).
2. **Privacy / compliance / data residency** (often decides open vs closed outright).
3. **Cost** (per-token or GPU $) at your expected volume.
4. **Latency / throughput** needs (recall decode-bound latency, 1.3.6).
5. **Context length** and **modalities** required.
6. **License** compatibility with your product.
7. **Tool/function-calling** quality (crucial for agents, Phase 5).
8. **Ecosystem/tooling maturity & support**.

**Python — build your *own* mini-eval (the habit that matters most):**
```python
# Compare candidate models on YOUR task, not a public leaderboard.
tasks = [
    {"prompt": "Extract issuer, coupon, maturity as JSON: <bond doc>",
     "expected_keys": ["issuer", "coupon", "maturity"]},
    # ... 50-200 representative, labeled examples ...
]

def score_model(call_fn):
    passed = 0
    for t in tasks:
        out = call_fn(t["prompt"])            # your API/local call
        try:
            import json; data = json.loads(out)
            passed += all(k in data for k in t["expected_keys"])
        except Exception:
            pass
    return passed / len(tasks)

# score_model(lambda p: openai_call(p, model="gpt-5.5"))
# score_model(lambda p: ollama_call(p, model="qwen3"))
# Pick the cheapest model that clears your quality bar.
```

**Interview angle.** "How do you choose a model for a new project?" (walk the framework above). "Why not just trust MMLU/leaderboards?" (contamination, overfitting, construct mismatch). "What license pitfalls exist with open models?" "How would you benchmark models for *your* use case?" (build a labeled eval set + metric).

**Finance use cases.**
1. **Compliance-driven disqualification:** A model's data-retention/license terms can rule it out for a regulated deployment regardless of its benchmark scores.
2. **Task-specific eval over hype:** Benchmark candidates on your own filing-extraction test set (with faithfulness checks) before choosing — leaderboard rank is secondary.

**E-commerce use cases.**
1. **Cost/quality fit:** Pick the **cheapest** model that clears a quality bar on *your* product-Q&A eval, not the top-ranked model.
2. **Multilingual requirement:** Choose a model whose card documents strong coverage (e.g., Qwen's 200+ languages) for a global storefront.

# Wrap-Up: 1.4 Model Landscape

## The through-line (closing Phase 1)
The **pretrain → SFT → align** lifecycle (1.3) is run by many labs, producing **closed** (1.4.1) and **open-weight** (1.4.2) models that differ in quality, cost, context, modality, and license. You choose among them by reading **cards, licenses, and benchmarks** — and, above all, by running **your own eval** (1.4.3). Notice how the whole phase connects: attention's O(n²) cost (1.1) and tokenization (1.2) set the **context window**; the lifecycle (1.3) sets **capability and cost**; and here (1.4) those properties become **selection criteria**. This decision — *which model* — is the doorway to the rest of the course: prompting (Phase 2) and RAG (Phase 4) squeeze more from whatever model you pick; agents (Phase 5) demand good tool-calling; fine-tuning (Phase 6) is mostly an **open-weight** activity; serving (Phase 8) is where self-hosting choices (vLLM, KV cache) pay off.

## Rapid-fire interview checklist
- Closed vs open-weight: definitions and the full trade-off list.
- "Open-weight" ≠ "open-source"; what a license does and doesn't grant.
- The capability/price tiering pattern and why you mix tiers (routing).
- License categories: Apache/MIT vs community (Llama) vs non-commercial vs API ToS (data retention).
- Key benchmarks (MMLU-Pro, GPQA, GSM8K/MATH, HumanEval/SWE-bench, Arena Elo) and their **traps** (contamination, overfitting, construct mismatch).
- The 8-point selection framework; why *your* eval beats public leaderboards.
- MoE's role in open models; long-context leaders; multilingual leaders.
- How to guard against silent closed-model updates (pin versions + regression evals).

## Mini glossary
**Closed/proprietary model** — API-only; weights withheld.
**Open-weight model** — downloadable weights; self-hostable & fine-tunable.
**Open-source vs open-weight** — data/code openness vs weights-only.
**Model card** — model documentation (use, size, context, limits, license).
**Permissive license** — Apache-2.0 / MIT (broad commercial use).
**Community license** — usable with conditions (some Llama releases).
**Benchmark contamination** — test data leaked into training.
**Chatbot Arena / Elo** — human-preference ranking.
**MoE** — mixture-of-experts (capacity at low per-token compute).
**Version pinning** — fixing an exact model version for reproducibility.

## Hands-on (ties to Phase 1's deliverable)
1. Open the model cards for one closed and one open model; extract context length, modalities, tool-calling support, and license into a comparison table.
2. Build a 30–50 example labeled eval for a task you care about (e.g., finance JSON extraction) and run the `score_model` snippet against 2–3 models.
3. Estimate monthly cost for each candidate at your expected volume (reuse the token math from 1.2.6); combine with quality to pick.
4. Check the license of an open model you'd deploy commercially and write one sentence on whether it's allowed.
