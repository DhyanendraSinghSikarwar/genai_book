## 1.3.0 Where this fits (the bridge from 1.2)

By the end of 1.2 we could turn text into `input_ids` and feed them into the embedding table at the bottom of the Transformer (1.1.1). But a freshly initialized Transformer produces **gibberish** — its weights are random. Two processes turn it into something useful:

- **Training** — adjust the weights so the model predicts real text well, then *behaves* the way we want.
- **Inference** — run the trained weights forward to generate new text.

The elegant part: **both are built on one objective — predict the next token.** Training *learns* the next-token distribution; inference *samples* from it. Hold that unifying idea as we go.

> **One-line thesis:** *An LLM is a next-token predictor. Pretraining teaches it the world's text; fine-tuning teaches it to answer; alignment teaches it manners; inference is just repeated next-token sampling made fast by caching.*

## 1.3.1 The Core Objective: Next-Token Prediction (Causal Language Modeling)

**Definition.** A decoder-only LLM is trained with **causal language modeling (CLM)**: given tokens `x₁…xₜ`, predict the probability distribution of the next token `xₜ₊₁` over the vocabulary. Training minimizes the **cross-entropy loss** between the model's predicted distribution and the actual next token — equivalently, it maximizes the likelihood of the training text.

**Intuition.** "The capital of France is ___" → the model must put high probability on "Paris." Do this over trillions of tokens and, to predict well, the model is *forced* to learn grammar, facts, reasoning patterns, code, and style. Prediction is the *pretext*; understanding is the *by-product*.

**The math (interview-ready).** For a sequence, the loss is the average negative log-likelihood:
```
L = − (1/T) Σₜ log P(xₜ | x₁ … xₜ₋₁)
```
This connects straight to 1.1: the causal **mask** (1.1.3) is what makes "predict the next token without peeking at the future" valid, and the **LM head** (1.1.7) produces the logits scored here. **Perplexity** = `exp(L)` is the standard intrinsic metric (lower = better).

**Python — the loss in ~15 lines (this is literally what pretraining/SFT optimizes):**
```python
import torch
import torch.nn.functional as F

# logits: model output, shape (batch, seq_len, vocab_size)
# input_ids: the tokens, shape (batch, seq_len)
def causal_lm_loss(logits, input_ids):
    # Predict token t+1 from position t → shift by one
    shift_logits = logits[:, :-1, :].contiguous()      # drop last position
    shift_labels = input_ids[:, 1:].contiguous()       # drop first token
    loss = F.cross_entropy(
        shift_logits.view(-1, shift_logits.size(-1)),  # (B*T, V)
        shift_labels.view(-1),                          # (B*T,)
    )
    return loss

# Perplexity is just the exponential of the loss:
# perplexity = torch.exp(loss)
```
The **"shift by one"** is the whole trick: the label for position *t* is the input token at position *t+1*.

## 1.3.2 Stage 1 — Pretraining (learning the world from raw text)

**Definition.** **Pretraining** is large-scale, **self-supervised** CLM over massive, diverse text corpora (web, books, code, etc.). It produces a **base (foundation) model** with broad knowledge and language skill — but *no instinct to follow instructions*. A base model **continues** text rather than **answering** it.

**Intuition & the "self-supervised" point.** No human labels are needed — the text *is* its own label (the next word). That's why it scales to trillions of tokens; data is essentially free.

**Example (base vs instruct behavior).**
- Prompt: *"List three risks of investing in bonds."*
- A **base** model might continue: *"…is a common exam question. Also list three benefits…"* (it mimics documents).
- An **instruct** model (after 1.3.3–1.3.4) actually answers with three risks.

**Scaling laws (know the headline).** Performance improves predictably with model size, data, and compute (Kaplan et al.). **Chinchilla** (Hoffmann et al.) refined this: for a fixed compute budget, many early models were *undertrained* — you should scale **data** roughly in proportion to **parameters** (~20 tokens per parameter as a rule of thumb). This is why newer models are trained on far more tokens.

**Cost reality.** Pretraining is the overwhelmingly expensive stage (thousands of GPUs, weeks, millions of dollars). **You will almost never do this.** Practitioners *use* base models and adapt them — the subject of the next stages and of Phase 6.

**Interview angle.** "What is a base/foundation model?" "Why is pretraining self-supervised?" "State a scaling law; what did Chinchilla change?" "Base vs instruct model — behavioral difference?" "Where do hallucinations originate?" (partly here: pretraining optimizes *plausibility*, not *truth*).

**Finance use cases.**
1. **Choosing base vs API:** Understanding pretraining cost justifies *using* a pretrained model via API/open weights rather than training one for a finance app.
2. **Knowledge cutoff awareness:** Pretraining data has a cutoff date → the model won't know recent filings/prices, motivating RAG and tools later.

**E-commerce use cases.**
1. **Leverage general knowledge:** Pretrained world knowledge lets a model reason about product categories out-of-the-box.
2. **Cutoff → freshness gap:** New products/prices post-cutoff must come from retrieval, not the model's memory.

## 1.3.3 Stage 2 — Supervised Fine-Tuning (SFT) / Instruction Tuning

**Definition.** **SFT** continues training the base model on curated **`(instruction, ideal response)`** pairs, formatted with the model's **chat template** (1.2.4). It teaches the model to *respond to instructions* rather than merely continue text. The result is an **instruct/chat model**.

**Intuition.** Same objective (next-token prediction) — but now the "next tokens" are *good answers written by humans (or strong models)*. The model learns the *shape* of helpful responses. Quality and diversity of data matter far more than sheer volume ("a few thousand excellent examples beat a million mediocre ones").

**A key detail — loss masking on the prompt.** During SFT we usually compute loss **only on the response tokens**, not the prompt tokens (we don't want the model learning to generate users' questions). This is done by setting prompt-token labels to `-100` (ignored by cross-entropy).

**Python — SFT with HuggingFace `trl` (the standard, concise path):**
```python
from datasets import load_dataset
from trl import SFTTrainer, SFTConfig

# Dataset with a "messages" field in chat format:
# [{"role":"user","content":"..."},{"role":"assistant","content":"..."}]
dataset = load_dataset("json", data_files="finance_sft.jsonl", split="train")

trainer = SFTTrainer(
    model="meta-llama/Llama-3.2-1B",     # a small base model
    train_dataset=dataset,
    args=SFTConfig(
        output_dir="sft-finance",
        num_train_epochs=1,
        per_device_train_batch_size=2,
        learning_rate=2e-5,
        max_seq_length=2048,
    ),
    # trl applies the chat template and masks prompt tokens for you
)
trainer.train()
```
*(In Phase 6 you'll do this efficiently with LoRA/QLoRA instead of full fine-tuning.)*

**Interview angle.** "What does SFT do that pretraining doesn't?" "Why mask the loss on prompt tokens?" "Is more data always better for SFT?" (no — quality/diversity dominate). "What data format does SFT use?"

**Finance use cases.**
1. **Terminology & disclaimer style:** SFT on curated finance Q&A makes the model use correct terms and always include required risk language for an advisory assistant.
2. **Structured extraction behavior:** SFT on `(document → JSON)` pairs teaches consistent field extraction from statements/invoices.

**E-commerce use cases.**
1. **Brand voice:** SFT on approved copy makes generated descriptions match tone and formatting guidelines.
2. **Support answer style:** SFT on high-quality resolved tickets teaches concise, policy-aligned responses.

## 1.3.4 Stage 3 — Alignment: RLHF and DPO (teaching judgment & preferences)

**The problem it fixes.** SFT teaches *a* good response, but there are many possible responses and we want the model to prefer the ones **humans actually like** (helpful, honest, harmless, well-formatted). We optimize on **human preferences**.

### 1.3.4.a RLHF (Reinforcement Learning from Human Feedback)
**Definition (three steps):**
1. **Collect preferences:** humans compare model outputs and rank them (A better than B).
2. **Train a reward model (RM):** a model that scores any response with a scalar "how much humans would like this."
3. **Optimize the LLM with RL (typically PPO):** update the policy (the LLM) to maximize the RM's reward, with a **KL-divergence penalty** that keeps it close to the SFT model (so it doesn't "reward-hack" into gibberish).

**Intuition.** The reward model is a *learned stand-in for human taste*; RL nudges the LLM toward higher-taste outputs without drifting too far from its trained behavior. RLHF (InstructGPT, 2022) is what made ChatGPT-style helpfulness possible.

### 1.3.4.b DPO (Direct Preference Optimization) — the modern simpler alternative
**Definition.** DPO skips the separate reward model and RL loop. Given preference pairs `(prompt, chosen, rejected)`, it optimizes the LLM **directly** with a classification-style loss to make **chosen** more likely and **rejected** less likely, while staying close to a reference model. **Cheaper, more stable, no RL machinery** — hence its popularity in open models (plus variants like IPO, KTO, ORPO).

**Python — DPO with `trl` (conceptual, standard API):**
```python
from trl import DPOTrainer, DPOConfig
from datasets import load_dataset

# Each row: {"prompt": "...", "chosen": "...", "rejected": "..."}
prefs = load_dataset("json", data_files="finance_prefs.jsonl", split="train")

trainer = DPOTrainer(
    model="sft-finance",           # start from your SFT model
    ref_model=None,                # trl can create the reference internally
    train_dataset=prefs,
    args=DPOConfig(output_dir="dpo-finance", beta=0.1, learning_rate=5e-6),
)
trainer.train()   # pushes 'chosen' up, 'rejected' down, near the reference
```
`beta` controls how strongly to trust preferences vs stay near the reference model.

**The full pipeline, summarized (recite this):** **Pretrain** (knowledge) → **SFT** (follow instructions) → **RLHF/DPO** (match human preferences). Optionally add domain adaptation at various points.

**Interview angle.** "Explain RLHF end-to-end." "Why the KL penalty in PPO?" "RLHF vs DPO — why pick DPO?" "What is reward hacking?" "Where does 'alignment tax' come from?" (aligned models can lose a little raw capability).

**Finance use cases.**
1. **Safety/compliance preferences:** DPO/RLHF to strongly prefer responses that refuse unauthorized advice and include mandated disclosures.
2. **Preferred explanation style:** Tune toward clear, hedged, well-cited explanations of decisions that clients and regulators rate higher.

**E-commerce use cases.**
1. **Helpfulness tuning:** DPO on ranked support replies so the assistant favors concise, policy-compliant answers customers upvote.
2. **On-brand persuasion without overclaiming:** Preference-tune to prefer accurate, appealing copy over exaggerated claims that hurt trust/returns.

## 1.3.5 Inference Mechanics: Logits → Softmax → Sampling (decoding)

We now switch from *making* the model to *using* it. Everything below is **autoregressive** generation.

**Definition.** Given the tokens so far, the model outputs **logits** (one score per vocab token, from the LM head, 1.1.7). We convert logits to a probability distribution and **choose the next token** according to a **decoding strategy**, append it, and repeat until a stop token or `max_tokens`.

**The decoding controls — precise definitions (all common interview topics):**

- **1.3.5.a Temperature `T`.** Divide logits by `T` before softmax. **Low `T` (→0):** sharper distribution → more deterministic, focused, repetitive. **High `T` (>1):** flatter → more diverse/creative. `T→0` ≈ **greedy**.
- **1.3.5.b Greedy decoding.** Always take the `argmax` token. Deterministic; can be dull/repetitive.
- **1.3.5.c Top-k sampling.** Keep only the `k` highest-probability tokens, renormalize, sample.
- **1.3.5.d Top-p (nucleus) sampling.** Keep the smallest set of tokens whose cumulative probability ≥ `p`, renormalize, sample. **Adapts** the candidate pool to the model's confidence — the common default (Holtzman et al.).
- **1.3.5.e Beam search.** Track several candidate sequences ("beams") and keep the highest-probability overall sequence. Good for constrained tasks (translation); rarely used for open-ended chat (bland, expensive).
- **Repetition/frequency/presence penalties** discourage loops; **stop sequences** and `max_tokens` end generation.

**Python — implement the sampling knobs yourself (clarifies exactly what the API params do):**
```python
import torch
import torch.nn.functional as F

def sample_next_token(logits, temperature=1.0, top_k=None, top_p=None):
    # logits: shape (vocab_size,) for the last position
    if temperature <= 0:                      # greedy
        return int(torch.argmax(logits))
    logits = logits / temperature

    if top_k is not None:                     # keep top-k
        kth = torch.topk(logits, top_k).values[-1]
        logits[logits < kth] = float("-inf")

    if top_p is not None:                     # nucleus / top-p
        sorted_logits, sorted_idx = torch.sort(logits, descending=True)
        probs = F.softmax(sorted_logits, dim=-1)
        cum = torch.cumsum(probs, dim=-1)
        mask = cum > top_p
        mask[1:] = mask[:-1].clone(); mask[0] = False   # keep at least 1 token
        sorted_logits[mask] = float("-inf")
        logits = torch.full_like(logits, float("-inf")).scatter(0, sorted_idx, sorted_logits)

    probs = F.softmax(logits, dim=-1)
    return int(torch.multinomial(probs, num_samples=1))
```

**Python — the same via HuggingFace (what you'll actually use):**
```python
from transformers import AutoModelForCausalLM, AutoTokenizer

tok = AutoTokenizer.from_pretrained("meta-llama/Llama-3.2-1B-Instruct")
model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3.2-1B-Instruct")

msgs = [{"role": "user", "content": "Explain a bond ladder in one sentence."}]
inputs = tok.apply_chat_template(msgs, add_generation_prompt=True, return_tensors="pt")

out = model.generate(
    inputs,
    max_new_tokens=80,
    do_sample=True,     # False = greedy/deterministic
    temperature=0.7,
    top_p=0.9,
    top_k=50,
)
print(tok.decode(out[0][inputs.shape[-1]:], skip_special_tokens=True))
```

**Why this matters (reliability lens).** Sampling is *why the same prompt yields different answers*. For **structured/JSON, extraction, math, code** → use **low temperature (often 0)**; for **brainstorming/marketing copy** → higher temperature/top-p. Sampling also underlies **hallucination**: the model always emits a *plausible* next token even when it doesn't "know" — it never says "I have no distribution here."

**Interview angle.** "Walk through generation token-by-token." "Temperature vs top-k vs top-p — when each?" "Why are outputs non-deterministic and how do you make them deterministic?" "Why does greedy repeat?" "Why can't we fully trust generated facts?"

**Finance use cases.**
1. **Deterministic extraction/audit:** Temperature 0 (greedy) + JSON constraints for reproducible, auditable figure extraction from filings.
2. **Consistent regulated comms:** Low temperature keeps client-facing statements on-message and repeatable.

**E-commerce use cases.**
1. **Creative vs factual split:** Higher temperature for ad taglines; low temperature for spec sheets and policy answers.
2. **A/B copy variants:** Sampling diversity intentionally produces multiple description variants for conversion testing.

## 1.3.6 Prefill, Decode, and the KV Cache (why inference is fast — and memory-hungry)

**Two phases of inference (define both — a favorite interview distinction):**
- **1.3.6.a Prefill.** The entire prompt is processed **in parallel** in one forward pass (all positions at once). This produces the first output token and populates the cache. Compute-bound.
- **1.3.6.b Decode.** Output tokens are generated **one at a time**, each requiring a forward pass that attends over all previous tokens. Memory-bandwidth-bound and inherently sequential → this dominates latency (recall 1.2.6: latency scales with *output* length).

**The KV cache (connecting to 1.1.9).** Naively, generating token *n* would recompute Keys/Values for all `n−1` prior tokens every step — hugely wasteful. The **KV cache** stores past tokens' Keys and Values so each decode step only computes the **new** token's Q and reuses cached K/V. This turns per-step cost from re-processing the whole prefix into a single-token step.

**The trade-off & why it constrains serving.** KV cache **trades memory for speed**. Its size ≈ `2 (K,V) × layers × kv_heads × head_dim × seq_len × batch × dtype_bytes`. So **longer context and bigger batches consume large GPU memory** — the core capacity constraint in production. Mitigations you learned in 1.1.9: **GQA/MQA** (fewer KV heads), **PagedAttention** (vLLM) for efficient memory management, and quantized KV caches.

**Python — see the KV cache in action (HuggingFace exposes it as `past_key_values`):**
```python
import torch
# Manual decode loop showing cache reuse (simplified)
inputs = tok("Interest rates are", return_tensors="pt")
out = model(**inputs, use_cache=True)          # prefill: process full prompt
past = out.past_key_values                     # <-- the KV cache
next_id = out.logits[:, -1].argmax(dim=-1, keepdim=True)

for _ in range(20):                            # decode: one token at a time
    out = model(input_ids=next_id, past_key_values=past, use_cache=True)
    past = out.past_key_values                 # cache grows by one token each step
    next_id = out.logits[:, -1].argmax(dim=-1, keepdim=True)
    # only the NEW token is fed in — prior K/V are reused from `past`
```
Notice we feed only the **single new token** after prefill; the cache supplies the rest.

**Interview angle.** "Prefill vs decode — what differs?" "What is the KV cache and why does it exist?" "How does KV-cache size scale, and why does it limit batch size / context length?" "How do GQA/MQA and PagedAttention help?" "Why is decode latency roughly linear in output length?"

**Finance use cases.**
1. **Long-filing throughput planning:** KV-cache memory determines how many concurrent long-document analyses a GPU can serve — a real capacity/cost calculation.
2. **Low-latency desk assistant:** Because decode dominates latency, capping output length and using GQA keeps responses within SLA.

**E-commerce use cases.**
1. **Concurrent shopper sessions:** KV-cache memory sets how many simultaneous chats fit per server; GQA/paged attention raise that ceiling.
2. **Streaming UX:** The KV cache enables fast token-by-token streaming for a responsive storefront assistant.

## 1.3.7 Practical Inference: Determinism, Hallucination, and Guardrails

**Definition & practical notes.**
- **Determinism.** For reproducible outputs: `do_sample=False` (greedy) or `temperature=0`, plus a fixed seed. *Caveat:* full bit-for-bit determinism can still be broken by batching, hardware, and floating-point non-associativity — important when reproducibility is a compliance requirement.
- **Hallucination.** The model outputs fluent, plausible tokens regardless of truth (roots in 1.3.2 pretraining + 1.3.5 sampling). Mitigations you'll build later: **RAG** grounding + citations (Phase 4), **tool use** for facts/math (Phase 5), lower temperature, and asking the model to abstain when unsure.
- **Structured output & stopping.** Low temperature + JSON schemas / constrained decoding + stop sequences make outputs parseable and bounded (Phase 2).

**Interview angle.** "How do you make an LLM feature reproducible?" "Why do models hallucinate and how do you reduce it?" "Given a factual finance question, how do you ensure correctness?" (RAG + tools + low temperature + citations, not raw generation).

**Finance use cases.**
1. **Auditability:** Deterministic settings + logged prompts/outputs support audit trails for regulated decisions.
2. **Grounded answers only:** Combine low temperature with RAG so figures come from source documents, not the model's memory.

**E-commerce use cases.**
1. **No invented specs:** Ground product answers in the catalog (RAG) to avoid hallucinated features that drive returns.
2. **Reliable structured tags:** Deterministic JSON output for attribute tagging keeps the catalog pipeline stable.

# Wrap-Up: 1.3 Training & Inference Lifecycle

## The through-line (backward and forward)
Fed by tokens (1.2) into the architecture (1.1), an LLM is trained on a single objective — **next-token prediction** (1.3.1). **Pretraining** (1.3.2) learns broad knowledge self-supervised; **SFT** (1.3.3) teaches it to follow instructions; **RLHF/DPO** (1.3.4) aligns it to human preferences. At **inference**, the LM head's **logits** become a distribution we **sample** from via temperature/top-k/top-p (1.3.5), executed in a **prefill + decode** loop accelerated by the **KV cache** (1.3.6) — which reconnects to attention's O(n²) cost and the context window from 1.1–1.2. Practical concerns — **determinism and hallucination** (1.3.7) — set up *why* the rest of the course exists: **RAG** (context limits + hallucination, Phase 4), **tools/agents** (bad arithmetic + real actions, Phase 5), **fine-tuning** (a hands-on redo of SFT/DPO, Phase 6), and **serving** (KV cache + throughput, Phase 8).

## Mini glossary
**CLM** — causal language modeling (next-token prediction).
**Perplexity** — `exp(loss)`; lower is better.
**Base vs instruct** — continues text vs follows instructions.
**SFT** — supervised fine-tuning on instruction/response pairs.
**RLHF** — reward-model + RL (PPO) alignment on human preferences.
**DPO** — direct preference optimization; no separate RM/RL.
**Logits** — pre-softmax vocabulary scores. **Temperature/top-k/top-p** — decoding controls.
**Prefill/decode** — parallel prompt pass vs one-token-at-a-time generation.
**KV cache** — cached past Keys/Values for fast decoding.
**Hallucination** — fluent but unfounded output.
