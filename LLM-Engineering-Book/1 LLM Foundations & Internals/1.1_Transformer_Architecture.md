## 1.1.0 Why the Transformer exists (the bridge)

You already know embeddings, softmax, and backprop. The Transformer is what happens when we ask: *"How can a network relate every word in a sentence to every other word, all at once, and still train fast?"*

**What came before.** Recurrent networks (RNN/LSTM/GRU) read text **sequentially**, carrying a hidden state forward one token at a time. Two fatal limitations:

1. **No parallelism in time.** Token *t* can't be computed until token *t−1* is done → slow training, poor GPU utilization.
2. **Long-range forgetting.** Information from token 1 is repeatedly overwritten by the time you reach token 500; gradients vanish over long distances even with LSTM gating.

Convolutional sequence models parallelized better but still needed many stacked layers to connect distant tokens (their "receptive field" grew slowly).

**The Transformer's bet.** Replace recurrence entirely with **attention**: a single operation that lets any token directly read from any other token in **one step**, and that runs as big matrix multiplications (ideal for GPUs). The 2017 paper's title — *Attention Is All You Need* — is literally the thesis. Everything else in this chapter is scaffolding that makes attention **stable, ordered, deep, and trainable**.

> **Mental model for the whole chapter:** *Attention mixes information across positions; the rest of the block refines each position; positional signals restore word order; residuals + normalization let us stack it very deep; the output head turns vectors back into words.*

## 1.1.1 Input Representation: Token Embeddings

**Definition.** Before any attention happens, each token ID (from the tokenizer) is mapped to a dense vector via a learned **embedding matrix** `E ∈ ℝ^(V × d_model)`, where `V` is vocabulary size and `d_model` is the model's hidden dimension (e.g., 768, 4096). Row *i* is the embedding for token *i*.

**Intuition.** The embedding is the token's *starting* meaning — a point in space where similar tokens sit near each other. Attention layers then *update* this vector using context, so by the top of the network the vector for "bank" has moved toward its context-specific sense.

**Example.** With `d_model = 4096`, the token "loan" becomes a 4096-dimensional vector. It begins generic; after several attention layers in the sentence "apply for a home loan," it absorbs "home" and "apply," sharpening toward the mortgage sense.

**Two practical details interviewers probe:**
- **Embedding scaling.** The original Transformer multiplies embeddings by √d_model before adding positional signals, to balance magnitudes.
- **Weight tying.** The input embedding matrix is often *shared* with the output projection (1.1.7), saving parameters and improving quality.

**Interview angle.** "What is the shape of the embedding matrix and what does a row mean?" "Why tie input and output embeddings?" "How is a token embedding different from a word2vec embedding?" (word2vec is *static/context-free*; Transformer embeddings become *contextual* after attention.)

**Finance use cases.**
1. **Semantic search over filings:** Embeddings place "liquidity risk" near "cash shortfall," enabling meaning-based retrieval of relevant 10-K passages rather than exact keyword matches.
2. **Entity normalization:** "JPM," "JPMorgan," "J.P. Morgan Chase" land close in embedding space, helping deduplicate counterparties across systems.

**E-commerce use cases.**
1. **Product similarity:** Embeddings of titles/descriptions cluster related items ("running shoes" near "trainers"), powering "similar products" and recommendations.
2. **Query understanding:** "cheap wireless earbuds" embeds near "budget bluetooth headphones," so search retrieves relevant SKUs despite different wording.

## 1.1.2 Positional Encoding: Restoring Order

**The problem it fixes.** Self-attention (next section) treats its input as a **set** — it is **permutation-invariant**. On its own it cannot tell "dog bites man" from "man bites dog." We must **inject position** into the token vectors.

**Definition.** A positional encoding adds (or otherwise mixes in) position information so the model knows each token's place in the sequence. It is applied at/near the input, on top of token embeddings.

**The four variants you must be able to compare:**

- **1.1.2.a Sinusoidal (fixed) — original 2017 Transformer.** Uses sine and cosine functions of different frequencies; each position gets a unique, deterministic vector added to its embedding.
  - *Why sinusoids?* Different frequencies let the model represent both fine and coarse position; linear combinations can express **relative** offsets, and it needs **no parameters** and can extrapolate to unseen lengths (in principle).
  - *Example:* position 3 and position 50 get distinct additive vectors, so identical tokens at different places differ.

- **1.1.2.b Learned absolute positions — BERT, GPT-2.** A trainable vector per position index.
  - *Pro:* flexible, data-driven. *Con:* **hard cap** at the maximum trained position; no natural extrapolation beyond it.

- **1.1.2.c Rotary Positional Embeddings (RoPE) — Llama, Mistral, Qwen (modern default).** Instead of *adding* a vector, RoPE **rotates** the Query and Key vectors by an angle proportional to their position. Because the dot product of two rotated vectors depends on their **difference** in angle, RoPE encodes **relative position directly inside attention**.
  - *Why it dominates now:* strong quality, and it supports **context-length extension** at inference via frequency scaling (NTK-aware, **YaRN**), letting a model trained at 8k run at 32k+.

- **1.1.2.d ALiBi (Attention with Linear Biases).** Adds a distance-proportional **penalty** to attention scores (farther-back tokens get more negative bias). No learned positional parameters and excellent **length extrapolation**.

**Interview angle.** "Why do Transformers need positional encodings but RNNs don't?" (RNNs are inherently ordered.) "Absolute vs relative position — trade-offs?" "How does RoPE encode relative position, and how does it enable context extension?" "If you must serve 4× the training context, what do you change?" (RoPE scaling / YaRN or a model trained/finetuned for long context.)

**Finance use cases.**
1. **Ordered event/transaction sequences:** "large deposit → immediate wire out" vs the reverse carries different risk; positional signals preserve this order when a model reads serialized histories.
2. **Long prospectus processing:** RoPE-style extension lets a single pass cover a lengthy document while keeping "definitions-before-references" ordering intact.

**E-commerce use cases.**
1. **Session/clickstream modeling:** The order search→view→add-to-cart→checkout predicts intent; positions keep the funnel meaningful.
2. **Long listings with late disclaimers:** Correct positional handling ensures warranty/return text near the end is still attended to when generating comparisons or answers.

## 1.1.3 Self-Attention (Scaled Dot-Product Attention)

This is the heart of the architecture — the mechanism the whole chapter is built around.

**Definition.** Self-attention lets each token compute a new representation as a **weighted sum of all tokens' values**, where the weights reflect **relevance**. "Self" because the sequence attends to itself.

**The Query/Key/Value framing (also your foundation for RAG later):**
- From each token's vector we produce three projections via learned matrices `W_Q, W_K, W_V`:
  - **Query (Q):** what this token is *looking for*.
  - **Key (K):** what each token *advertises* (an index/label).
  - **Value (V):** the actual *content* to be aggregated.

**The formula — memorize and be able to derive it:**

```
Attention(Q, K, V) = softmax( (Q · Kᵀ) / √d_k ) · V
```

Step by step:
1. **Scores:** `Q · Kᵀ` — a dot product of every Query with every Key → an `n × n` matrix of raw relevance scores.
2. **Scale by √d_k:** divide by the square root of the key dimension.
   - **Why?** For large `d_k`, dot products grow large in variance, pushing softmax into a near one-hot region where gradients vanish (saturation). Dividing by √d_k normalizes variance to ~1, keeping gradients healthy. *This is exactly why it's "scaled" dot-product attention* — a top interview point.
3. **Softmax:** turn each row of scores into a probability distribution (weights sum to 1).
4. **Weighted sum of Values:** multiply weights by `V` → each token's context-aware output.

**Masking — two kinds (don't confuse them):**
- **Causal (look-ahead) mask** — used in decoders/generation. Future positions are set to **−∞** before softmax so their weight becomes 0; a token can only attend to itself and earlier tokens. This is what makes next-token prediction valid.
- **Padding mask** — ignores `<pad>` filler tokens added to make a batch rectangular, so they contribute nothing.

**Cross-attention (preview of 1.1.6).** When **Q comes from one sequence and K,V from another**, it's *cross-attention* (e.g., decoder attending to an encoder's output, or fusing retrieved documents). Contrast with *self-attention* where Q,K,V come from the same sequence.

**Worked example.** Sentence: *"The bank raised interest rates."* Processing "bank": its Query matches strongly with the Keys of "interest" and "rates," so softmax gives them high weight; the resulting weighted sum of Values shifts "bank" toward the **financial** meaning (not riverbank). This is **contextual disambiguation**, the property RNNs struggled to deliver over distance.

**Cost — the fact that governs LLM engineering.** Attention is **O(n²·d)** in sequence length `n`: the score matrix is `n × n`. Doubling context ~quadruples compute and memory. This single line explains FlashAttention, sliding-window/sparse attention, and the KV cache — and why long context is expensive.

**Interview angle.** "Implement scaled dot-product attention." "Why divide by √d_k?" "Causal vs padding mask." "Self- vs cross-attention." "What's the time/space complexity and why does it matter in production?"

**Finance use cases.**
1. **Filing/earnings analysis:** Attention lets "material adverse effect" pull in a distant risk paragraph, correctly weighting it for risk flagging or summarization across long documents.
2. **Fraud-signal linking:** Across a message, attention connects scattered cues ("gift card," "urgent," "wire," "overseas") regardless of distance, improving fraud scoring.

**E-commerce use cases.**
1. **Search intent binding:** For "red running shoes for flat feet," attention binds "flat feet" to "shoes" (not "red"), surfacing orthopedic options.
2. **Aspect-based review understanding:** Attention ties sentiment to the right aspect ("great battery," "cracked screen") for accurate product-page summaries.

## 1.1.4 Multi-Head Attention

**The problem it fixes.** A single attention computation forces the model to blend *all* kinds of relationships into one weighting. But language has many simultaneous relationships (syntax, coreference, position, topic). One head is a bottleneck.

**Definition.** Multi-head attention runs **h independent attention functions ("heads") in parallel**, each with its own smaller `W_Q, W_K, W_V` (dimension `d_k = d_model / h`). Their outputs are **concatenated** and passed through an output projection `W_O`.

**Intuition.** Each head can specialize in a different subspace/relationship:
- one head tracks **subject↔verb** agreement,
- another resolves **pronoun↔antecedent** (coreference),
- another attends to **adjacent** tokens (local patterns),
- another links **distant** but topically related tokens.
Concatenating gives the next layer a rich, multi-perspective view. Total compute is similar to one big head because each head is `1/h` the width.

**Example.** In *"The customer said she would return it because it was defective,"* one head links "she"→"customer," another links "it"→(the product), another ties "defective"→"return." Multi-head lets all these be captured at once.

**Modern efficiency variants (know these — they're everywhere now):**
- **Multi-Query Attention (MQA):** all heads **share a single K and V** (many Qs, one K/V). Slashes the KV-cache memory (see 1.1.9) at a small quality cost.
- **Grouped-Query Attention (GQA):** a **middle ground** — heads are grouped, each group shares K/V. Used by Llama 2/3, Mistral. Best quality/memory trade-off, so it's the current default.

**Interview angle.** "Why multiple heads instead of one?" "What are the shapes through multi-head attention?" "What do MQA and GQA optimize, and what's the trade-off?" "Do heads truly specialize?" (empirically yes, though not perfectly interpretable).

**Finance use cases.**
1. **Multi-relationship contract/filing reasoning:** Different heads track defined terms, parties, and conditions simultaneously in dense legal-financial text.
2. **Serving efficiency (GQA/MQA):** Reduced KV cache lets a bank serve more concurrent document-analysis requests per GPU.

**E-commerce use cases.**
1. **Complex query parsing:** Separate heads handle attribute ("wireless"), constraint ("under $50"), and category ("earbuds") in one search query.
2. **High-throughput assistants (GQA/MQA):** Smaller KV cache means more simultaneous shopper chats on the same hardware.

## 1.1.5 The Position-wise Feed-Forward Network (FFN)

**The problem it fixes.** Attention **mixes** information *across* tokens but is largely a (weighted) linear combination. We still need per-token **nonlinear processing** and capacity to *store and transform* knowledge.

**Definition.** The FFN is applied **independently and identically to each position**: two linear layers with a nonlinearity between them, typically expanding the hidden dimension ~4× and projecting back:

```
FFN(x) = W₂ · activation(W₁ · x + b₁) + b₂     # e.g., d_model → 4·d_model → d_model
```

**Intuition.** If attention is "gather relevant context," the FFN is "think about what you gathered." It's where a large share of parameters live and where much factual/knowledge processing happens. This is also the block that **Mixture-of-Experts (MoE)** makes *sparse* (see 1.1.9).

**Activations (evolution — a nice interview detail):**
- **ReLU** — original Transformer.
- **GELU** — smoother; used by BERT, GPT-2.
- **SwiGLU** (a gated variant) — used by Llama/PaLM; better quality per parameter, the modern favorite.

**Example.** After attention has assembled "home" + "loan" + "apply" into a token's representation, the FFN transforms it into features useful for predicting the next token (e.g., "application," "rate").

**Interview angle.** "What does the FFN do that attention can't?" "Why the 4× expansion?" "Why is the FFN applied position-wise?" "What is SwiGLU and why use it?" "Where do most parameters sit — attention or FFN?" (usually FFN).

**Finance use cases.**
1. **Domain knowledge capacity:** FFN width lets the model encode financial concepts (amortization, covenants) used when explaining a credit decision.
2. **Cost-efficient scale (MoE FFN):** Sparse experts serve high query volumes for internal finance assistants at lower compute.

**E-commerce use cases.**
1. **Attribute/world knowledge:** FFN capacity stores product knowledge (compatibility, materials) for answering "will this fit model X?"
2. **Efficient catalog assistants (MoE):** Sparse FFNs handle millions of SKUs cost-effectively.

## 1.1.6 Making It Deep: Residual Connections & Layer Normalization

**The problem it fixes.** We want to stack **dozens** of attention+FFN blocks. Naively, deep nets suffer vanishing/exploding gradients and unstable activations. Two tools fix this.

- **1.1.6.a Residual (skip) connections.**
  - **Definition:** add a sublayer's input to its output: `x + Sublayer(x)`.
  - **Why:** creates a gradient "highway" straight to earlier layers (borrowed from ResNet), so very deep stacks train; each layer learns a *refinement* of the representation rather than rebuilding it from scratch.

- **1.1.6.b Layer Normalization.**
  - **Definition:** normalize a token's activation vector across the **feature dimension** (per token, not per batch — unlike BatchNorm), then scale/shift with learned parameters. Stabilizes training and smooths the loss landscape.
  - **Pre-LN vs Post-LN (real engineering choice):** *Post-LN* (norm after add) was original but can be unstable/needs warmup for deep models; *Pre-LN* (norm before the sublayer) is far more stable at depth and is the **modern default**.
  - **RMSNorm:** a cheaper LayerNorm variant that skips mean-centering (normalizes by root-mean-square only); used in Llama-family models for speed.

**The canonical block (recite this):** For a Pre-LN decoder block:
```
x = x + MultiHeadCausalAttention(LayerNorm(x))
x = x + FFN(LayerNorm(x))
```
Stack **N** of these (N = 12, 32, 80, … depending on size).

**Interview angle.** "Why residual connections?" "LayerNorm vs BatchNorm — why LayerNorm for sequences?" "Pre-LN vs Post-LN trade-offs?" "What is RMSNorm and why is it used?"

**Finance use cases.**
1. **Deep nuanced-language reasoning:** Many stable layers resolve compounding qualifiers ("not unlikely to be material") common in legal/financial text.
2. **Stable domain fine-tuning:** Pre-LN/RMSNorm stability matters when fine-tuning on smaller, noisier proprietary financial corpora.

**E-commerce use cases.**
1. **Rich multi-step reasoning:** Depth enables comparing several products across many attributes in one response.
2. **Reliable fine-tunes:** Stable normalization helps when adapting a model to a brand's catalog and tone.

## 1.1.7 Assembling the Model: Encoder, Decoder, and the Output Head

Now we place blocks into full architectures — and connect back to *which tasks* each shape serves.

**1.1.7.a Three architectural shapes.**
- **Encoder-only (BERT):** **bidirectional** self-attention (every token sees the whole input). Trained via masked-language modeling. **Best for understanding:** classification, embeddings, extraction, retrieval. Not a text generator.
- **Decoder-only (GPT, Llama, Claude, Mistral):** **causal** self-attention (see only the past). Trained on next-token prediction. **Best for generation** — this is what "LLM" usually means today.
- **Encoder-Decoder / seq2seq (T5, BART, original Transformer):** an encoder builds a full input representation; a decoder generates output while **cross-attending** (its Queries read the encoder's Keys/Values). **Best for input→output transformation:** translation, summarization.

**Why decoder-only dominates modern LLMs.** A single next-token objective scales cleanly, casts *every* task as "text in → text out," and simplifies training and serving. Interview favorite: *"Why is GPT decoder-only but BERT encoder-only, and what does that imply about their usable tasks?"*

**1.1.7.b The output head (language-model head).**
- **Definition:** the final block's output vectors are projected by a linear layer to **logits** over the vocabulary (size `V`), then **softmax** gives next-token probabilities.
- **Weight tying** (from 1.1.1) often shares this projection with the input embedding matrix.
- These logits feed decoding (temperature/top-k/top-p) — the subject of the *inference* topic (1.3), which this chapter sets up.

**Worked example (matching architecture to task).**
- *Classify* a product review's sentiment → encoder-only (BERT): bidirectional context → one label.
- *Write a reply* to the review → decoder-only (GPT-style).
- *Translate* the review → encoder-decoder (T5/BART).

**Interview angle.** "Compare the three architectures and give a task for each." "What is cross-attention and where is it used?" "What does the LM head do and what are logits?" "Why decoder-only for chatbots?"

**Finance use cases.**
1. **Encoder-only:** Classify transactions/emails as fraud vs legitimate; embed filings for semantic search.
2. **Decoder-only / enc-dec:** Generate plain-English explanations of credit decisions; summarize/translate earnings materials for clients.

**E-commerce use cases.**
1. **Encoder-only:** Route support tickets by category; embed products for recommendations.
2. **Decoder-only / enc-dec:** Auto-generate product descriptions from attributes; translate listings for international marketplaces.

## 1.1.8 The Full Forward Pass (putting every piece together)

Trace one pass through a **decoder-only** LLM — this is the synthesis interviewers love:

1. **Tokenize** text → token IDs.
2. **Embed** IDs via `E` (× √d_model) → token vectors.
3. **Add positional information** (sinusoidal/learned/RoPE) so order is known.
4. For each of **N blocks** (Pre-LN):
   a. `LayerNorm` → **Multi-Head Causal Self-Attention** (with KV cache at inference) → **residual add**.
   b. `LayerNorm` → **FFN (SwiGLU)** → **residual add**.
5. **Final LayerNorm** → **LM head** (linear → logits over vocab).
6. **Softmax + decoding** (temperature/top-k/top-p) → pick next token.
7. **Append** the token and **repeat** (autoregressive generation) until a stop token or max length.

**One-sentence summary.** *Embed and position the tokens → alternately mix context (attention) and refine per token (FFN) through many residual, normalized layers → project to vocabulary logits → sample the next token → loop.*

## 1.1.9 Modern Efficiency & Scaling (industry-relevant extensions)

The vanilla 2017 design is rarely shipped unchanged. Know these, because production and interviews assume them:

- **1.1.9.a KV cache.** At inference, store past tokens' Keys/Values so each new step only computes the new Query and reuses cached K/V — turning repeated recomputation into a memory lookup. **Trades memory for speed.** Its size scales with `context × layers × heads × head_dim × 2 × batch`, which is *why long context and large batches are memory-hungry*. (Explored fully in the inference topic 1.3.)
- **1.1.9.b MQA / GQA** (from 1.1.4): shrink the KV cache by sharing K/V across heads/groups → more throughput and longer context per GPU.
- **1.1.9.c FlashAttention:** an IO-aware exact-attention algorithm that avoids materializing the full `n×n` matrix in slow memory → big speed/memory wins. It doesn't change the math, only how it's computed.
- **1.1.9.d Sparse / sliding-window attention** (e.g., Mistral): each token attends only to a local window (plus some global), reducing the O(n²) burden for long sequences.
- **1.1.9.e Mixture-of-Experts (MoE)** (e.g., Mixtral): replace the dense FFN with many expert FFNs, routing each token to a few → far more parameters (capacity) at roughly constant compute per token.

**Interview angle.** "How would you serve a 128k-context model efficiently?" (GQA + FlashAttention + paged KV cache + possibly sliding-window). "Why does batch size trade off against context length?" (shared GPU memory via KV cache). "What is MoE and why is it efficient?"

**Finance use cases.**
1. **Long-document throughput:** FlashAttention + GQA make full-filing analysis servable under memory limits.
2. **Concurrent risk queries:** Efficient KV management sets how many simultaneous analyst requests a GPU can handle.

**E-commerce use cases.**
1. **Massive concurrent chat:** KV-cache efficiency and GQA determine how many shopper sessions fit per server.
2. **Cost-efficient scale (MoE + sliding-window):** Serve large catalogs and long conversations affordably.

# Wrap-Up: 1.1 Transformer Architecture

## The through-line (how each piece answers the previous one)
RNNs were sequential and forgetful → **attention** (1.1.3) lets any token read any other in one parallel step. But attention ignores order → **positional encoding** (1.1.2) restores it, on top of **token embeddings** (1.1.1). One attention weighting is a bottleneck → **multi-head** (1.1.4) captures many relationships at once. Attention only mixes context → the **FFN** (1.1.5) adds per-token nonlinear processing and knowledge. To stack this deep → **residuals + normalization** (1.1.6). Arrange blocks into **encoder/decoder** shapes and add an **output head** (1.1.7); run the whole **forward pass** autoregressively (1.1.8); and make it practical with **KV cache, GQA, FlashAttention, MoE** (1.1.9). *Every later topic — tokenization economics, training/inference, model choice, RAG, agents, fine-tuning, serving — is an engineering response to something introduced here.*

## Mini glossary
**d_model** — the model's hidden dimension. **d_k** — per-head key/query dimension.
**Q/K/V** — query, key, value projections. **Logits** — pre-softmax vocab scores.
**Causal mask** — blocks attention to future tokens. **Cross-attention** — Q from one sequence, K/V from another.
**RoPE** — rotary positional embedding (relative position via rotation).
**RMSNorm** — LayerNorm variant without mean-centering.
**GQA/MQA** — grouped/multi-query attention (share K/V to shrink KV cache).
**FFN** — position-wise feed-forward sublayer. **MoE** — mixture-of-experts (sparse FFN).
**KV cache** — cached past Keys/Values to speed autoregressive decoding.
**Pre-LN** — normalize before the sublayer (stable at depth).
