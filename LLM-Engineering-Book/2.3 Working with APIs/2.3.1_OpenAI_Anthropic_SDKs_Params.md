## 2.3.1.0 Where this fits (the bridge from 2.2)

Up to now the API call has been a black box in our code samples — `client.chat.completions.create(...)`. That black box is where all of your design choices become a concrete HTTP request, and where a handful of parameters quietly decide whether your feature is **deterministic or creative, cheap or expensive, fast or slow, complete or truncated**. Getting these wrong is one of the most common sources of "the model is being weird" bug reports, and almost all of them trace back to a misunderstanding of what the parameter controls at the sampling level (1.3.5).

There is also a practical reality: the two dominant SDKs — OpenAI and Anthropic — are **90% conceptually identical and 10% different in ways that will trip you up** (Anthropic *requires* `max_tokens`; it takes `system` as a separate argument rather than a message; it calls stop sequences `stop_sequences`). Knowing both, and knowing the shared mental model beneath them, means you can move between providers (and support multi-provider apps, which most production systems become) without relearning anything.

> **One-line thesis:** *An LLM API call is a sampler you configure. The SDK shapes the request; the parameters shape the probability distribution and where generation stops. Learn what each knob does to sampling, and the two big SDKs become the same tool with different spellings.*

## 2.3.1.a The SDKs: Setup, Auth, and the Request Shape

**Definition.** The **OpenAI** and **Anthropic** Python SDKs are official libraries that wrap the providers' REST APIs: they handle authentication, request serialization, response objects, retries on transport errors, streaming, and async. You create a **client** once (it reads your API key), then call a **chat/messages method** with a list of role-tagged messages and parameters.

**Authentication and configuration — the part people get wrong first.** Both SDKs read the API key from an **environment variable** by default (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`). This is deliberate and important: you should **never hard-code keys** in source (a security and secret-leakage risk — recall the "no secrets in prompts/code" theme from 2.1.3/2.2.3). Load them from the environment or a secrets manager. In local development a `.env` file plus `python-dotenv` is the common pattern; in production, use your platform's secret store. The client object is cheap to create once and reuse; you don't make a new client per request.

**The request shape — a shared mental model.** Both APIs take, at minimum: a **model** identifier, a **list of messages** (each with a `role` and `content`), and **generation parameters**. The message roles are exactly the ones from tokenization/chat-templates (1.2.4): **system** (persona, rules — 2.1.1.c), **user** (the request), and **assistant** (the model's prior turns, and how you supply few-shot examples — 2.1.1.b). The conversation is **stateless**: the API does not remember prior turns for you — *you* resend the full message history each call. This is a frequent surprise and has direct cost implications (every prior turn is re-sent as input tokens — 1.2.6), and it's why conversation memory becomes an explicit concern in agents (Phase 5).

**The one structural difference worth memorizing.** OpenAI puts the system prompt as a **message** with `role: "system"` in the messages list. Anthropic takes the system prompt as a **separate top-level `system` parameter**, and its messages list contains only `user`/`assistant` turns. Same concept, different placement.

**Python — OpenAI: create a client and a basic call:**
```python
from openai import OpenAI
client = OpenAI()   # reads OPENAI_API_KEY from the environment

resp = client.chat.completions.create(
    model="gpt-5.5",
    messages=[
        {"role": "system", "content": "You are a concise financial analyst."},
        {"role": "user",   "content": "Explain a bond ladder in two sentences."},
    ],
    temperature=0.3,
    max_tokens=120,
)
print(resp.choices[0].message.content)
print(resp.usage)   # prompt_tokens, completion_tokens, total_tokens  <- cost tracking
```

**Python — Anthropic: the same intent, note the differences:**
```python
from anthropic import Anthropic
client = Anthropic()   # reads ANTHROPIC_API_KEY

msg = client.messages.create(
    model="claude-opus-4-8",
    system="You are a concise financial analyst.",   # SEPARATE param, not a message
    messages=[
        {"role": "user", "content": "Explain a bond ladder in two sentences."},
    ],
    max_tokens=120,        # REQUIRED by Anthropic (OpenAI treats it as optional)
    temperature=0.3,
)
print(msg.content[0].text)
print(msg.usage)          # input_tokens, output_tokens
```

**Reading the response — and why `usage` matters.** Both responses carry the generated text plus a **`usage`** object reporting input/output token counts. You should log this on every call: it's how you track cost (1.2.6), detect prompt bloat, and set budget alerts. The response also carries a **stop/finish reason** — telling you *why* generation ended (natural stop, hit `max_tokens`, matched a stop sequence, tool call). Ignoring the finish reason is how you ship silently **truncated** answers; we return to this under `max_tokens` below.

**Sync, async, and streaming — a note.** Both SDKs offer an **async client** (`AsyncOpenAI`, `AsyncAnthropic`) for concurrency (important when you fan out many calls — e.g., processing a batch, or parallel tool calls from 2.2.2), and **streaming** (token-by-token output) for responsive UX. Streaming and async are their own topic (2.3.2), but it's worth knowing the same `create(...)` call gains `stream=True` and that async is a drop-in `await`.

**Finance use cases.**
1. **Provider-portable analytics service:** Because the two SDKs share a mental model, a document-analysis service can route between OpenAI and Anthropic (for cost, availability, or quality per task) behind a thin adapter, with `usage` logged for per-provider cost attribution.
2. **Auditable calls:** Logging model id, parameters, message history, and `usage`/finish-reason per request creates the audit trail regulated finance workflows expect.

**E-commerce use cases.**
1. **High-concurrency support:** The async client lets a storefront assistant handle many simultaneous sessions efficiently (ties to KV-cache/throughput realities from 1.3.6).
2. **Cost dashboards:** Persisting `usage` per call powers a per-feature cost dashboard so bulk description generation stays within budget.

## 2.3.1.b The Core Parameters — What Each Knob Actually Does

Every parameter here is a control over the **sampling process** from 1.3.5 (logits → temperature scaling → top-k/top-p filtering → sample) or over **when generation stops**. Understanding them at that level is what lets you set them on purpose.

**2.3.1.b.i `temperature` — the randomness/creativity dial.**
Recall from 1.3.5 that temperature divides the logits before softmax. **Low temperature (→0)** sharpens the distribution toward the single most likely token, making output **focused, consistent, and near-deterministic** — the right choice for extraction, classification, structured output, code, and anything you'll parse or audit. **High temperature (>1)** flattens the distribution, increasing diversity and "creativity" at the cost of coherence and reproducibility — useful for brainstorming, marketing copy variants, or when you *want* varied samples (e.g., self-consistency, 2.1.2). The typical usable range is roughly 0–1 for most tasks; going much above 1 tends toward incoherence. A subtle but important point: **temperature does not make the model "smarter" or "more accurate"** — it only changes how adventurously it samples. For factual reliability you lower temperature *and* add grounding/validation (2.2.3), not the reverse.

**2.3.1.b.ii `top_p` (nucleus) and `top_k` — the other side of sampling.**
`top_p` restricts sampling to the smallest set of tokens whose cumulative probability reaches `p` (1.3.5); `top_k` restricts to the `k` most likely tokens. These are **alternative/complementary ways to constrain randomness**. The practical guidance most providers give — and a good rule to internalize — is to **tune *either* `temperature` *or* `top_p`, not both at once**, because they interact in ways that are hard to reason about. In most applied work, adjusting `temperature` alone is sufficient; reach for `top_p` when you want to cap the "long tail" of unlikely tokens while keeping some diversity.

**2.3.1.b.iii `max_tokens` / `max_completion_tokens` — the output length cap (and a top source of bugs).**
This limits how many tokens the model may **generate** (it does not limit the input). Two things make it consequential. First, **cost and latency scale with output tokens** (1.2.6, 1.3.6) — capping output is your most direct lever on both. Second, and more dangerously: if the model hits the cap mid-sentence, generation **stops abruptly and the output is truncated** — and you'll often get *invalid* JSON if you were doing structured output. This is why you must **check the finish/stop reason**: a finish reason of `length` means "I ran out of room, this is incomplete," and you should raise the cap or handle it, not treat the truncated text as a complete answer. Anthropic **requires** `max_tokens` on every call (forcing you to think about it); OpenAI makes it optional (and defaults can bite you). Set it generously enough to fit the longest legitimate answer, but no larger than needed.

**2.3.1.b.iv `stop` / `stop_sequences` — telling the model where to end.**
A **stop sequence** is a string (or list of strings) that, when generated, causes the model to **halt immediately** (the stop text itself is not included in the output). This is a precision tool for **controlling output boundaries**: if you've asked the model to emit an answer inside `</answer>` or to stop before a delimiter you use for the next section, a stop sequence enforces it. It's especially useful in older-style completion formats, custom templates, and when you want to prevent the model from "running on" past the part you care about. In modern chat/tool APIs it's used less often (the model usually stops naturally), but it remains valuable for tight formatting control and for hand-rolled ReAct-style loops (2.1.2/2.2.2) where you stop at "Observation:".

**2.3.1.b.v `seed` and reproducibility.**
Some APIs accept a **`seed`** to make sampling **more reproducible**: with the same seed, model, and parameters, you tend to get the same (or very similar) output. This is valuable for testing, debugging, and any workflow that values consistency. But note the caveat from 1.3.7: **true bit-for-bit determinism isn't guaranteed** even with a seed, because backend batching, hardware, and floating-point non-associativity introduce variation. Treat `seed` as "best-effort reproducibility," and for hard determinism combine `temperature=0` + `seed` + version-pinned models — and still expect occasional drift.

**2.3.1.b.vi `frequency_penalty` and `presence_penalty` — anti-repetition dials (OpenAI).**
`frequency_penalty` reduces the probability of tokens **proportional to how often they've already appeared** (discourages verbatim repetition); `presence_penalty` reduces probability for tokens that have appeared **at all** (encourages introducing new topics). Both default to 0 and are usually left there; raise them modestly if you see the model looping or being repetitive. Understanding them as *additive adjustments to logits before sampling* keeps them demystified.

**2.3.1.b.vii `n` — multiple completions in one request (OpenAI).**
`n` asks for several independent completions to the same prompt in one call. It's how you cheaply generate variants (A/B copy) or gather multiple reasoning paths (self-consistency, 2.1.2). Note that output tokens — and cost — scale with `n`.

**2.3.1.b.viii The reasoning-model exception (2026, important).**
Dedicated **reasoning models** (o-series, Claude extended thinking, DeepSeek R1, etc. — see 2.1.2) often **ignore or restrict `temperature`/`top_p`** and instead expose an **effort / thinking-budget** parameter (e.g., `reasoning_effort`, or a "thinking tokens" budget) that controls how much test-time compute they spend before answering. So the "which knob do I turn?" answer is **model-dependent**: classic chat models → temperature/top_p; reasoning models → effort/thinking budget. Setting temperature on a reasoning model may do nothing — a common point of confusion worth internalizing now.

**Python — parameters set deliberately for two different jobs:**
```python
# Job 1: deterministic extraction (parse-and-store) -> low temp, capped, stop-bounded
extract = client.chat.completions.create(
    model="gpt-5.5",
    messages=[{"role": "user", "content": "Return the invoice total as JSON: <doc>"}],
    temperature=0,            # determinism (1.3.5)
    max_tokens=100,           # enough for the JSON, no more
    stop=["\n\n"],            # halt after the JSON block
    seed=42,                  # best-effort reproducibility
)
if extract.choices[0].finish_reason == "length":
    raise ValueError("output truncated -- raise max_tokens")   # check finish reason!

# Job 2: creative marketing variants -> higher temp, multiple samples
variants = client.chat.completions.create(
    model="gpt-5.5",
    messages=[{"role": "user", "content": "Write a punchy 1-line tagline for wireless earbuds."}],
    temperature=0.9,          # diversity
    n=5,                      # five options in one call
    presence_penalty=0.4,     # nudge toward varied wording
    max_tokens=30,
)
for c in variants.choices:
    print(c.message.content)
```

**Python — the Anthropic parameter analogues:**
```python
msg = client.messages.create(
    model="claude-opus-4-8",
    system="Extraction assistant.",
    messages=[{"role": "user", "content": "Return the invoice total as JSON: <doc>"}],
    max_tokens=100,               # required
    temperature=0,
    stop_sequences=["\n\n"],      # note the name difference vs OpenAI's `stop`
)
if msg.stop_reason == "max_tokens":
    raise ValueError("output truncated -- raise max_tokens")   # Anthropic's finish-reason field
```

**Finance use cases.**
1. **Extraction vs narrative, tuned separately:** A pipeline uses `temperature=0` + tight `max_tokens` + stop sequences for booking-grade field extraction, but a higher temperature for generating a readable client-facing summary — the *same* model, configured per job.
2. **Truncation safety on long analyses:** Checking the finish reason on lengthy risk write-ups prevents a truncated, misleading summary from being sent as if complete.

**E-commerce use cases.**
1. **Bulk copy generation with `n`:** Generating five tagline variants per product in one call (with a higher temperature) feeds A/B testing while controlling cost via small `max_tokens`.
2. **Deterministic tagging at scale:** `temperature=0` + `seed` gives stable, repeatable attribute tags across catalog re-runs, so re-processing doesn't churn the data.

# Wrap-Up: 2.3.1 SDKs & Core Parameters

## The through-line (backward and forward)
Every technique in Phases 1–2 is delivered through an **API call**, and this chapter demystified that call. The **SDKs** (2.3.1.a) share one mental model — client + role-tagged, *stateless* messages + parameters + a response carrying text, `usage`, and a finish reason — with a few Anthropic-specific spellings (`system` as a param, required `max_tokens`, `stop_sequences`). The **parameters** (2.3.1.b) are all controls over the sampling process from 1.3.5 or over stopping: `temperature`/`top_p`/`top_k` shape randomness (tune one, not both; low for anything parsed/audited), `max_tokens` caps output (drives cost/latency and, if hit, truncates — always check the finish reason), `stop` bounds output precisely, `seed` gives best-effort reproducibility, penalties curb repetition, `n` yields variants — while **reasoning models** swap temperature for an effort/thinking budget. This sets up the rest of 2.3: **streaming** (2.3.2), **token counting & cost estimation** (2.3.3), and **rate limits & error handling** (2.3.4) — each building on the request/response and `usage` fundamentals established here.

## Quick parameter reference
| Parameter | Controls | Typical setting | Watch out for |
|---|---|---|---|
| `temperature` | Randomness (logit scaling) | 0 for parsed/audited; 0.7–1 for creative | Doesn't improve accuracy; ignored by many reasoning models |
| `top_p` / `top_k` | Randomness via token filtering | tune *instead of* temperature | Don't tune together with temperature |
| `max_tokens` | Output length cap | just above longest valid answer | Truncation if hit — check finish reason; required by Anthropic |
| `stop` / `stop_sequences` | Where generation halts | delimiters/section breaks | Stop text is excluded from output |
| `seed` | Reproducibility | fix for tests/consistency | Not truly deterministic (1.3.7) |
| `frequency`/`presence_penalty` | Repetition / novelty | usually 0 | Small nudges only |
| `n` | # completions | >1 for variants/self-consistency | Cost scales with `n` |
| effort/thinking budget | Reasoning depth (reasoning models) | per task difficulty | Replaces temperature on those models |

## Selected interview questions (short answers)
1. **Key structural difference between the OpenAI and Anthropic SDKs?** OpenAI puts `system` as a message and treats `max_tokens` as optional; Anthropic takes `system` as a separate param and *requires* `max_tokens` (and calls stops `stop_sequences`).
2. **Is the chat API stateful?** No — it's stateless; you resend the full message history each call (which costs input tokens every turn).
3. **What does `temperature` actually do, and does it improve accuracy?** Scales logits before softmax to change randomness; it does not improve factual accuracy — lower it and add grounding/validation for reliability.
4. **Why should you tune temperature *or* top_p, not both?** They both constrain sampling and interact unpredictably together.
5. **What happens if `max_tokens` is hit, and how do you detect it?** Output is truncated (often invalid JSON); detect via the finish/stop reason (`length`/`max_tokens`).
6. **What is a stop sequence and is it included in the output?** A string that halts generation when produced; it is *not* included in the output.
7. **Does `seed` guarantee identical outputs?** No — best-effort reproducibility only; batching/hardware/float non-associativity cause drift.
8. **Which parameter matters on reasoning models instead of temperature?** An effort/thinking-budget parameter controlling test-time compute.
9. **Why log `usage` on every call?** Cost tracking, prompt-bloat detection, and budget alerts.

## Mini glossary
**Client** — SDK object holding auth/config, reused across calls.
**Message roles** — system/user/assistant (1.2.4).
**Stateless API** — history must be resent each call.
**`usage`** — token counts returned per response.
**Finish/stop reason** — why generation ended (natural, length, stop, tool).
**`temperature`** — randomness via logit scaling.
**`top_p`/`top_k`** — randomness via token-set filtering.
**`max_tokens`** — output length cap.
**Stop sequence** — string that halts generation.
**`seed`** — best-effort reproducibility.
**Effort/thinking budget** — reasoning-model compute control.
