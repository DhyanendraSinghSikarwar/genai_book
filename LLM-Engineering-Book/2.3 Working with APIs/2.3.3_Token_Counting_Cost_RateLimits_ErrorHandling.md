## 2.3.3.0 Where this fits (the bridge from 2.3.1–2.3.2)

By now you can shape a request (2.3.1) and stream it (2.3.2). But two questions decide whether that request survives contact with production: **"What will this cost at our volume?"** and **"What happens when the API says no?"** Neither is glamorous, and both are where LLM projects quietly fail — a feature that works beautifully in testing becomes unaffordable at a million calls, or falls over the first time it hits a rate limit during a traffic spike.

The connective idea is that **tokens are the currency of everything here**. Tokens determine cost (you're billed per token — 1.2.6), tokens determine rate-limit consumption (limits are often expressed in tokens-per-minute), and token-heavy requests are the ones most likely to be throttled or truncated. So this chapter starts with *counting* tokens, uses that to *estimate* cost, then covers the *limits* that token throughput runs into, and finally the *errors* — many of which are limit-related — you must handle. It's one connected story about operating within finite, priced capacity.

> **One-line thesis:** *Count tokens to forecast cost, engineer around rate limits with concurrency control and backoff, and treat every API error as an expected event with a defined response. This operational layer is what makes an LLM feature affordable, scalable, and reliable — the difference between "it works" and "it works in production."*

## 2.3.3.a Token Counting

**Definition.** **Token counting** is measuring how many tokens a piece of text (or a full chat request) will consume *before* or *after* sending it. You count **before** to check you fit within the context window (1.2.5) and to estimate cost; you read the actual counts **after** from the response's `usage` object (2.3.1).

**Why you count before sending, explained.** Two concrete needs. First, **context-window safety**: if your prompt + expected output exceeds the model's window, the request fails or truncates (1.2.5) — counting lets you detect this and trim/chunk *proactively* (the seed of RAG, Phase 4). Second, **cost forecasting**: since cost is a linear function of tokens, counting the input (and estimating output) tells you the price of a call before you make it, which is essential for budgeting a feature at scale. Counting is cheap and local; a surprise truncation or a runaway bill is not.

**How to count — the tools.** For OpenAI models, **`tiktoken`** is the official, local tokenizer library: you load the encoding for a model and encode text to get the exact token list. Crucially, `tiktoken` runs **offline** — no API call, no cost — so you can count freely. Anthropic provides a **count-tokens endpoint/method** for exact counts on its models. There's an important subtlety for **chat** requests: the token count is *not* just the sum of message contents — the chat format adds a few tokens per message for role markers and structure (recall the special/chat-template tokens from 1.2.4). So counting a chat request means accounting for that per-message overhead, not just concatenating strings.

**Intuition — a fuel gauge before the trip.** Counting tokens is checking how much fuel a journey needs before you set off: you confirm you have enough range (context window) and you know the cost of the trip (price per token). Reading `usage` afterward is checking the odometer — the ground truth of what you actually burned. You want both: the estimate to plan, the actuals to reconcile.

**A practical estimation shortcut (from 1.2.6).** When you don't need exactness, the rule of thumb is **~1 token ≈ 4 characters ≈ ¾ of a word** for English (~750 words ≈ 1,000 tokens), with **code, numbers, and non-English text costing more** tokens per character. Use the shortcut for back-of-envelope planning; use `tiktoken`/the count endpoint when precision matters (billing, tight context budgets).

**Python — exact counting with `tiktoken`, including chat overhead:**
```python
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o")   # pick the model's encoding

def count_text_tokens(text: str) -> int:
    return len(enc.encode(text))

def count_chat_tokens(messages: list[dict]) -> int:
    # Approx per-message overhead for role/format tokens (varies by model family)
    tokens = 0
    per_message_overhead = 4
    for m in messages:
        tokens += per_message_overhead
        tokens += count_text_tokens(m["content"])
        tokens += count_text_tokens(m.get("role", ""))
    tokens += 2   # priming tokens for the assistant reply
    return tokens

msgs = [{"role": "system", "content": "You are concise."},
        {"role": "user", "content": "Summarize this 10-K risk section: ..."}]
print(count_chat_tokens(msgs))          # estimate BEFORE sending
```

**Python — trusting the actuals afterward (and reconciling):**
```python
resp = client.chat.completions.create(model="gpt-5.5", messages=msgs, max_tokens=200)
u = resp.usage
print(u.prompt_tokens, u.completion_tokens, u.total_tokens)   # ground truth for billing
```

**Finance use cases.**
1. **Context-budget enforcement:** Before sending a long filing, count tokens to confirm it fits the window; if not, trigger chunking/retrieval so the request never silently truncates a critical clause.
2. **Per-request cost pre-check:** A compliance batch counts input tokens up front to reject or split pathologically large documents before they incur outsized cost.

**E-commerce use cases.**
1. **Prompt-bloat detection:** Counting tokens across a description-generation template catches an accidentally huge few-shot block before it multiplies cost across a million SKUs.
2. **Fit checks for long reviews:** Counting review-summarization inputs ensures they fit the window, routing oversized ones to chunked handling.

## 2.3.3.b Cost Estimation (and the Levers That Cut It)

**Definition.** **Cost estimation** is forecasting the monetary cost of LLM usage from token counts and per-token prices. The base formula is simple and worth memorizing:

```
cost = (input_tokens  × input_price_per_token)
     + (output_tokens × output_price_per_token)
```

**The details that matter.** Providers price **input and output tokens separately, and output is typically more expensive** (often several times more) because it's generated sequentially in the decode phase (1.3.6). Prices are usually quoted per **million tokens**, and they **vary by model tier** — a frontier model can cost 10–50× a small model for the same tokens (recall the capability/price ladder from 1.4.1). This is why **model routing** — sending easy subtasks to a cheap model and only hard ones to a frontier model — is one of the highest-leverage cost levers (you'll build this in the Phase 9 capstone). To estimate a *feature's* cost, you multiply per-call cost by expected volume: `per_call_cost × calls_per_day × 30` gives a monthly figure, and doing this early prevents the classic "the demo was cheap, production is $40k/month" surprise.

**The cost-reduction levers, explained (this is the practically important part).**
- **Prompt caching.** Both major providers offer **caching of a stable prompt prefix** (e.g., a long system prompt, few-shot block, or reference document reused across calls). Cached input tokens are billed at a **large discount** (often ~90% off) on subsequent calls. When you repeatedly send the *same* leading context — extremely common with system prompts and RAG scaffolds — structuring the prompt so the reusable part comes first and enabling caching can slash input cost dramatically. This is the single biggest lever most teams under-use.
- **Batch API.** For **non-urgent, high-volume** work (nightly extraction, bulk generation), providers offer a **Batch API** that processes requests asynchronously within a window (e.g., 24h) at roughly **half price**. If latency doesn't matter, batching halves cost.
- **Model routing / right-sizing.** Use the smallest model that clears your quality bar on *your* eval (1.4.3); reserve expensive models for the hard cases.
- **Shorter prompts + retrieval instead of stuffing.** Every token you don't send is a token you don't pay for; RAG (Phase 4) sends only relevant chunks rather than whole documents.
- **Capping `max_tokens`.** Since output dominates cost, limiting output length directly limits spend (2.3.1).
- **Reasoning-token awareness (2026).** Reasoning models bill their hidden **thinking tokens** as output — powerful but potentially expensive; match reasoning effort to task difficulty (2.1.2).

**Intuition — tokens are a metered utility.** Treat the LLM like electricity: you pay for what you draw, output "current" costs more than input, premium appliances (frontier models) draw far more, and there are off-peak rates (Batch API) and insulation (caching) that cut the bill. Good engineering here is the same as good energy management: don't draw more than you need, reuse what you can, and schedule non-urgent load cheaply.

**Python — a reusable cost estimator:**
```python
PRICES = {  # $ per 1M tokens (illustrative — always check current pricing)
    "gpt-5.5":       {"in": 2.50, "out": 10.00},
    "gpt-5.5-mini":  {"in": 0.15, "out": 0.60},
}

def estimate_cost(model, input_tokens, output_tokens, cached_input_tokens=0):
    p = PRICES[model]
    billable_input = input_tokens - cached_input_tokens          # cached prefix discounted
    cost = (billable_input      / 1_000_000) * p["in"]
    cost += (cached_input_tokens / 1_000_000) * p["in"] * 0.10   # ~90% off cached
    cost += (output_tokens       / 1_000_000) * p["out"]
    return cost

# One call, then project to monthly volume:
per_call = estimate_cost("gpt-5.5", input_tokens=1200, output_tokens=300)
print(f"per call: ${per_call:.5f}   monthly @100k/day: ${per_call*100_000*30:,.0f}")

# Compare routing to the mini model for the easy 80% of traffic:
mixed = 0.8*estimate_cost("gpt-5.5-mini",1200,300) + 0.2*estimate_cost("gpt-5.5",1200,300)
print(f"blended per call with routing: ${mixed:.5f}")
```

**Finance use cases.**
1. **Caching a fixed policy corpus:** A compliance assistant that prepends the same regulatory text to every query enables prompt caching, cutting input cost ~90% on that shared prefix across millions of calls.
2. **Batch overnight extraction:** Processing the day's filings via the Batch API halves the cost of a workload where same-day (not real-time) results are fine.

**E-commerce use cases.**
1. **Routing for bulk copy:** Generating standard descriptions on a cheap model and only escalating flagship products to a frontier model roughly halves blended cost at catalog scale.
2. **Cached brand-voice prompt:** The shared brand-guidelines system prompt is cached, so per-SKU generation only pays full price for the small variable part.

## 2.3.3.c Rate Limits

**Definition.** **Rate limits** are provider-imposed caps on how much you can call the API in a time window, protecting shared infrastructure and enforcing fair use. They're commonly expressed as **RPM** (requests per minute), **TPM** (tokens per minute), and sometimes RPD/TPD (per day) or concurrent-request limits. Limits are tied to your account/organization and usage tier, and they **rise as your usage tier increases** (spend history unlocks higher limits).

**Why TPM matters more than you'd think.** Because limits are often **token**-based, a few large requests can exhaust your TPM as fast as many small ones — a single 100k-token context call consumes a big slice of a per-minute token budget. So rate-limit planning is really **token throughput** planning, which loops back to counting (2.3.3.a): you can predict when you'll hit TPM by summing the tokens of in-flight requests. Hitting a limit returns an **HTTP 429 "Too Many Requests"** error, frequently accompanied by headers telling you your remaining budget and when it resets.

**Engineering within rate limits, explained.** There are three complementary techniques:
- **Concurrency control:** cap how many requests are in flight at once (e.g., a semaphore or a bounded worker pool), so you don't burst past RPM/TPM. This is the **bulkhead** pattern — isolating and bounding resource use.
- **Client-side throttling / smoothing:** deliberately pace requests (a token-bucket rate limiter) to stay under the limit rather than slamming into it and relying on retries.
- **Backoff on 429:** when you *do* get throttled, **retry with exponential backoff and jitter** (from 2.2.3), ideally honoring the `Retry-After` / reset headers the provider sends. The jitter is important at scale so that many workers don't all retry at the same instant and re-trigger the limit (the "thundering herd").

For genuinely large workloads, the right answer is often **not** to hammer the synchronous API harder but to use the **Batch API** (2.3.3.b), which is designed for high volume and priced lower — sidestepping rate-limit pressure entirely.

**Intuition — a highway with a metered on-ramp.** The API is a highway with limited lanes (RPM) and a weight limit (TPM). Concurrency control is metering your own on-ramp so you never exceed the lanes; throttling is maintaining a steady speed; backoff is what you do when the ramp meter turns red — you wait a randomized moment and try again rather than flooring it. Batch is taking the freight route built for bulk.

**Python — a bounded, throttled, backing-off async worker pool:**
```python
import asyncio, random
from openai import AsyncOpenAI
client = AsyncOpenAI()

sem = asyncio.Semaphore(10)   # concurrency cap -> stay under RPM (bulkhead)

async def call_with_backoff(messages, max_retries=5):
    async with sem:                                  # bound in-flight requests
        for attempt in range(max_retries):
            try:
                return await client.chat.completions.create(
                    model="gpt-5.5-mini", messages=messages, max_tokens=200)
            except Exception as e:                   # includes 429 rate-limit errors
                if attempt == max_retries - 1:
                    raise
                wait = min(2 ** attempt + random.random(), 30)   # exp backoff + jitter
                await asyncio.sleep(wait)

async def run_batch(list_of_messages):
    return await asyncio.gather(*(call_with_backoff(m) for m in list_of_messages))
```

**Finance use cases.**
1. **Throttled nightly batch:** A large statement-processing job uses a bounded worker pool + backoff to stay under TPM, completing reliably instead of failing mid-run on 429s.
2. **Tier planning:** Forecasting peak token throughput (from counting) informs a request for a higher usage tier ahead of a product launch so limits don't throttle real customers.

**E-commerce use cases.**
1. **Traffic-spike resilience:** During a sale, concurrency caps + backoff keep the storefront assistant degrading gracefully under rate pressure rather than erroring out.
2. **Bulk generation via Batch API:** Catalog-wide description generation moves to the Batch API to avoid saturating synchronous rate limits (and to cut cost).

## 2.3.3.d Error Handling

**Definition.** **Error handling** is the discipline of anticipating the ways an API call can fail and responding to each appropriately, so failures become **managed events** rather than crashes or bad user experiences. With LLM APIs — network services fronting non-deterministic models — errors are **normal and frequent**, so handling them well is not optional polish; it's core reliability.

**A taxonomy of failures (know which are retryable — this is the crux).** The single most important distinction is **transient vs permanent**, because it dictates whether retrying helps:
- **Retryable (transient):** **429** (rate limit), **500/502/503/504** (server errors, overload), and network timeouts/connection resets. These often succeed on a later attempt → **retry with exponential backoff + jitter** (2.2.3), honoring `Retry-After` when present.
- **Not retryable (permanent, usually your fault):** **400** (bad/malformed request — e.g., exceeded context window, invalid params), **401** (bad/missing API key), **403** (permission), **404** (unknown model), **422** (validation). Retrying these just wastes calls — **fix the request** (or surface the error) instead.
- **Content-level "failures" (not HTTP errors at all):** a successful response that fails *your* validation (2.2.3) or a **truncated** output (finish reason `length`, 2.3.1). These are handled by the validation/retry and finish-reason checks you already learned — not by transport retries.

**Why blind "retry everything" is a bug.** Retrying a 400 (context too long) will fail every time and burn your rate budget; retrying a 401 will never authenticate. Effective handling **branches on the error type**: back off on transient errors, fix-and-don't-retry on client errors, validate-and-re-prompt on content errors, and after repeated failures, **stop and degrade gracefully** (the **circuit breaker** from 2.2.3) — return a cached/default answer, queue for later, or route to a human. Also crucial: **idempotency** — because retries mean a request may run more than once, any operation with side effects (a tool that transfers money, 2.2.2) must be safe to repeat (use idempotency keys), or you risk double-execution.

**The observability dimension (forward to Phase 8).** You can't handle what you can't see. Log every failure with its type, the model, token counts, and latency; emit metrics (error rate by type, retry counts, cost) and alerts (e.g., spike in 429s, budget threshold crossed). This telemetry — built on the `usage` and finish-reason data from 2.3.1 — is what turns error handling from reactive firefighting into proactive operations, and it's the foundation of the LLMOps work in Phase 8.

**Intuition — errors are weather, not accidents.** In distributed systems, failures aren't exceptional; they're the climate you operate in. You don't build a house that collapses in the first storm and call the storm unexpected — you build for weather. Branch on the kind of storm (transient vs permanent), have an umbrella (backoff), a roof (circuit breaker/fallback), and a rain gauge (monitoring). Treating errors as expected is the mindset shift that makes systems robust.

**Python — error-type-aware handling with the SDK's typed exceptions:**
```python
import time, random
from openai import (RateLimitError, APITimeoutError, InternalServerError,
                    BadRequestError, AuthenticationError)

def robust_call(messages, max_retries=5):
    for attempt in range(max_retries):
        try:
            return client.chat.completions.create(
                model="gpt-5.5", messages=messages, max_tokens=300)
        except (RateLimitError, APITimeoutError, InternalServerError) as e:
            # TRANSIENT -> back off and retry
            if attempt == max_retries - 1:
                return fallback_response(reason=str(e))     # circuit-break to a safe default
            time.sleep(min(2 ** attempt + random.random(), 30))
        except BadRequestError as e:
            # PERMANENT (e.g., context too long) -> do NOT retry; fix or surface
            raise ValueError(f"request problem, will not retry: {e}")
        except AuthenticationError:
            raise                                            # config error -> fail fast
```

**Finance use cases.**
1. **Idempotent transactions under retry:** A `transfer` tool call carries an idempotency key so a retried request after a timeout can't move money twice — essential where double-execution is catastrophic.
2. **Graceful compliance-bot degradation:** On repeated provider outages, the assistant returns a safe "system busy, please retry" message and logs the incident rather than throwing raw errors at users.

**E-commerce use cases.**
1. **No duplicate refunds:** An `issue_refund` action uses idempotency keys so transport retries can't refund twice; failures route to a review queue.
2. **Fallback answers during spikes:** When the model API is overloaded, the shopping assistant serves a cached FAQ answer or a "try again shortly" notice, keeping the storefront usable.

# Wrap-Up: 2.3.3 Token Counting, Cost, Rate Limits & Error Handling

## The through-line (backward and forward)
Tokens are the currency that ties this chapter together. You **count** them (2.3.3.a — `tiktoken`/count endpoints, plus the ~4-chars-per-token shortcut) to guard the context window and forecast **cost** (2.3.3.b), which you then cut with the real levers: **prompt caching**, the **Batch API**, **model routing**, shorter prompts/RAG, and `max_tokens` caps — while watching reasoning-token spend. Token throughput also runs into **rate limits** (2.3.3.c — RPM/TPM/429), which you engineer around with **concurrency control (bulkheads), throttling, backoff+jitter**, and batching. And every call can fail, so **error handling** (2.3.3.d) branches on **transient vs permanent**: back off on 429/5xx, fix-don't-retry on 4xx, validate-and-re-prompt on content failures, and **circuit-break to a fallback** after repeated failure — all made safe by **idempotency** and visible by **monitoring**. This completes Phase 2: you can now prompt, structure, validate, and *operate* LLM calls affordably and reliably. It connects directly to **Phase 4 (RAG)** — the ultimate token-cost and context lever — and **Phase 8 (Deployment/LLMOps)**, where caching, batching, rate-limit handling, and monitoring become production infrastructure.

## Quick reference
| Concern | Key tools/levers | One-line rule |
|---|---|---|
| Token counting | `tiktoken`, Anthropic count endpoint, `usage` | Count before (plan), read `usage` after (reconcile) |
| Cost estimation | in/out prices, caching, Batch API, routing | Output costs most; cache prefixes; route to small models |
| Rate limits | RPM/TPM, semaphore, throttle, backoff, Batch API | Plan token throughput; bound concurrency; back off on 429 |
| Error handling | typed exceptions, backoff, circuit breaker, idempotency | Retry transient, fix permanent, degrade gracefully |

## Selected interview questions (short answers)
1. **Why count tokens before sending?** To stay within the context window and to forecast cost; counting is free/local with `tiktoken`.
2. **Why isn't a chat request's token count just the sum of message text?** The chat format adds per-message role/structure tokens (1.2.4).
3. **The cost formula?** `input_tokens×in_price + output_tokens×out_price`; output is usually pricier and prices vary by model tier.
4. **Biggest under-used cost lever?** Prompt caching of a stable prefix (system prompt, few-shot, reference docs) — often ~90% off cached input.
5. **When use the Batch API?** High-volume, non-urgent work — roughly half price and it sidesteps synchronous rate limits.
6. **What are RPM and TPM, and why does TPM bite?** Requests- and tokens-per-minute limits; a few large-context calls can exhaust TPM fast.
7. **What does a 429 mean and how do you respond?** Rate limited — retry with exponential backoff + jitter, honoring `Retry-After`.
8. **Which errors are retryable vs not?** Retryable: 429, 5xx, timeouts. Not: 400/401/403/404/422 (fix the request). Content failures → validate/re-prompt.
9. **Why is idempotency important with retries?** A retried side-effecting call could execute twice; idempotency keys prevent double-execution.
10. **What is a circuit breaker here?** After repeated failures, stop retrying and degrade gracefully (fallback/default/queue/human).

## Mini glossary
**Token counting** — measuring tokens before/after a call.
**`tiktoken`** — local OpenAI tokenizer for offline counting.
**`usage`** — actual token counts returned per response.
**Prompt caching** — discounted reuse of a stable prompt prefix.
**Batch API** — async, lower-cost bulk processing.
**Model routing** — sending easy tasks to cheaper models.
**RPM / TPM** — requests / tokens per minute limits.
**429** — HTTP "Too Many Requests" (rate limited).
**Backoff + jitter** — increasing, randomized retry delays.
**Bulkhead** — bounding concurrency to isolate/limit load.
**Circuit breaker** — stop-and-degrade after repeated failure.
**Idempotency key** — makes a repeated request safe to run once.
