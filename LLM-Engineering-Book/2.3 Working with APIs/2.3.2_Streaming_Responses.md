## 2.3.2.0 Where this fits (the bridge from 2.3.1 and 1.3.6)

Recall the two phases of inference from 1.3.6: **prefill** (the whole prompt is processed in parallel to produce the first token) and **decode** (output tokens are generated **one at a time**, each a separate forward pass). A crucial consequence hides in that description: the model **produces its output incrementally anyway**. A blocking API call simply *hides* this — it waits for the entire decode loop to finish, then hands you the whole string at once. From the user's point of view, that means staring at a spinner for the full generation time, which for a long answer can be many seconds.

**Streaming removes the hiding.** Instead of buffering the whole answer server-side, the provider forwards each token (or small chunk) to you the instant it's decoded. The total generation time is unchanged — the model isn't faster — but the user starts seeing words almost immediately, which transforms *perceived* latency. This is why the ChatGPT-style "typing" effect is the norm: it's not a UI gimmick, it's the decode loop (1.3.6) made visible. Understanding streaming therefore isn't just an API trick; it's the natural surface of how these models actually generate.

> **One-line thesis:** *Streaming exposes the autoregressive decode loop as it happens — the answer isn't produced faster, but the user sees it sooner. That single change to responsiveness comes with new engineering costs: you now handle a sequence of partial fragments instead of one complete result.*

## 2.3.2.a What Streaming Is and How It Works

**Definition.** **Streaming** is a mode where the API returns the response as a **sequence of incremental events** over a single long-lived HTTP connection, rather than one complete JSON body. Each event carries a small **delta** — usually a token or a few characters — which you append to a growing buffer until a terminal event signals completion. You opt in with `stream=True` (OpenAI) or the streaming methods/`stream=True` (Anthropic).

**The transport, briefly (worth knowing for interviews and debugging).** Under the hood this typically uses **Server-Sent Events (SSE)** over HTTP: the server keeps the connection open and pushes text-encoded events as they're ready, each conventionally prefixed `data: ...`, ending with a sentinel (OpenAI sends a final `data: [DONE]`). The SDKs parse this stream for you and expose it as a **Python iterator** (sync) or **async iterator** (async), so in practice you just loop over events — but knowing it's SSE explains why streaming works through proxies/browsers, why it's one-directional (server→client), and why a dropped connection mid-stream is a real failure mode you must handle.

**The shape of the events differs by provider — a key practical detail.**
- **OpenAI** streams **chunk objects** that mirror the normal response but with a `delta` instead of a full `message`. You read `chunk.choices[0].delta.content` (which may be `None` for non-text events like role announcements or tool-call fragments).
- **Anthropic** streams **typed events** — `message_start`, then repeated `content_block_delta` (carrying `delta.text`), then `content_block_stop`, `message_delta` (which carries stop reason and final usage), and `message_stop`. It's a more explicit event model, which makes it easier to know *what kind* of thing just arrived.

**Intuition — a stream is a conveyor belt, not a package.** A blocking call is a package: it shows up once, complete, and you open it. A stream is a conveyor belt of small parts: you must **assemble** them yourself, watch for the "last part" signal, and be ready for the belt to jam (a mid-stream error) before all parts arrive. Almost every complication in the rest of this chapter follows from that one shift: **you no longer receive a finished object; you receive fragments and reconstruct it.**

**Python — OpenAI streaming (accumulate deltas):**
```python
from openai import OpenAI
client = OpenAI()

stream = client.chat.completions.create(
    model="gpt-5.5",
    messages=[{"role": "user", "content": "Explain a bond ladder."}],
    stream=True,
    stream_options={"include_usage": True},   # ask for a final usage event (see 2.3.2.c)
)

full = []
for chunk in stream:                          # iterator of partial chunks
    delta = chunk.choices[0].delta.content
    if delta:                                 # may be None for role/tool events
        print(delta, end="", flush=True)      # render immediately (the "typing" effect)
        full.append(delta)
answer = "".join(full)                         # reassembled complete text
```

**Python — Anthropic streaming (typed events + a helper):**
```python
from anthropic import Anthropic
client = Anthropic()

# Low-level: iterate typed events
with client.messages.stream(
    model="claude-opus-4-8",
    max_tokens=300,
    messages=[{"role": "user", "content": "Explain a bond ladder."}],
) as stream:
    for text in stream.text_stream:            # convenience: yields text deltas only
        print(text, end="", flush=True)
    final = stream.get_final_message()          # fully assembled message + usage/stop_reason
print("\n", final.usage)
```

The Anthropic SDK's `text_stream` and `get_final_message()` illustrate a general principle: the SDKs give you both a **low-level event iterator** (full control) and **high-level helpers** (assemble text, get the final object). Use the helper unless you need per-event control (e.g., for tool-call streaming, below).

**Finance use cases.**
1. **Responsive analyst copilot:** Streaming a multi-paragraph filing summary lets the analyst start reading in ~1 second instead of waiting for the whole thing — the difference between a tool that feels sluggish and one that feels live.
2. **Long report generation with progress:** For a lengthy risk write-up, streaming provides implicit progress feedback so the user knows the system is working, reducing abandonment.

**E-commerce use cases.**
1. **Storefront chat UX:** The typing effect makes a shopping assistant feel conversational and immediate, which measurably improves engagement.
2. **Live product descriptions in an admin tool:** A merchandiser watching copy stream in can hit "stop" the moment it's going the wrong way (see cancellation, below), saving time and tokens.

## 2.3.2.b The Complications: Parsing, Structured Output, and Tool Calls

Streaming's benefit is responsiveness; its cost is that **you receive an incomplete object over time**, which breaks several things that "just worked" with blocking calls. Handling these well is what separates a robust streaming implementation from a fragile one.

**Complication 1 — you can't parse partial JSON.** This is the big one, and it ties directly back to 2.2 (structured output). If you asked the model for JSON, then **mid-stream you have a broken, unparseable fragment** like `{"issuer": "Ac` — valid JSON only exists once the stream completes. So the naive approach is: **buffer the entire stream, then parse and validate** (with Pydantic, 2.2.3) exactly as you would for a blocking call. In other words, for machine-consumed structured output, streaming buys you *little* on the parsing side — you still wait for completion before you can act. Streaming is primarily for **human-readable** output where partial text is *useful to display*. There are advanced tools for **incremental/partial JSON parsing** (parsing a growing JSON string into a partial object so you can render fields as they arrive — `instructor` and some SDKs support "partial" or "streamed" structured objects), but treat that as an optimization, not the default. The core rule: **display text as it streams; validate structured data only when the stream is complete.**

**Complication 2 — streaming tool calls.** From 2.2.2, a tool call's `arguments` are JSON. When streaming, those arguments arrive **fragment by fragment** across multiple events (OpenAI sends `delta.tool_calls` pieces; Anthropic sends `input_json_delta` events). You **must accumulate the argument fragments until the tool call is complete**, and only *then* parse the JSON and execute the function. Executing on a partial argument string is a bug waiting to happen. So a streaming agent loop is: stream → detect a tool-call has started → buffer its argument deltas → on the tool-call's stop event, parse + validate + execute → feed the result back → continue. The logic is identical to 2.2.2's loop; you've just added a reassembly step for the arguments.

**Complication 3 — mid-stream errors and dropped connections.** With a blocking call, an error arrives as a clean exception before you have any output. With streaming, the connection can **fail partway** — after you've already displayed half an answer. You might receive an **error event** in the stream, or the connection may simply drop. Robust handling means: wrap the iteration in try/except, be prepared to have **partial output already shown**, and decide your recovery policy (retry the whole request — noting the user may see the beginning twice — or gracefully truncate with an error notice). This is genuinely harder than blocking error handling and is a common source of production bugs; it also interacts with the retry logic from 2.2.3 (a mid-stream failure is a transport failure, but one where you've already emitted tokens).

**Complication 4 — cancellation.** A real advantage streaming *enables*: because output is incremental, the user (or your code) can **cancel** generation early — closing the stream/connection stops the model from generating further tokens, which **saves cost and latency**. This is the "stop generating" button. Implementing it means holding a reference to the stream and closing it on a cancel signal. It's a feature you get almost for free with streaming and can't easily do with blocking calls.

**Python — accumulating a streamed tool call correctly:**
```python
# OpenAI: tool-call arguments arrive in fragments; reassemble before executing.
stream = client.chat.completions.create(
    model="gpt-5.5", messages=messages, tools=tools, stream=True,
)
name, args_buf, call_id = None, "", None
for chunk in stream:
    tc = chunk.choices[0].delta.tool_calls
    if tc:
        piece = tc[0]
        if piece.id:  call_id = piece.id
        if piece.function.name:      name = piece.function.name
        if piece.function.arguments: args_buf += piece.function.arguments   # accumulate!
# Only now, after the stream, is args_buf complete and parseable:
import json
args = json.loads(args_buf)          # safe: full JSON assembled
result = AVAILABLE[name](**args)     # execute (2.2.2), then continue the loop
```

**Python — safe structured output over a stream (buffer, then validate):**
```python
buf = []
for chunk in client.chat.completions.create(model="gpt-5.5", messages=msgs,
                                             stream=True):
    d = chunk.choices[0].delta.content
    if d: buf.append(d)              # do NOT parse mid-stream
raw = "".join(buf)
record = BondRecord.model_validate_json(raw)   # validate only when complete (2.2.3)
```

**Finance use cases.**
1. **Streamed narrative + validated data, together:** A copilot streams the human-readable explanation for responsiveness *and*, once complete, validates the structured figures (Pydantic) before booking them — getting UX and correctness without compromising either.
2. **Cancellable long computations:** An analyst who sees a report heading the wrong way cancels mid-stream, immediately stopping token spend on a wasted generation.

**E-commerce use cases.**
1. **Streaming support agent with tools:** The assistant streams its reply text while correctly reassembling a streamed `lookup_order` tool call before executing it — responsive *and* correct.
2. **"Stop generating" in a copy tool:** Merchandisers cancel over-long descriptions early, saving cost across thousands of generations.

## 2.3.2.c Cost, Usage, and When *Not* to Stream

**Usage accounting changes subtly when streaming.** With a blocking call, the response's `usage` object (2.3.1) is always present. When streaming, token usage historically was **not** included by default — you had to either count tokens yourself or explicitly request it. Today both providers give you the final counts: OpenAI via `stream_options={"include_usage": True}` (which appends a final usage event), and Anthropic via the `message_delta`/final-message events that carry `usage`. The practical lesson: **if you rely on `usage` for cost tracking (and you should — 1.2.6), explicitly enable/collect it when streaming**, or you'll silently lose your cost telemetry.

**Streaming does not reduce cost or total time.** A common misconception: streaming is *not* a performance optimization for the model. The same number of tokens are generated at the same speed; you pay the same amount and the total wall-clock generation time is identical. What changes is **time-to-first-token (TTFT)** and *perceived* latency. So the decision to stream is a **UX decision**, not a cost or throughput one.

**When *not* to stream (deciding deliberately).** Streaming adds real complexity (partial parsing, mid-stream errors, tool reassembly), so skip it when it buys nothing:
- **Machine-to-machine / batch pipelines** where no human is watching output arrive — you need the complete, validated object anyway (e.g., nightly extraction of thousands of documents). Blocking is simpler and just as fast.
- **Short outputs** (a classification label, a small JSON record) — generation finishes so quickly that streaming's responsiveness gain is negligible while its complexity cost remains.
- **When you must validate before showing anything** — if you can't display partial output safely (e.g., it must pass a guardrail first, 2.2.3), streaming to the user prematurely is counterproductive.

Conversely, **stream when a human is reading a non-trivial amount of generated text in real time** — chat UIs, long summaries, drafting tools. That's the sweet spot where the responsiveness win clearly outweighs the added engineering.

**A note on the full-stack picture (forward to Phase 8).** Streaming is end-to-end: to deliver the typing effect to a browser, your **backend** must also stream to the frontend — e.g., FastAPI's `StreamingResponse` (or SSE/WebSockets) forwarding each token as it arrives from the provider. So a streaming feature is a decision that propagates through your whole stack (model API → backend → client), which is why it's worth deciding early. We'll implement this in the serving/deployment phase; for now, just know the responsiveness the user sees depends on *every* hop streaming, not just the model call.

**Finance use cases.**
1. **Batch = blocking:** A nightly job extracting fields from 50,000 statements streams nothing — it needs complete, validated records, and blocking calls are simpler and equally fast.
2. **Interactive = streaming, with usage captured:** A live advisory chat streams for responsiveness while explicitly collecting the final `usage` event so per-conversation cost is still tracked for billing/compliance.

**E-commerce use cases.**
1. **Bulk generation = blocking:** Generating descriptions for a million SKUs offline uses blocking calls; there's no human to benefit from streaming, and complete outputs feed validation directly.
2. **Live assistant = streaming:** The customer-facing chatbot streams for a snappy feel, with the backend forwarding tokens via `StreamingResponse` to the browser.

# Wrap-Up: 2.3.2 Streaming Responses

## The through-line (backward and forward)
Streaming is the **decode loop of 1.3.6 made visible**: the model generates one token at a time regardless, and streaming simply forwards each token as it's produced (over SSE), so the user sees output almost immediately (2.3.2.a). It doesn't make generation faster or cheaper — it improves **perceived** latency, a **UX** choice. The price is that you now assemble a result from **fragments**, which breaks partial JSON parsing (buffer-then-validate), requires **reassembling streamed tool-call arguments** before executing (2.2.2), and demands harder **mid-stream error handling** — while unlocking **early cancellation** to save cost (2.3.2.b). Remember to **explicitly collect `usage`** when streaming, and to **not stream** for batch/machine/short-output work where it adds complexity for no benefit (2.3.2.c). This connects backward to structured output/validation (2.2) and tool calls (2.2.2), and forward to **serving** (Phase 8), where you'll stream all the way to the browser. Next up in 2.3: **token counting & cost estimation** (2.3.3) — the telemetry you just learned to preserve through a stream.

## Decision guide: stream or not?
| Situation | Stream? | Why |
|---|---|---|
| Chat UI / long summary a human reads live | **Yes** | Big perceived-latency win |
| Drafting/copy tool with a person watching | **Yes** | Enables read-along + cancel |
| Batch extraction of many documents | **No** | No human watching; need complete validated object |
| Short label / small JSON record | **No** | Finishes instantly; complexity not worth it |
| Output must pass a guardrail before display | **No (or buffer)** | Can't safely show partial output |

## Selected interview questions (short answers)
1. **Does streaming make the model faster or cheaper?** No — same tokens, cost, and total time; it improves time-to-first-token and *perceived* latency (a UX benefit).
2. **What transport underlies streaming?** Usually Server-Sent Events (SSE) over a long-lived HTTP connection; the SDK exposes it as an iterator.
3. **Why can't you parse JSON mid-stream?** Partial output is invalid JSON until complete; buffer the full stream, then parse/validate (2.2.3).
4. **How do you handle a streamed tool call?** Accumulate the argument fragments until the call completes, then parse, validate, and execute.
5. **How is error handling harder when streaming?** Failures can occur mid-stream after partial output is shown; you must handle dropped connections/error events and decide a recovery policy.
6. **What does streaming uniquely enable?** Early cancellation ("stop generating"), which saves cost/latency.
7. **How do you get token `usage` when streaming?** Explicitly request/collect it (OpenAI `stream_options={"include_usage": True}`; Anthropic's final message/`message_delta` events).
8. **When should you NOT stream?** Batch/machine-to-machine pipelines, very short outputs, or when output must be validated before any display.
9. **What must also stream for the user to see the effect?** The whole path — model→backend→client (e.g., FastAPI `StreamingResponse`), not just the model call.

## Mini glossary
**Streaming** — receiving output as incremental token/chunk events.
**Delta / chunk** — an incremental fragment of the response.
**SSE** — Server-Sent Events; the usual streaming transport.
**Time-to-first-token (TTFT)** — delay until the first token appears.
**Perceived latency** — how responsive it *feels* vs total time.
**Buffer-then-validate** — accumulate the full stream before parsing structured output.
**Streamed tool call** — tool arguments arriving in fragments to reassemble.
**Cancellation** — closing the stream early to stop generation.
**`include_usage`** — option to get token counts in a stream.
**`StreamingResponse`** — FastAPI mechanism to stream to the client (Phase 8).
